# 车辆基层智能子系统两步演进计划

## 目标

把当前进程内车辆本地控制逻辑拆成“模拟一辆列车上基层智能子系统”的独立边界。中央系统仍负责列车接入会话、信号网/电网并入、主循环编排和状态持久化；单车基层智能子系统负责本车自检、本车保护、ATO/TCMS 本地决策和车辆物理输入生成。

## 第一步：进程内子系统边界

状态：已落地。

实现内容：

1. 新增 `com.railwaysim.vehicle.onboard` 包。
2. 新增 `OnboardTrainSubsystemManager`，按 `trainId` 注册、调用和移除单车子系统实例。
3. 新增 `OnboardTrainSubsystem`，承接原本控制门面内的本车决策逻辑。
4. 新增 `OnboardTrainControlInput`、`OnboardTrainControlOutput`、`OnboardTrainRegistration`，明确中央到车载基层子系统的进程内接口。
5. `TrainManager` 改为直接通过 `OnboardTrainSubsystemManager` 调用单车子系统，不再直接承担本车控制逻辑。
6. 旧兼容门面已删除，旧测试已迁移到 onboard 边界。

第一步边界：

| 项 | 归属 |
|---|---|
| 列车 ADD/DELETE/CLEAR、并入/退出信号网和电网 | 中央系统侧 `TrainManager`、`ExternalTrainControlSession` |
| 本车门状态、自检、受流、超载、牵引/制动可用、本地 ATO/TCMS 决策 | `OnboardTrainSubsystem` |
| 车辆纵向动力学计算 | `VehiclePhysicsClient` 及本地/外部物理适配器 |
| 调度策略、MA 生成、进路联锁、供电分区状态 | 不属于单车基层智能子系统 |

## 第二步：外置为独立车辆控制节点

状态：已落地基础版。

目标是把第一步的进程内接口替换为可部署节点协议：

1. 定义 `OnboardTrainSubsystemClient`，隐藏本地调用、HTTP、UDP 或后续 gRPC 实现差异。
2. 增加车辆控制节点注册表，记录 `trainId`、`subsystemId`、心跳、数据质量、租约过期时间和最后一次控制输出。
3. 把 `OnboardTrainControlInput/Output` 固化为跨进程接口，明确字段单位、枚举和失败降级策略。
4. 在外部节点超时或失联时，中央系统不接管车辆内部控制，先标记节点降级并保留本地 fallback 用于演示。
5. 通过 `DUAL_SHADOW` 或类似模式让进程内子系统与外置节点并行比对，再逐步切换权威输出。

当前实现：

| 项 | 落点 |
|---|---|
| 本地节点 | `LocalOnboardTrainSubsystemClient` 包装进程内 `OnboardTrainSubsystem`。 |
| 外部节点客户端 | `HttpOnboardTrainSubsystemClient` 调用 `/api/onboard-trains/{trainId}/registration`、`/control`、`/report`。 |
| 模式切换 | `railway.simulation.onboard-subsystem-mode=IN_PROCESS/EXTERNAL_HTTP/DUAL_SHADOW`。 |
| 节点注册表 | `OnboardTrainNodeState` 记录 requested/active mode、连接状态、数据质量、心跳和租约。 |
| 状态查询 | `GET /api/vehicle/onboard-subsystems` 返回中央侧已纳管的车辆控制节点状态。 |
| 降级策略 | `EXTERNAL_HTTP` 失败时回落本地节点；`DUAL_SHADOW` 本地权威、外部只做在线验证。 |

第二步不改变调度、信号、供电的责任边界：调度不越级控车，信号继续提供 MA/限速结果，供电继续提供分区约束。
