# 分计划03：车辆与供电闭环（WP5～WP6）

> 上级计划：[真实车辆FMU集成实施计划](../真实FMU集成实施计划.md)
>
> 前置条件：9000真实FMU服务完成多实例和生命周期验收。
>
> 执行边界：将9300切换为批量FMU执行器，并贯通9200多车牵引/再生物理耦合。

## 1. 阶段目标

形成固定100 ms显式闭环：9300全车状态机产生指令，一次调用9000推进所有车辆实例，按分区聚合电网侧功率后一次提交9200，并在下一周期使用新的网压和能力约束。

## 2. WP5：9300批量物理执行器

### 2.1 实施项

1. 新增`VehiclePhysicsExecutor`接口及`FmuHttpVehiclePhysicsExecutor`、`JavaFallbackVehiclePhysicsExecutor`。
2. 把9300车辆周期拆为：
   - Prepare：全车状态机和物理输入；
   - Execute：一次9000 `/step-fleet`；
   - Apply：写回输出、生成报告和聚合供电负荷。
3. 新列车发送INIT，正常列车发送STEP；移除列车时同步删除9000实例。
4. 单实例错误只回退该车；9000整体错误或协议预检失败时整批Java fallback。
5. fallback列车不得自动热切回FMU；只有RESET/RESYNC成功后才能恢复。
6. 增加`physicsMode`、模型版本、参数哈希、批次时延、fallback数量和数据质量。
7. 8080旧FMU适配器只保留兼容模式，split模式禁止8080和9300双写同一FMU。

### 2.2 验证命令

```bash
mvn -f vehicle-runtime-service/pom.xml test
mvn -f backend/pom.xml test
```

### 2.3 退出条件

- 20辆列车一个tick只产生一次9000请求。
- 正常结果保持现有`VehiclePhysicsOutput`和车辆报告字段兼容。
- 单车FMU错误不影响其他FMU实例。
- 9000超时时9300能在本周期完成fallback并标记`DEGRADED`。

### 2.4 回滚

通过`vehicle-runtime.physics-mode=JAVA_FALLBACK`切回Java模型，不删除状态或数据库字段。

## 3. WP6：9200再生能力和多车耦合

### 3.1 实施项

1. 9200约束响应增加电网侧`regenPowerAvailableWatts`。
2. 首版区段预算：
   - 失电、隔离或欠压时为0；
   - 无可逆变电所时，预算取上一周期同区段牵引电功率；
   - 按当前区段车辆数等分，避免多车重复使用完整预算。
3. 9300、9000和FMU贯通该字段。
4. 9300只向9200提交`tractionPowerWatts`、`regenPowerWatts`和`railCurrentAmps`；两个机械功率字段不进入供电负荷。
5. 9200按区段计算实际吸收和未吸收回馈，并返回下一周期约束。
6. 固定耦合时序为`T(n)`车辆负荷影响`T(n+1)`供电约束，不做周期内隐式迭代。

### 3.2 核心联调场景

1. 两车同区段：A增加牵引负荷，下一周期B的网压或可用功率下降，继而机械功率、牵引力或加速度下降。
2. 两车不同区段：无拓扑联供时互不影响。
3. 同区段牵引车和制动车：回馈功率受区段预算限制，未吸收部分明确记录。
4. 再生预算为0：总制动力不变，再生制动力归零并由非再生制动补足。

### 3.3 验证命令

```bash
PYTHONPATH=power-network-service python -m app.self_test
PYTHONPATH=power-network-service pytest -q power-network-service/tests
mvn -f vehicle-runtime-service/pom.xml test
./scripts/test-fmu-power-closed-loop.sh
```

### 3.4 退出条件

- 同区段耦合和不同区段隔离均有自动断言。
- 多车总回馈不超过区段预算，未吸收功率可查询。
- `P_grid≈P_mechanical/0.88`、`P_regen_grid≈P_regen_mechanical*0.35`误差不超过1%。
- 100 ms连续闭环不存在tick倒退或多次提交同一供电负荷。

### 3.5 回滚

- 9300切回Java fallback。
- 9200保留新增字段并将再生预算配置为0，保持API向后兼容。
