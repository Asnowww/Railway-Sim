# 1000车FMU通信与同步优化验收记录

## 1. 验收结论

本轮完成了1000辆列车同时运行的真实FMU通信压测、瓶颈定位、分片与二进制协议优化，以及实例生命周期同步加固。

结论分为两部分：

| 验收项 | 结果 | 说明 |
|---|---|---|
| 100车完整周期与FMU批次性能 | 通过 | 正式100 tick中9300内部和FMU批次p95均为27 ms，完整HTTP往返p95为35.636 ms |
| 1000车实例隔离、连续步进、物理输出完整性 | 通过 | 正式100 tick中无fallback、无FMI错误、无负速度、列车集合完整 |
| 20 ms FMU、100 ms TCMS差级周期 | 通过 | 每个TCMS tick固定执行5次20 ms FMU子步，控制量在5个子步内保持 |
| 9300实例生命周期与tick互斥 | 通过 | 删除、清空、RESET、RESYNC与`stepFleet`共用同步门禁，不会在子步中途释放实例 |

因此当前实现可作为1000车功能仿真和容量验证版本，但不能宣称在本机Docker Desktop的`linux/amd64`兼容执行环境中满足100 ms硬实时p95。机器可读正式报告见[wp8-1000-train-scale.json](wp8-1000-train-scale.json)。

100车同口径补测已通过全部性能门槛，机器可读报告见[wp8-100-train-scale.json](wp8-100-train-scale.json)。

本记录聚焦9300与9000的车辆/FMU链路。包含8080约束编排、9200权威供电步进、中央快照和WebSocket构建的完整链路优化与100/1000车补测，见[完整车电 Tick 百车、千车优化验收记录](07-完整车电Tick百车千车优化验收记录.md)。

## 3. 优化前基线

### 3.1 既有自动化覆盖

优化前的`test-vehicle-runtime-performance.py`只创建20辆车，短基准仅覆盖1、2、10、20车，长稳也仅覆盖20车；仓库中没有1000车通信或并发验收。

### 3.2 单9000实测

在既有单进程9000上直接执行真实FMU：

| 场景 | INIT | 单次20 ms STEP均值 | 单次STEP最大值/小样本p95 | 每次请求 | 每次响应 |
|---|---:|---:|---:|---:|---:|
| 20车 | 51.388 ms | 5.119 ms | 6.327 ms | 约14.9 KB | 约15.0 KB |
| 1000车 | 1074.120 ms | 99.084 ms | 100.698 ms | 约726～738 KB | 约738～741 KB |

1000车单次20 ms物理步已接近100 ms，而一个TCMS周期需要连续执行5次，因此单进程9000的物理链路约需495 ms，结构上无法满足100 ms TCMS周期。

同进程绕开HTTP、Pydantic和JSON后，1000实例的FMU原生顺序STEP均值仍约68.8 ms，说明瓶颈由两部分组成：FMU实例顺序计算，以及HTTP/校验/序列化约30 ms额外开销。

## 4. 已实施优化

### 4.1 稳定分片和并行请求

9300支持通过`FMU_BASE_URLS`配置多个9000地址，按`Math.floorMod(trainId.hashCode(), shardCount)`稳定选择分片。一个tick内各分片由虚拟线程并发调用，同一列车在分片数不变时始终进入同一9000进程。

分片失败按列车返回`FMU_SHARD_UNAVAILABLE`，不会把其他健康分片的列车一并判失败；删除实例只发送给所属分片，`reset-all`并行广播到全部分片。

### 4.2 MessagePack和紧凑子步

9300与9000之间新增MessagePack链路：

1. 第1个20 ms子步发送完整车辆输入，建立或刷新9000内的静态输入快照。
2. 第2～5个子步只发送`trainId`、位置、速度、累计消耗能量和累计再生能量。
3. 中间子步只回传下一子步必需的5个字段，最后一个子步仍返回完整物理输出。
4. 紧凑请求使用小型幂等哈希，不再为每个中间子步反复把完整静态输入转为JSON后计算SHA-256。
5. 后续子步复用首子步已经完成的静态字段校验，只重新校验变化字段、模型版本、参数集、tick和仿真时间。

JSON端点继续保留；只有`fmu-binary-protocol-enabled=true`时启用紧凑MessagePack子步，避免旧9000或测试桩被隐式切换协议。

### 4.3 同步机制

| 层级 | 当前机制 | 作用 |
|---|---|---|
| 9300全局tick | `VehicleRuntimeManager.stepFleet()`同步执行 | 同一9300不并发推进两个中央tick |
| 9300生命周期 | 注册、删除、清空、RESET、RESYNC与tick使用同一同步门禁 | 防止tick中途释放或迁移实例 |
| 9300到分片 | 各分片并发、分片内每个子步批量发送 | 利用多核并保持同车顺序 |
| 9000分片 | 每个`FmuManager`使用`RLock` | 保证该分片的tick、缓存和实例表原子更新 |
| 单车FMU | `TrainFMUInstance`使用`RLock` | 防止同一FMU实例并发step/reset/close |

新增自动化测试会阻塞一个正在执行的FMU请求，同时发起列车删除，并验证删除必须等待整个fleet tick结束后才能执行。

## 5. 方案对比数据

以下数据均用于技术决策；不同阶段的样本数不完全相同，正式验收以第6节100 tick报告为准。

| 方案 | 1000车FMU链路结果 | 决策 |
|---|---:|---|
| 单9000顺序执行 | 单个20 ms STEP均值99.084 ms；5子步约495 ms | 不满足架构周期 |
| Python线程池并行单进程FMU | STEP均值147.380 ms，较单线程恶化约48.7%；并发instantiate还触发原生段错误 | **否决并回滚** |
| 10分片、JSON全量输入输出 | 稳态FMU约143～170 ms，完整墙钟约186～215 ms | 分片有效但序列化仍重 |
| 20分片、JSON全量输入输出 | FMU均值约152.667 ms，未优于10分片 | **否决**；超过主机有效核心后竞争加剧 |
| 10分片、完整MessagePack | FMU约126～134 ms | 保留，继续压缩中间子步 |
| 10分片、紧凑请求但完整响应 | FMU均值约107.7 ms | 保留，继续压缩中间响应和哈希 |
| 10分片、紧凑请求/响应/哈希/校验 | 正式100 tick：FMU均值87.61 ms、p95 103 ms | 当前正式结果 |

线程池实验表明FMPy/FMU实例不能按普通Python计算任务处理：并发实例化会进入原生库不安全路径，线程调度也会增加锁和上下文切换。当前采用“多进程分片、分片内顺序、分片间并行”。

## 7. 危险项与运维约束

### 7.1 运行中禁止直接修改分片数

当前映射使用`trainId.hashCode() % shardCount`。分片数从10改为12时，大量列车会被重新映射到另一个9000进程。实测在未清理旧实例的情况下有835辆车进入fallback。

改变`FMU_BASE_URLS`数量或顺序必须执行：

1. 暂停中央tick。
2. 对旧分片执行`reset-all`，或按新拓扑对全部列车执行受控RESYNC。
3. 重启9300并确认新拓扑健康。
4. 从新运行的tick 1重新建立实例。
5. 确认全部分片实例总数等于中央列车数后再恢复自动步进。

当前没有一致性哈希和在线实例迁移，不得把修改分片列表当作无损扩缩容。

### 7.2 Docker Compose不会自动创建10个分片

本轮`docker-compose.yml`只向9300透传`FMU_BASE_URLS`和二进制协议开关；正式1000车验收的10个9000进程由独立容器启动。部署环境必须由编排系统提供10个稳定地址，不能只设置10个URL却仍只启动一个9000进程。

示例配置：

```bash
export FMU_BASE_URLS='http://fmu-0:9000,http://fmu-1:9000,http://fmu-2:9000,http://fmu-3:9000,http://fmu-4:9000,http://fmu-5:9000,http://fmu-6:9000,http://fmu-7:9000,http://fmu-8:9000,http://fmu-9:9000'
export FMU_BINARY_PROTOCOL_ENABLED=true
```

### 7.3 100 ms不是当前环境的硬实时保证

正式结果存在少量超过100 ms的内部tick，调用方完整JSON往返更明显超过100 ms。当前可用于加速仿真和客户功能演示，不应作为安全关键实时控制时限证明。下一阶段若要求1000车端到端100 ms p95，需要继续减少9300对8080的重复字段，或为中央链路增加版本化的二进制/增量快照协议。

## 8. 自动化与复现

新增脚本：

```bash
./scripts/test-fmu-1000-scale.py \
  --vehicle http://127.0.0.1:9300 \
  --train-count 1000 \
  --warmup-ticks 2 \
  --sample-ticks 100
```

脚本记录每个tick的墙钟时间、9300内部时间、FMU批次时间和请求/响应字节数，并检查列车集合、数据质量、fallback、FMI错误和负速度。输出默认写入`wp8-1000-train-scale.json`。

## 9. 后续建议

1. 保留10分片作为当前10核验收机默认值，不继续增加分片数。
2. 生产部署固定分片列表顺序，并把分片拓扑作为运行配置版本管理。
3. 将1000车脚本纳入独立容量流水线，不放入每次单元测试；单元测试保留1000车稳定分片和同步门禁。
4. 若客户要求“1000车端到端每100 ms一帧”，下一项工作应优先优化8080与9300的3.82 MB全量响应，而不是继续在单个9000进程内加线程。
