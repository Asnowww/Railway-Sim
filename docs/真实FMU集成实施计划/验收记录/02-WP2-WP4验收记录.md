## 2. WP2验收证据

| 验收项 | 当前证据 | 结果 |
|---|---|---|
| 方程完整性 | 固定OpenModelica 1.27.0检查报告为38个方程、38个变量，无欠定或超定 | 通过 |
| 内部连续状态 | FMU内部维护位置、速度、累计受流电能和累计回馈电能；正常STEP不写外部同名状态 | 通过 |
| 输入类型 | 车门、紧急制动、受流可用性为Boolean；故障码为Integer | 通过 |
| 控制边界 | FMU变量中没有MA、限速、站停或信号控制输入；曲线半径仅保留在9000请求元数据 | 通过 |
| 牵引限制 | 低速受240 kN限制，高速机械功率精确受3.2 MW限制，低供电时满足`P_mech<=P_grid_available×0.88` | 通过 |
| 功率口径 | `mechanicalTractionPowerWatts`为电机侧；`tractionPowerWatts`为受流侧，实测3.2 MW对应3.636363 MW | 通过 |
| 阻力与坡道 | Davis阻力和10‰坡道场景均在解析值1%容差内 | 通过 |
| 制动与再生 | 常用/紧急制动力分别不超过220/300 kN；再生是总制动力子集且不重复扣除 | 通过 |
| 再生预算 | 回馈功率不超过`regenPowerAvailableWatts`；预算为0时回馈为0但总制动力不消失 | 通过 |
| 保护与联锁 | 900 V及以下切除牵引；900～1000 V报告低压；门未锁切除牵引；紧急制动切除牵引 | 通过 |
| 能量积分 | 稳态场景满足`ΔE=P×0.1/3.6e6`，误差不超过1% | 通过 |

上述场景由`fmu-service/tests/test_fmu_runtime.py`直接驱动真实`FMU2Slave`，没有调用Python简化模型。

  -f fmu-service/Dockerfile --target fmu-builder \
  -t railway-sim-fmu-builder-clean:local .

docker build --platform linux/amd64 \
  -f fmu-service/Dockerfile --target test \
  -t railway-sim-fmu-test:local \

# 19 passed in 1.28s

docker build --platform linux/amd64 \
  -f fmu-service/Dockerfile --target runtime \
  -t railway-sim-fmu-runtime:local \
  --build-arg SOURCE_COMMIT=89bd1276f45265cab7bb10baa3902080d2448e1b .

  railway-sim-fmu-runtime:local

## 6. 已知边界与下一阶段入口

- 9300仍运行现有Java物理队列；WP5负责改为Prepare、一次批量9000调用、Apply三阶段。本记录不把9000独立可用误写为生产闭环已接入。
- 9200真实再生预算和同供电段多车耦合由WP6实施；本阶段只验证车辆FMU正确消费瓦特级预算输入。
- `SimpleFallbackModel`仅保留为离线参考，9000在线路由不可达；9300的Java fallback不受影响。
- 本阶段没有修改演示前端或`App.vue`。
