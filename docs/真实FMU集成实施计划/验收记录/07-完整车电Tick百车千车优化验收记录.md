# 完整车电 Tick 百车、千车优化验收记录

## 1. 验收结论

本轮按正式拆分边界贯通并压测完整链路：

`8080 约束编排 -> 9300 车辆控制 -> 10 个 9000 FMU 分片 -> 9200 权威供电步进 -> 8080 只读镜像与 WebSocket 快照`。

100 车和 1000 车正式样本均通过。两个场景各执行 5 个预热 tick 和 20 个采样 tick，合计 25 次供电步进；9200 的 `acceptedStepCount` 均连续增加 25，且每个中央 tick 的 `lastAcceptedTick` 都与中央 tick 相等。未出现 FMU fallback、FMI 错误或供电负荷转发失败。

| 场景 | 完整中央 tick P50 / P95 | HTTP 往返 P50 / P95 | 9300 内部 P50 / P95 | 结果 |

| 1000 车 | 256.066 / 397.160 ms | 272.623 / 411.783 ms | 103.457 / 126.173 ms | 通过 |

## 2. 权威边界和单写约束

完整 tick 的职责固定如下：

1. 8080 计算轨道、信号、调度和联锁约束，并在车辆 tick 前保证 9200 已加载当前线路拓扑。
2. 拆分模式下 8080 不再构造或提交本地供电约束，发给 9300 的供电约束为空。
3. 9300 使用自身保存的权威列车位置向 9200 查询本 tick 的供电约束，再执行车辆控制和 5 次 20 ms FMU 子步。
4. 9300 按供电分区汇总全部列车负荷，并将“分区负荷 + 全部新位置”作为一次原子 step 提交给 9200。
5. 9200 是供电状态和下一周期供电约束的唯一计算方；9300 是拆分模式下每 tick 供电负荷的唯一写入方。
6. 8080 在车辆 tick 后只读取 9200 权威快照，合并到中央快照和 WebSocket，不补写负荷。

首次 tick 另做了干净进程验证：9200 初始 `lastAcceptedTick=-1`、`acceptedStepCount=0`，中央 tick 1 后分别为 `1` 和 `1`。这证明中央只执行一次拓扑初始化，9300 的首次权威 step 不会再被第二次 bootstrap 清空。

## 3. 优化内容

| 优化点 | 修改 | 作用 |
|---|---|---|
| 9300 HTTP 客户端 | 按当前 9200 URL 和超时复用 `RestClient` | 避免每次约束查询和供电 step 重建客户端 |
| 9200 内部紧凑接口 | 新增 `/constraints/query-compact` 和 `/step-compact` | 9300 只接收下一周期约束，不重复返回完整供电快照 |
| 中央供电镜像 | 拆分模式跳过已无写入用途的本地分区负荷聚合 | 消除中央重复计算，保持 9200 单一权威 |
| 供电初始化 | 成功 bootstrap 后保留就绪门禁，权威快照再校验状态 | 防止同一首次 tick 二次 bootstrap 和计数清零 |
| 调度与信号 | 仅在 1 s 调度评价周期计算调度所需的预备 MA；最终信号约束仍每 100 ms 计算 | 去掉非评价 tick 的重复全车信号计算 |
| 告警持久化 | 每 tick 告警改用 JDBC 批量 upsert，并开启 MySQL 批处理重写 | 将千车逐条 SQL 往返合并为批处理 |
| 负荷汇总 | 9300 单线程同步区段使用普通有序 Map | 去掉无意义的并发 Map 开销 |

原有公开供电接口保持不变；紧凑接口只用于 9300 到 9200 的内部高频链路。

## 5. 周期和验收口径

| 层级 | 周期/步长 | 当前语义 |
|---|---:|---|
| FMU | 20 ms | 每个 100 ms 车辆 tick 内执行 5 个子步 |
| 9300 TCMS | 100 ms | 查询约束、车辆控制、FMU、状态应用、向 9200 原子提交一次 |
| 8080 车辆/轨道/信号/供电镜像 | 100 ms | 编排完整 tick；不在拆分模式写供电负荷 |
| 8080 调度评价 | 1000 ms | 只在评价周期执行预备信号和调度计算 |

100 车 P95 已低于 100 ms。1000 车功能、同步和单写语义通过，但本机完整中央 tick P95 为 397.160 ms，因此不能据此声明千车端到端 100 ms 硬实时。快照响应约 2.77 MB、千车告警持久化和调度评价峰值是下一阶段主要压缩方向。

## 6. 复现命令

在隔离的 8080、9300、9200 和 10 个 FMU 分片已启动后执行：

```bash
python3 scripts/test-full-chain-performance.py \
  --train-count 100 \
  --warmup-ticks 5 \
  --sample-ticks 20 \
  --label after-optimization-100 \
  --output docs/真实FMU集成实施计划/验收记录/wp9-full-chain-100-after.json

python3 scripts/test-full-chain-performance.py \
  --train-count 1000 \
  --warmup-ticks 5 \
  --sample-ticks 20 \
  --label after-optimization-1000 \
  --output docs/真实FMU集成实施计划/验收记录/wp9-full-chain-1000-after.json
```

脚本先 reset 并 pause 中央，再逐次调用 `POST /api/simulation/tick`，从而排除自动调度器与人工压测并发推进同一 tick 的干扰。
