### 2.1 故障恢复

执行`test-deployment-recovery.py`，结果为PASS：

1. 两车INIT后均进入真实FMU ACTIVE状态。
2. 删除一个9000实例后，9000批次预检拒绝不完整STEP，9300在同一tick整批Java fallback。
3. 显式`RESYNC`后两车恢复真实FMU且`fallbackTrainCount=0`。
4. 停止9000后9300整批fallback；9000重启后仍保持粘滞fallback，不发生自动热切回。
5. 再次显式`RESYNC`后恢复。
6. 错误参数哈希返回HTTP 409和`FMU_PARAMETER_SET_MISMATCH`。

机器可读故障报告：[wp7-recovery.json](wp7-recovery.json)。

### 2.2 实际联调发现并修复的问题

| 问题 | 真实原因 | 修复 |
|---|---|---|
| 9000首次拒绝9300请求 | 9300把内部`deltaSeconds`重复放入单车DTO，9000严格契约禁止额外字段 | 单车JSON忽略该字段，步长只保留批次`stepSizeSeconds=0.1`，增加回归断言 |
| 独立9200启动后车辆报受流丢失 | 9200未被中央bootstrap时使用内置750 V参考模型 | 启动直接加载项目`power_third_rail.yaml`，健康接口暴露配置哈希、1500 V和5分区 |

## 3. WP8性能与长稳

正式基准使用1、2、10、20车各100个采样tick，随后以20车连续执行6000个0.1 s tick。列车跨5个真实供电段交错布置；同段过载耦合由WP6专项测试负责。

| 车数 | FMU批次p95 | 完整车辆/供电周期p95 | fallback tick | FMI错误 | 新增漏100 ms截止时间 |
|---:|---:|---:|---:|---:|---:|
| 1 | 6.00 ms | 16.00 ms | 0 | 0 | 0 |
| 2 | 5.00 ms | 14.00 ms | 0 | 0 | 0 |
| 10 | 6.00 ms | 16.05 ms | 0 | 0 | 0 |
| 20 | 7.00 ms | 17.05 ms | 0 | 0 | 0 |
| 20车6000 tick | 7.00 ms | 16.00 ms | 0 | 0 | 0 |

长稳仿真时间600.0 s，墙钟执行193.134 s；HTTP往返p95为26.625 ms，9300内部完整周期最大60 ms，未漏截止时间。每步列车集合、实例隔离和关键物理量有限性检查均通过。`degradedTicks=3144`来自真实欠压/限流等物理保护状态，不是Java fallback或FMI执行错误，单独保留用于后续供电容量分析。

机器可读性能报告：[wp8-performance.json](wp8-performance.json)。

1000车扩展压测、分片与二进制通信优化属于后续容量专项，详见[1000车FMU通信与同步优化验收记录](06-1000车通信与同步优化验收记录.md)。该专项在功能和实例同步方面通过，但在`linux/amd64`兼容执行环境中的100 ms硬截止时间p95尚未完全通过，不改变本节原20车验收结论。
