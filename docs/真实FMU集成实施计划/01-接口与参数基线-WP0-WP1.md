# 分计划01：接口与参数基线（WP0～WP1）

>
> 执行边界：只冻结跨服务契约并统一车辆参数，不接入真实FMU，不改变当前Java fallback主路径。

> 阶段状态：**[已完成]**。验收记录见[01-WP0-WP1验收记录](验收记录/01-WP0-WP1验收记录.md)。

## 1. 阶段目标

完成9300、9000、9200之间的字段、单位、生命周期、机械/电功率口径和错误协议冻结，并使`config/train_params.yaml`成为9300和9000实际读取的车辆参数事实来源。

本阶段结束时必须满足：

- 3.2 MW在全部契约中只表示电机侧最大机械功率。
- `tractionPowerWatts`和`regenPowerWatts`保持电网侧功率语义。
- `mechanicalTractionPowerWatts`、`mechanicalRegenPowerWatts`和`regenPowerAvailableWatts`进入冻结契约。
- 9300不再硬编码车辆质量、力、功率、效率、Davis系数和车辆保护电压。
- 9300和9000对同一YAML计算相同`parameterSetId`。

## 2. WP0：接口、单位和功率口径冻结

### 2.1 实施项

1. 将`config/fmu_mapping.yaml`拆为`parameters`、`initialState`、`stepInputs`、`metadata`、`outputs`五组。
2. 从FMU必需输入中移除MA、限速、站距、停车距离、状态机状态和曲线半径；这些字段只保留为请求元数据。
3. Boolean输入固定为FMI Boolean，故障码固定为FMI Integer并由9000映射为字符串。
4. 增加9000请求和响应JSON Schema，覆盖总计划第7节全部字段、范围、生命周期命令和错误信封。
5. 同步9300、9000、9200 DTO草案：
   - 9300→9000新增`lifecycleCommand`、`modelVersion`、`parameterSetId`、`traceId`、`regenPowerAvailableWatts`和`currentCollectionAvailable`。
   - 9000→9300新增两个机械功率诊断字段、实例状态、数据质量和FMI状态。
   - 9200→9300新增`regenPowerAvailableWatts`。
6. 增加向后兼容测试：现有8080和前端忽略新增字段时仍可工作。

### 2.2 交付物

- 更新后的`config/fmu_mapping.yaml`。
- `fmu-service/contracts/step-fleet-request.schema.json`。
- `fmu-service/contracts/step-fleet-response.schema.json`。
- 三服务字段追踪测试和JSON契约测试。

### 2.3 验证命令

```bash
PYTHONPATH=fmu-service python -m unittest discover -s fmu-service/tests -p 'test_contract*.py'

```

### 2.4 退出条件

- 所有必需变量都有名称、类型、单位、来源、更新频率和消费者。
- 示例JSON同时通过Schema和三服务反序列化测试。
- 同一字段不再同时表示机械功率和电功率。
- 不改变当前Java fallback执行结果。

### 2.5 回滚

恢复旧映射和新增DTO；因为本工作包不切换物理执行器，回滚后9300继续使用Java fallback。

## 3. WP1：YAML车辆参数统一

### 3.1 实施项

1. 9300增加Jackson YAML依赖和`vehicle-runtime.train-params-path`，默认指向`config/train_params.yaml`。
2. 建立不可变车辆参数对象，校验：
   - 质量、力、功率和电压大于0；
   - 牵引/再生效率在`(0,1]`；
   - `cutoffVoltage < minVoltage <= nominalVoltage`；
   - `regenBrakeRatio`在`[0,1]`。
3. 用参数对象替换`VehicleLoadPolicy`和`VehicleSimulationQueue`中的车辆硬编码常量。
4. 9300和9000按YAML原始字节计算`sha256:<64位十六进制>`格式的`parameterSetId`。
5. 9000启动时读取同一YAML；请求参数哈希不匹配时返回HTTP 409。
6. 保持`trainMassKg`为9300每步计算并发送的总质量；载荷变化不要求重建FMU实例。

### 3.2 参数验收基线

| 参数 | 当前值 | 单位 |
|---|---:|---|
| 空车质量 | 198000 | kg |
| 最大载荷质量 | 72000 | kg |
| 最大机械牵引功率 | 3200000 | W |
| 最大牵引力 | 240000 | N |
| 牵引效率 | 0.88 | 1 |
| 常用/紧急制动力 | 220000 / 300000 | N |
| 再生比例/效率 | 0.45 / 0.35 | 1 |
| Davis A/B/C | 1800 / 45 / 3.2 | 对应SI单位 |
| 额定/最低/切除电压 | 1500 / 1000 / 900 | V |

### 3.3 验证命令

```bash
mvn -f vehicle-runtime-service/pom.xml test
PYTHONPATH=fmu-service python -m unittest discover -s fmu-service/tests -p 'test_parameters*.py'
shasum -a 256 config/train_params.yaml
```

### 3.4 退出条件

- 修改任一YAML参数后，9300和9000产生相同的新哈希。
- 非法参数使服务启动失败，并在错误中指出精确YAML路径。
- 9300源码中不再存在上述车辆参数的重复硬编码。
- Java fallback回归测试通过。

### 3.5 回滚

本阶段必须整体回滚参数加载器和常量替换，禁止出现部分参数来自YAML、部分参数来自旧常量的混合状态。
