# 分计划04：部署、验收与收口（WP7～WP8）

> 上级计划：[真实车辆FMU集成实施计划](../真实FMU集成实施计划.md)
>
> 前置条件：9300、9000、9200的100 ms多车闭环已通过。
>
> 执行边界：完成容器编排、故障恢复、性能和回归验收；不修改`App.vue`。

## 1. WP7：容器编排和故障恢复

### 1.1 实施项

1. 扩展`docker-compose.yml`，加入9000、9200、9300和8080，保留现有数据库服务。
2. 9000和9300使用同一车辆YAML；9000镜像包含固定FMU制品。
3. 启动顺序：数据库、9200和9000健康后启动9300，9300健康后启动8080。
4. 停止顺序：先8080/9300，再9000/9200，确保9000释放全部FMU实例。
5. 配置连接和读取超时、健康检查、重启策略及Java fallback开关。
6. 暴露模型版本、参数哈希、实例数、批次时延、FMI错误、fallback数量、重复/乱序tick等指标。
7. 增加9000停机、单实例失败、参数不匹配和恢复后RESYNC故障注入脚本。

### 1.2 验证命令

```bash
docker compose config
docker compose build
docker compose up -d
docker compose ps
./scripts/test-fmu-failure-recovery.sh
docker compose down
```

### 1.3 退出条件

- 全新环境可以一次启动四个服务并通过健康检查。
- 9000停机时9300整批fallback；单实例失败时只回退该车。
- 恢复后不自动热切回，显式RESYNC后状态连续。
- 参数哈希不一致时系统明确拒绝FMU步进并降级。

### 1.4 回滚

使用Compose profile或`vehicle-runtime.physics-mode=JAVA_FALLBACK`禁用9000依赖；不删除数据库卷。

## 2. WP8：系统验收

### 2.1 阻断级场景

- FMI 2.0 Co-Simulation、linux64二进制和变量描述校验。
- INIT、STEP、RESET、RESYNC、删除、重复tick、冲突tick和乱序tick。
- 零牵引、恒牵引、恒功率、坡道、常用/紧急制动、欠压、门联锁和再生限制。
- 一份FMU多实例隔离。
- 同区段多车物理耦合和不同区段隔离。
- 9000整体故障、单实例故障、参数漂移和恢复重同步。
- 8080、9300、9200、9000及前端回归。

### 2.2 量化门槛

- 机械牵引功率不超过3.2 MW。
- 电网侧牵引功率与机械功率/0.88误差不超过1%。
- 电网回馈功率与机械再生功率×0.35误差不超过1%。
- 20车9000批量p95不超过50 ms。
- 完整车辆与供电周期p95不超过80 ms。
- 20车连续10分钟无100 ms截止时间遗漏、tick倒退、实例串扰、NaN或Infinity。
- 稳态功率和能量积分误差不超过1%。

### 2.3 验证命令

```bash

./scripts/test-fmu-acceptance.sh

mvn -f backend/pom.xml test
PYTHONPATH=power-network-service python -m app.self_test
npm --prefix frontend run build
docker compose down
```
