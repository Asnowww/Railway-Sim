# 9 号线真实线路数据接入 —— 开发方案

> 数据源：《视景系统公里标最新数据+站台位置20260703.xlsx》（OOXML strict 格式，openpyxl 需特殊处理）
> 现状基线：backend 以 `line-topology-demo.yaml`（虚构 6 站 6.25km）运行；前端综合态势图为 App.vue 内写死坐标的拓扑。
> 撰写日期：2026-07-13

## 一、新数据解读（已完成解析）

### 1.1 三张表的数据字典

**Sheet1「上行视景边号公里标」（13 条边）/ Sheet2「下行视景边号公里标」（12 条边）**

| 列 | 含义 |
|---|---|
| 视景边号 | 视景系统 UDP 协议中的 2 字节 segNo（对应后端 `TrackSegmentDefinition.rawSegmentId`） |
| 起始/终点公里标(m) | 该边在本方向线位上的里程范围 |
| 长度(m) | 终点-起始 |

- 上行边号序列：`2,10,13,16,19,20,23,27,34,38,42,46,48`，覆盖 0 → **16489.02m**
- 下行边号序列：`3,4,7,11,14,17,21,24,28,36,43,47`，覆盖 0 → **16484.57m**
- 上下行边号**不重叠**（视景系统按物理股道独立编号），同一区间两方向里程略有差异（线位不同）——**必须按两条独立股道建模，不能共用一套区段**。

**Sheet3「站台位置」（26 行 = 13 站 × 2 股道）**

| 列 | 含义 |
|---|---|
| Track | `1`=上行股道，`0`=下行股道 |
| StaID | 101~113 |
| StationName | 郭公庄、丰台科技园、科怡路、丰台南路、丰台东大街、七里庄、六里桥、六里桥东、北京西站、军事博物馆、白堆子、白石桥南、国家图书馆 |
| StationKm | 站中心公里标（精确停车目标） |
| StopLeftKm / StopRightKm | 站台两端公里标（恒为中心 ±59m，站台长 118m —— 与 B 型 6 编组车长 `TrainState.lengthMeters=118` 精确吻合） |
| PlatSide | 站台侧（left/right），上下行同站相反 |

即：**北京地铁 9 号线全线**（参考图：国家图书馆↕郭公庄）。现有 `line-ggz.yaml` 只是郭公庄站场 0~4028m 切片；`power_third_rail.yaml` 的分区命名（西段-GGZ至KYL、FSP丰山坡等）本来就按这条线设计——新数据是对既有演示线的**全线替换**，不是新增线路。

### 1.2 与现有系统的字段落点

| xlsx 字段 | 系统落点 | 现状 |
|---|---|---|
| 视景边号 | `TrackSegmentDefinition.rawSegmentId` → `VisionUdpPacketBuilder` 发包边号 | 字段已存在，但 **YAML 加载路径不解析**（`YamlLineDataLoader.RawSeg` 没有该键），yaml 线路下发包边号恒为 0 |
| 起止公里标 | `segments[].start_meters/end_meters` | 直接支持 |
| Track 0/1 | `segments[].track`（自由标签，引擎已支持同里程平行 track，见 topology-demo 的 north/loop） | 直接支持 |
| StationKm | `stations[].position_meters` → `StationDefinition.centerMeters` → 9300 bootstrap 停站判定 | 支持，但**每站只有一个值**，无法表达上下行两个不同的 StationKm |
| StopLeft/RightKm、PlatSide | 无对应字段 | **模型缺失**（`StationDefinition` 只有 id/name/centerMeters/platformIds） |

## 二、现状差距（按层）

**后端（4 条）**
1. `YamlLineDataLoader` 不解析 `raw_segment_id` → 视景 UDP 边号错误（发 0）。
2. `StationDefinition` 单中心值：上下行 StationKm 差最多 ~20m（六里桥 8199 vs 8179），单值会让某一方向停站判定/DMI"距前方车站"偏移；停车窗口（±59m 站台端部）与 9300 `stopping-control.station-stop-window-meters=10` 的语义需要区分（前者是站台物理范围，后者是精停容差，**不冲突**，但站台范围值得进模型供前端画站台、供越站判定用）。
3. 两端无折返/渡线定义：直线双向线到达终点即堵死（已在联调中实证）。上下行独立建模后，若不加终点联络，上行车到国家图书馆、下行车到郭公庄后永久占用末端区段。
4. `power_third_rail.yaml` 分区里程（P01 0-4000m 等 5 区，总长约 16.5km 需核对）要与新线对齐。

**前端（3 条）**
5. 当前态势图（App.vue 内写死 `topologyNodes/topologyEdges` 坐标、百分比投影）与真实拓扑完全脱钩——**必须改为 `/api/infrastructure/topology` 数据驱动**，这是本方案前端部分的核心。
6. 无"上下行双线"版式：需要参考图的直线双线横排（竖图横置）。
7. 16.5km 全线站间距极不均（最长站间 5.49km 军博→北京西站边 34？实际最长站间约 1.9km；但首站间 1.35km vs 短站间 0.79km），真实里程比例下站名会挤压重叠，需要"等距站点"显示模式。

## 三、目标设计

### 3.1 新线路配置 `config/line-m9.yaml`（由转换脚本生成）

```yaml
line:
  id: m9
  name: 北京地铁9号线（郭公庄—国家图书馆）
  length_meters: 16490

stations:            # 主值取上行 StationKm；platforms 为扩展键（阶段2）
  - id: S101
    name: 郭公庄站
    position_meters: 372.0
    platforms:                       # ← 扩展：双向站台
      - track: up
        center_meters: 372.0
        stop_left_meters: 313.0
        stop_right_meters: 431.0
        side: left
      - track: down
        center_meters: 372.0
        stop_left_meters: 313.0
        stop_right_meters: 431.0
        side: right
  # ... S102~S113 同构

nodes:               # 每方向线性链，脚本按边界自动生成
  - id: NU00        # 上行 0m
  - id: NU01        # 上行 215.49m（边2/边10 分界）
  # ...
segments:
  # ===== 上行（郭公庄→国家图书馆），id = U+视景边号 =====
  - id: U02
    from: NU00
    to: NU01
    start_meters: 0
    end_meters: 215.49
    raw_segment_id: 2              # ← 扩展键：视景边号
    speed_limit_meters_per_second: 22.2
    track: up
  # ... U10 U13 U16 U19 U20 U23 U27 U34 U38 U42 U46 U48
  # ===== 下行（国家图书馆→郭公庄），id = D+视景边号 =====
  - id: D03
    ...
    track: down
```

限速：xlsx 未提供，先统一 80km/h（22.2m/s），车站邻接边可降 60km/h，后续有数据再精化。

### 3.2 后端小改（两处，均为向后兼容的加法）

1. **`YamlLineDataLoader`**：`RawSeg` 增加 `raw_segment_id`（缺省 0），透传给 `TrackSegmentDefinition.rawSegmentId` → 视景 UDP 边号立即正确。
2. **站台扩展**：`YamlStation` 增加可选 `platforms[]`；`StationDefinition` 增加 `List<PlatformDefinition>`（track/center/stopLeft/stopRight/side）。消费方：
   - 9300 bootstrap `stations[]` 按列车 direction 取对应 track 的 center（修复上下行 20m 偏差）；
   - `/api/infrastructure/topology` 原样输出 platforms，供前端画站台矩形与停车窗口。
   - 不改任何既有字段 → 旧 yaml（topology-demo 等）零影响。

### 3.3 终点折返（消除"终点堵死"）

两端各加一条折返渡线（switch + 短 segment 连接上/下行末端节点）+ 两条 route（`R_TB_GGZ`、`R_TB_NLIB`）。这是让双向线**连续循环运营**的最小拓扑；折返调度逻辑（AR 模式/自动折返命令）后端已有 turnbackInfo 语义钩子，首期可先由调度 REQUEST_ROUTE 手动触发折返进路，自动折返列为二期。

### 3.4 前端态势图（横向双线，参考图竖转横）

新建独立组件（替换 App.vue 内写死拓扑的图区），数据全部来自 `/api/infrastructure/topology` + WS 快照：

```
国图 ◄─────────────────────────────────────────────── 下行 track=down
      ●白石桥南 ●白堆子 ●军博 ●北京西站 ●六里桥东 ●六里桥 ……●郭公庄
      ────────────────────────────────────────────────► 上行 track=up
      [P05][P04]        [P03]        [P02]        [P01]   ← 供电分区带
```

- **两条水平主线**：上行在下、下行在上（或按运营习惯翻转，做成配置项），间距 ~60px；两端画折返渡线斜线。
- **站点渲染**：站名一排居中标注；每股道在 StopLeft→StopRight 画站台矩形（PlatSide 决定画线上侧/下侧），站中心画停车标。
- **双显示模式**（工具栏切换）：
  - `里程比例`：x = 公里标线性映射（工程视角，区段长度真实）；
  - `等距站点`：13 站均分 x 轴，站间区段按边数等分（示意图视角，参考老师给的图）——站名不重叠，默认此模式。
  - 两种模式共用同一投影函数 `xOf(track, meters)`，图层代码不感知模式。
- **图层沿用现有语义**：区段占用四态着色（id 旁标注视景边号 tooltip）、信号机色+形状、道岔菱形、列车按 `direction` 落到对应股道（DOWN→track up 的映射跟随后端 launch 约定，实施时以实车验证为准）、MA 带、供电分区带（底部单条，按 P01~P05 里程）。
- **交互**：悬停区段显示"U23 · 视景边23 · 5146.6–8352.2m · 占用/限速"；点击站台显示双向 StationKm/停车窗口。

### 3.5 视景系统对帧（数据的另一半价值）

- **协议语义已确认（2026-07-13）**：4B"车头位置" = **相对该视景边起点道岔的位移，单位 mm**（边内偏移）。
- 现有实现已符合该语义：`VisionUdpPacketBuilder.segmentOffsetMeters()` 就是 `绝对里程 − 边起点`（clamp 到 [0, 边长]），无需改换算——加载器补上 `rawSegmentId` 后，边号 + 边内 mm 位移即完整正确。
- **双线适配的真正改动点**：`resolveSegment()` 目前按里程匹配区段；上下行区段建模后**里程区间重叠**，必须按列车 `direction`（或 track 标签）过滤各自方向的边表再匹配，否则下行车会被算到上行边号上。这是 P4 唯一的代码修改（一处过滤条件）。
- 100ms 周期推送目前无内建调度（联调备忘已记录），若现场需要连续推流，加一个可开关的 `@Scheduled` 即可。

## 五、风险与决策点

1. **前端架构现状**：当前 App.vue 是写死拓扑的单文件实现（此前重构版已被团队分支覆盖）。P3 有两种做法：(a) 只重写图区组件、外壳不动（方案默认，侵入最小）；(b) 借机恢复 topology 驱动的组件化结构。**建议 (a) 先落地，避免再次与团队分支冲突**——请与团队确认合并策略后再动 App.vue。
2. **direction↔track 映射**：现有 launch 请求 direction=DOWN 对应哪条股道，需在 P2 用实车验证后固化为常量，避免上下行画反。
3. **上下行公里标不同源**：跨方向比较（如追踪间隔）必须先归一到同一参考系，P2 中在 topology API 输出每站双向 StationKm，前端/调度按 track 各自计算，不做跨轨直接相减。
4. ~~视景"车头位置mm"语义~~ **已确认：相对起点道岔的边内位移（mm）**，现有 `segmentOffsetMeters()` 实现即正确；风险转移为 resolveSegment 的方向过滤（见 §3.5）。
5. **xlsx 为 OOXML strict**：openpyxl 原生打不开（已踩坑），转换脚本必须用已验证的手写解析器，不要依赖 pandas/openpyxl 默认路径。
