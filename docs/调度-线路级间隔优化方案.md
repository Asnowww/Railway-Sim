# 调度-线路级间隔优化方案

> 刘倩 | 2026-07-13 | 基于“本车命令”的线路级时间间隔优化

---

## 1. 目标

线路级优化用于把时间间隔调度从“单车独立响应”升级为“全线统一决策”。

核心原则：

```text
线路级优化只负责统一选择哪些车需要调度。
最终执行仍然输出多条只作用本车的 DispatchCommand。
每条命令的 trainId / regulatedTrainId 必须指向同一辆本车。
frontTrainId 只作为本车间隔参考对象，不直接控制前车。
```

因此，线路级优化不是新增“一条线路命令控制多辆车”，而是：

```text
全线状态
  -> 线路级优化器
  -> 线路级调节方案 LineRegulationPlan
  -> 多条本车 DispatchCommand
  -> 车辆/信号/调度约束执行链路
```

---

## 2. 为什么需要线路级优化

当前本车调度已经解决了“每条命令到底调谁”的问题，但单车策略仍有三个不足：

| 问题 | 单车策略表现 | 线路级优化目标 |
|---|---|---|
| 局部动作互相打架 | 相邻车可能一辆追赶、一辆放慢 | 统一筛选动作组合 |
| 动作过多 | 多个扰动同时打开时每个都生成命令 | 每轮只处理最关键的 1~2 辆车 |
| 信号不可执行时仍生成追赶 | 本车想追赶，但 MA/进路不允许 | 降级为观察并说明信号约束 |

线路级优化的价值不是替代本车命令，而是让本车命令更少、更稳、更有整体目标。

---

## 3. 输入数据

线路级优化器使用调度总循环中的现有状态，不要求信号轨道侧新增接口。

```text
simulationRunId
simulatedAt
currentPlan
trainProfiles
trains
authorities
openDisturbances
activeCommands
routeReservations
```

关键字段：

| 数据 | 用途 |
|---|---|
| `TrainRunProfile.headwayActualSec` | 本车相对前车的实际间隔 |
| `TrainRunProfile.headwayDeviationSec` | 实际间隔 - 目标间隔 |
| `TrainRunProfile.frontTrainId` | 本车参考前车 |
| `TrainRunProfile.status / speedMps` | 判断本车在站还是运行中 |
| `MovementAuthority.speedLimitMetersPerSecond` | 判断追赶是否受信号限速约束 |
| `MovementAuthority.authorityEndMeters` | 判断 MA 距离是否足够 |
| `activeCommands` | 避免同车重复调度和动作震荡 |
| `routeReservations` | 判断进路请求/接受/拒绝状态 |

---

## 4. 输出数据

线路级优化器输出一个 `LineRegulationPlan`，用于解释本轮调度方案。

```text
LineRegulationPlan
  planId
  generatedAt
  objective
  status
  targetHeadwaySec
  currentMaxAbsHeadwayErrorSec
  predictedMaxAbsHeadwayErrorSec
  decisions[]
```

每个 `LineRegulationDecision` 对应一辆本车：

```text
LineRegulationDecision
  trainId
  regulatedTrainId
  frontTrainId
  action
  commandType
  status
  reason
  currentHeadwaySec
  targetHeadwaySec
  currentHeadwayErrorSec
  predictedHeadwayErrorSec
  priorityScore
  signalConstraint
  commandId
```

其中：

```text
trainId = regulatedTrainId
```

第一版必须保持这个约束，避免重新出现“调度对象混乱”。

---

## 5. 第一版规则

第一版采用规则型启发式优化，不引入复杂求解器。

### 5.1 候选车筛选

只处理有有效间隔偏差或晚点恢复需求的列车：

```text
abs(headwayDeviationSec) > tolerance
或 headwayState = TOO_SHORT / TOO_LONG
或 openDisturbance.headwayDirection = SCHEDULE_LATE
```

容差取：

```text
max(5s, targetHeadwaySec * 0.1)
```

### 5.2 优先级

优先处理偏差最大的车：

```text
priorityScore = abs(headwayErrorSec)
```

如果该车已有活动调度命令：

```text
PENDING / SENT / APPLIED
```

则本轮降级为 `OBSERVE`，避免重复下发。

### 5.3 每轮动作上限

第一版每轮最多生成 2 条本车命令。

```text
maxActionsPerRound = 2
```

这样可以避免扰动同时出现时动作过多。

### 5.4 间隔过小

业务含义：

```text
本车离前车太近，需要扩大间隔。
```

动作：

| 本车状态 | 动作 |
|---|---|
| 在站/停车 | `EXTEND_DWELL` |
| 运行中 | `SPEED_BIAS < 1` |
| 已有有效信号低速约束 | `OBSERVE` |

### 5.5 间隔过大

业务含义：

```text
本车离前车太远，需要缩小间隔。
```

动作：

| 本车状态 | 动作 |
|---|---|
| 在站/停车 | `SHORTEN_DWELL` |
| 运行中且 MA 允许 | `SPEED_BIAS > 1` |
| 运行中但 MA 不足或限速过低 | `OBSERVE` |

### 5.6 本车晚点

本车晚点不直接等于追赶，必须结合间隔：

| 间隔状态 | 动作 |
|---|---|
| 间隔过小 | `OBSERVE`，不追赶 |
| 间隔正常 | 轻微 `SPEED_BIAS > 1` 或缩短停站 |
| 间隔过大 | `SHORTEN_DWELL` / `SPEED_BIAS > 1` |

### 5.7 相邻车冲突抑制

第一版只做轻量抑制：

```text
相邻两辆车不同时生成相反方向的强动作。
如果前一辆已被选中，后一辆候选动作与其相反，则后一辆降级观察。
```

例如：

```text
TR-002 本车放慢
TR-003 本车追赶
```

如果它们相邻且都属于强动作，第二个动作会被降级为观察，避免局部震荡。

---

## 6. 信号边界

线路级优化不改变信号轨道职责。

调度侧可以判断：

```text
MA 距离是否充足
MA 限速是否过低
进路是否被拒绝或等待
```

但调度侧不能越过：

```text
信号移动授权
联锁进路
轨道限速
车辆动力学限制
```

当追赶动作被信号约束限制时，线路级方案应显示：

```text
action = OBSERVE
signalConstraint = MA_LIMITED / ROUTE_BLOCKED / SPEED_LIMITED
reason = 本车追赶受信号约束，等待 MA/进路条件改善
```

## 7. 前端展示建议

调度页新增“线路级调节方案”面板：

```text
线路级目标：恢复目标间隔
目标间隔：300s
当前最大偏差：180s
预计最大偏差：95s
本轮动作：2 条
状态：执行中 / 观察 / 无需调节
```

决策列表：

```text
TR-002 本车放慢
参考前车 TR-001
实际/目标 150s / 300s
预计偏差 -150s -> -95s
命令 SPEED_BIAS 0.75
信号约束 NONE

TR-003 观察
参考前车 TR-002
原因：相邻动作冲突，等待下一轮
```

这样前端既能展示线路级方案，也能保持每条动作只作用本车。

---

## 8. 验收标准

| 场景 | 期望 |
|---|---|
| 单辆车间隔过小 | 生成本车放慢或延长停站 |
| 单辆车间隔过大 | 生成本车追赶或缩短停站 |
| 多辆车同时异常 | 每轮最多生成 2 条命令 |
| 同车已有活动命令 | 本轮观察，不重复下发 |
| 相邻车动作冲突 | 后选动作降级观察 |
| 追赶受 MA 限制 | 不强行追赶，显示信号约束 |
| 快照展示 | `lineRegulationPlan` 能解释目标、指标、决策和命令 |

---

## 9. 后续增强

第一版完成后可继续增强：

```text
预测到站/发车时间 ETA
predictedHeadwaySec
headwayTrend
更细的 speedLimitSources
进路失败 conflictSegmentIds / conflictTrainIds
线路级优化冷却窗口
多方向/折返场景支持
```

这些增强不改变“本车命令”的执行边界。
