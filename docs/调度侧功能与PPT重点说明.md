# 调度侧功能与重点说明

> 本文档用于组长制作阶段汇报 PPT，重点说明调度侧已经实现的功能、核心闭环、与信号轨道侧的对接关系，以及当前仍需完善的边界。

## 1. 调度侧定位

调度侧负责在仿真运行过程中读取线路、车辆、信号授权等实时状态，结合运营计划和当前扰动情况，生成调度决策与进路申请。

当前调度侧不负责创建车辆，也不直接控制轨道设备。车辆由轨道/车辆仿真侧创建，轨道资源和进路由信号轨道侧维护。调度侧主要完成以下职责：

- 读取车辆实时状态，识别上线车辆、位置、速度、站点、故障等级等信息。
- 读取运营计划，掌握目标发车间隔、默认停站时间、计划车次和线路交路。
- 监测列车运行间隔、晚点、停站、拥挤等状态，识别扰动。
- 生成时间间隔调度命令，例如放慢、追赶、扣车、调整停站等。
- 生成路线/交路计划，并按计划向信号侧申请进路。
- 记录调度命令、进路申请、信号反馈、调度效果指标，支撑前端展示与闭环评估。

调度侧的核心目标是：**从“看见列车状态”到“生成调度动作”，再到“接收执行反馈并评估效果”，形成闭环调度能力。**

## 2. 已实现功能概览

### 2.1 运营计划接入

调度侧已经接入当前运营计划，能够根据仿真时间解析对应时段的运行参数。

主要能力：

- 读取当前计划 ID、线路 ID、运行时段。
- 获取目标发车间隔 `departureIntervalSec`。
- 获取默认停站时间 `defaultDwellTimeSec`。
- 向前端提供运营计划摘要、车次计划、间隔目标等信息。

该能力为时间间隔调度和车队交路调度提供基础参数。

### 2.2 时间间隔闭环调度

时间间隔调度主要关注本车与前车之间的运行间隔是否偏离计划目标。

调度侧已经实现：

- 读取列车实时位置、速度、站点、停站状态。
- 计算本车与前车的实际间隔。
- 与计划目标间隔进行对比。
- 判断间隔过大、过小或基本正常。
- 基于本车生成调度动作，而不是同时混合调整前车/后车，逻辑更清晰。

当前支持的典型调度动作：

| 场景 | 调度动作 | 目的 |
|---|---|---|
| 本车与前车间隔过小 | 放慢本车、延长停站、扣车 | 拉大间隔，避免追踪过近 |
| 本车与前车间隔过大 | 追赶前车、缩短停站、释放扣车 | 缩小间隔，恢复均衡 |
| 本车晚点 | 提高运行优先级、缩短停站 | 尽量恢复计划 |
| 本车正常 | 保持观察 | 避免过度干预 |

前端已经有调度评估面板，用于展示：

- 当前目标间隔。
- 实际间隔。
- 间隔偏差。
- 当前调度动作。
- 预测改善效果。
- 调度命令执行状态。

当前时间间隔调度已经具备演示闭环，但还需要更多信号/轨道反馈来让效果更真实。

### 2.3 线路级间隔优化

在线路级优化中，调度侧不再只根据单车局部状态生成一个命令，而是从全线列车间隔出发，生成一组协调动作。

当前设计思想：

- 每辆车仍然只接受作用于本车的命令。
- 线路级优化器统一判断全线间隔偏差。
- 对多辆车分别生成本车调度动作。
- 通过多条本车命令组合实现线路级均衡。

这样可以同时满足两个要求：

- 调度动作仍然清晰可解释，每条命令只作用于一辆车。
- 优化目标升级为全线均衡，而不是只处理单个局部间隔。

### 2.4 路线调度与车队交路循环

调度侧已经实现车队交路循环计划。

当前标准循环交路为：

```text
R_UP -> R_TB_LIB -> R_DOWN -> R_TB_GGZ
```

四条路线含义如下：

| routeId | 含义 |
|---|---|
| `R_UP` | 郭公庄到国家图书馆，上行正线 |
| `R_TB_LIB` | 国家图书馆折返，上行转下行 |
| `R_DOWN` | 国家图书馆到郭公庄，下行正线 |
| `R_TB_GGZ` | 郭公庄折返，下行转上行 |

车辆初始点识别规则：

| 车辆初始位置 | 起始路线 | 后续循环 |
|---|---|---|
| 郭公庄 `S101` | `R_UP` | `R_UP -> R_TB_LIB -> R_DOWN -> R_TB_GGZ` |
| 国家图书馆 `S113` | `R_DOWN` | `R_DOWN -> R_TB_GGZ -> R_UP -> R_TB_LIB` |

也就是说，上下行可以理解为两条单行道，调度侧按车辆起点自动分配对应交路。

### 2.5 车辆自动识别与自动分配交路

调度侧现在不要求前端手动选择车辆路线，而是根据轨道/仿真侧传来的车辆状态自动识别。

车辆自动分配条件：

- 车辆 ID 不为空。
- 车辆故障等级 `faultLevel <= 1`。
- 该车辆此前没有生成过车队交路。
- 车辆位于两个端点之一：郭公庄 `S101` 或国家图书馆 `S113`。

端点识别有两层：

1. 优先识别 `currentStationId`。
2. 如果 `currentStationId` 缺失，则根据 `positionMeters` 靠近端点米标进行兜底判断。

当前端点米标判断逻辑：

- 靠近 `R_UP.entryMeters` 或 `R_DOWN.exitMeters`，认为车辆在郭公庄 `S101`。
- 靠近 `R_UP.exitMeters` 或 `R_DOWN.entryMeters`，认为车辆在国家图书馆 `S113`。
- 容差为 `150m`。

自动分配入口：

- 调度每个 tick 自动检查新上线端点车辆。
- 前端保留“按车辆起点生成交路”按钮，用于演示或手动补扫。

自动生成参数：

| 参数 | 当前默认值 |
|---|---|
| 循环次数 | 2 |
| 发车间隔 | 当前运营计划 `departureIntervalSec` |
| 首车等待 | 0 秒 |

### 2.6 进路申请与信号侧连接

车队交路生成后，调度侧不会直接改变轨道资源，而是将当前交路 leg 转换为 `OperationPlan`。

到计划发车时间后，调度侧自动生成 `REQUEST_ROUTE` 命令，向信号侧申请进路。

典型命令字段如下：

```json
{
  "source": "CIRCULATION_PLAN",
  "operationPlanId": "OP-xxx",
  "circulationId": "CIRC-xxx",
  "circulationLegId": "CIRC-xxx-L01",
  "cycleIndex": 0,
  "legIndex": 0,
  "routeId": "R_UP",
  "direction": "UP",
  "originPointId": "S101",
  "destinationPointId": "S113",
  "segmentIds": ["U02", "U10"],
  "plannedDepartureAt": "..."
}
```

信号侧返回进路申请结果后，调度侧会更新计划状态。

当前已支持的信号反馈包括：

- 进路申请接受。
- 进路申请拒绝。
- 失败原因。
- 是否可重试。
- 进路状态记录。

## 3. 调度闭环主流程

当前调度闭环可以概括为：

```mermaid
flowchart LR
  A["轨道/车辆侧创建车辆"] --> B["仿真快照 trains"]
  B --> C["调度侧识别车辆位置与状态"]
  C --> D["自动分配车队交路"]
  D --> E["生成 OperationPlan"]
  E --> F["到点生成 REQUEST_ROUTE"]
  F --> G["信号侧校验并建立进路"]
  G --> H["返回 accepted / rejected"]
  H --> I["调度侧更新交路状态"]
  I --> J["前端展示计划、命令、反馈与效果指标"]
```

时间间隔调度闭环可以概括为：

```mermaid
flowchart LR
  A["列车实时状态"] --> B["计算实际间隔"]
  B --> C["对比目标间隔"]
  C --> D["识别过大/过小/晚点"]
  D --> E["生成本车调度动作"]
  E --> F["命令进入调度队列"]
  F --> G["车辆/信号/轨道状态变化"]
  G --> H["调度侧评估效果"]
```

## 4. 前端调度页面重点

当前前端调度页面主要用于展示调度闭环过程，而不是重复展示完整线路拓扑。

重点展示内容：

- 当前运营计划和目标间隔。
- 上线车辆状态。
- 时间间隔偏差。
- 调度动作和命令状态。
- 车队交路循环计划。
- 当前 leg、下一发车、循环进度。
- 进路申请和信号反馈。
- 调度前后关键指标变化。

前端展示原则：

- 首页/总览页展示全局状态。
- 调度页重点展示调度决策和效果评估。
- 路线图不作为调度页主功能，避免与首页拓扑重复。
- 调度页更多关注“为什么调度、调了什么、效果如何”。

## 5. 当前亮点

### 5.1 调度逻辑与轨道车辆解耦

调度侧不创建车辆，而是读取轨道/车辆侧创建出的车辆状态。这符合系统分工：

- 轨道/车辆侧负责车辆对象和物理运行。
- 信号轨道侧负责进路、道岔、区段、信号机。
- 调度侧负责运行计划、交路计划、调度决策。

### 5.2 车辆路线可自动分配

现在车辆只要出现在端点附近，调度侧就能自动判断其起点并分配交路。

这使得后续 001/002 初始上线、003/004 稍后上线的场景可以自然接入。

### 5.3 交路计划和信号进路申请已打通

调度侧已经能够从车队交路生成进路申请命令，并通过 `REQUEST_ROUTE` 进入信号侧处理。

这说明路线调度不是只停留在前端展示，而是已经接入后端命令链路。

### 5.4 时间调度具备效果评估基础

调度侧已经能展示目标间隔、实际间隔、偏差、动作和预测改善效果，为 PPT 展示“调度有效性”提供数据基础。

## 6. 与信号轨道侧需要对接的内容

### 6.1 车辆状态字段

调度侧需要轨道/车辆侧稳定提供：

| 字段 | 用途 |
|---|---|
| `id` | 识别车辆，避免重复分配 |
| `currentStationId` | 优先判断车辆所在端点或到站状态 |
| `positionMeters` | 端点识别兜底、运行位置判断 |
| `faultLevel` | 判断车辆是否可上线分配 |
| `status` | 判断车辆是否停车、停站、运行中 |
| `speedMetersPerSecond` | 判断是否可发车、是否停车 |
| `direction` | 辅助判断上下行和前端展示 |

最低要求：

```json
{
  "id": "TR-001",
  "currentStationId": "S101",
  "positionMeters": 0,
  "faultLevel": 0
}
```

如果 `currentStationId` 暂时不能稳定提供，则至少要保证 `positionMeters` 接近真实端点米标。

### 6.2 标准 routeId 对齐

信号侧需要保证以下 routeId 可申请：

```text
R_UP
R_TB_LIB
R_DOWN
R_TB_GGZ
```

调度侧会以这些 routeId 生成车队交路和进路申请。

### 6.3 REQUEST_ROUTE 反馈

调度侧希望信号侧返回结构化进路申请结果：

```json
{
  "accepted": false,
  "routeId": "R_UP",
  "trainId": "TR-001",
  "failureCode": "TRACK_OCCUPIED",
  "rejectReason": "TRACK_OCCUPIED: U02 is occupied by another train",
  "retryable": true
}
```

关键字段：

| 字段 | 用途 |
|---|---|
| `accepted` | 判断进路申请是否成功 |
| `failureCode` | 前端展示和后续策略判断 |
| `rejectReason` | 说明拒绝原因 |
| `retryable` | 判断是否可以自动重试 |

### 6.4 进路生命周期事件

为了实现真实闭环，信号侧最好提供进路生命周期：

```text
AVAILABLE
VALIDATING
SETTING_SWITCHES
LOCKED
OCCUPIED
RELEASING
RELEASED
REJECTED
CONFLICTED
CANCELLED
```

调度侧希望事件至少包含：

```json
{
  "trainId": "TR-001",
  "routeId": "R_UP",
  "fromStatus": "LOCKED",
  "toStatus": "OCCUPIED",
  "detail": "...",
  "simulatedAt": "..."
}
```

后续调度侧可以将交路 leg 的完成条件从“进路申请 accepted”升级为“进路 RELEASED 或车辆真实到站”。

### 6.5 移动授权与限速来源

调度侧需要读取信号侧移动授权：

```json
{
  "trainId": "TR-001",
  "authorityEndMeters": 1200,
  "speedLimitMetersPerSecond": 15,
  "reason": "ROUTE_LOCKED",
  "currentSegmentId": "U02",
  "endSegmentId": "U10",
  "reasonCode": "NORMAL"
}
```

如果能进一步拆分限速来源，前端评估会更清楚：

```json
{
  "speedLimitSources": {
    "baseLimit": 22,
    "dispatchLimit": 18,
    "signalLimit": 15,
    "vehicleFaultLimit": null,
    "effectiveLimit": 15
  }
}
```

## 7. 当前仍需完善的点

### 7.1 交路完成还需要真实反馈

当前交路 leg 的推进主要依赖进路申请成功反馈。严格运营闭环中，更合理的是：

```text
REQUEST_ROUTE accepted -> 进路申请成功
route LOCKED/OCCUPIED  -> 进路实际执行
route RELEASED         -> 进路释放
车辆到达目标站          -> 当前 leg 完成
```

因此后续需要把交路完成从“申请成功”改为“进路释放或车辆真实到站”。

### 7.2 调度效果仍依赖反馈质量

时间间隔调度的效果展示已经有基础指标，但如果要更明显，需要车辆和信号侧持续返回：

- 车辆真实速度变化。
- 到站/离站事件。
- 移动授权变化。
- 限速来源。
- 进路生命周期。

这些反馈越完整，调度效果越容易量化。

### 7.3 路线颗粒度待进一步确定

当前路线是较粗粒度的四条标准路线：

```text
R_UP / R_TB_LIB / R_DOWN / R_TB_GGZ
```

如果后续需要更精细的区间级路线调度，可以继续拆细 routeId。但需要与信号侧统一：

- routeId 命名。
- route 覆盖的区段。
- 保护区段。
- 道岔需求。
- 冲突关系。

## 8. PPT 建议页结构

可以按下面结构拆成 PPT：

1. **调度侧模块定位**
   - 调度侧不创建车辆，只做计划、决策、命令和评估。

2. **调度侧总体闭环**
   - 状态输入 -> 调度判断 -> 命令生成 -> 信号/车辆执行 -> 反馈评估。

3. **时间间隔调度功能**
   - 实际间隔、目标间隔、偏差判断、本车调度动作。

4. **线路级间隔优化**
   - 全线统一判断，多辆车分别执行本车命令。

5. **车队交路循环路线调度**
   - `R_UP -> R_TB_LIB -> R_DOWN -> R_TB_GGZ`。
   - S101/S113 自动识别起点。

6. **车辆自动识别与路线自动分配**
   - `currentStationId` 优先，`positionMeters` 兜底。
   - tick 自动分配后续上线车辆。

7. **与信号侧的 REQUEST_ROUTE 对接**
   - 调度生成进路申请，信号返回 accepted/rejected。

8. **前端调度页面展示重点**
   - 运营计划、间隔偏差、调度动作、交路计划、信号反馈、效果评估。

9. **当前亮点**
   - 车辆识别自动化。
   - 时间调度和路线调度均已进入后端闭环。
   - 调度命令与信号进路申请链路已打通。

10. **后续完善方向**
    - 基于 route RELEASED/车辆到站推进交路。
    - 完善移动授权和限速来源反馈。
    - 根据组长和信号侧意见确定路线颗粒度。

## 9. 一句话汇报口径

调度侧目前已经实现了运营计划接入、时间间隔调度、线路级间隔优化、车队交路循环计划，以及根据车辆初始点自动分配路线并向信号侧申请进路的能力；后续重点是接入更完整的信号进路生命周期、移动授权、限速来源和车辆到站反馈，把调度计划从“申请成功推进”升级为“真实运行反馈闭环”。
