# 调度模块补充功能与信号对接缺口

本文面向调度模块后续实现和与信号轨道模块联调。结论先放前面：当前系统已经具备“调度提交进路请求 → 信号联锁检查/扳道岔/锁闭 → 调度接收反馈并更新预约”的基本闭环，但道岔/路线调度仍偏演示逻辑，不是严格的地铁运营路径控制。要做得更像真实地铁，需要把调度从“靠近哪个进路就申请哪个进路”升级为“按列车运行计划、站台、服务路径和进路序列逐段申请”。

## 1. 当前闭环现状

### 1.1 调度到信号

当前调度侧统一通过 `DispatchCommand` 生成指令，仿真主循环消费以下进路类命令并交给信号联锁：

- `REQUEST_ROUTE`：申请建立进路。
- `REROUTE`：重新选路，信号侧按 detail/routeId 匹配可用进路。
- `CANCEL_ROUTE`：取消已预约或已锁闭的进路。

信号侧说明中要求调度使用：

```http
POST /api/dispatch/commands
```

不要使用调试接口：

```http
POST /api/dispatch/route/establish
```

原因是调试接口会绕过调度命令记录、decision、reservation 和操作闭环。

### 1.2 信号到调度

信号侧已经补充结构化结果：

```json
{
  "accepted": false,
  "rejectReason": "TRACK_OCCUPIED:T02 is occupied by another train",
  "failureCode": "TRACK_OCCUPIED",
  "retryable": true
}
```

调度侧需要消费这些字段，更新 `RouteReservation.failureCode`、`failureCategory`、`retryable`，用于页面展示、重试判断和后续换路策略。

本次代码已补齐调度侧结构化反馈解析：

- 运行时把信号返回的 `failureCode` 和 `retryable` 透传进 `DispatchCommandFeedback.details`。
- `InterlockingFeedbackParser` 优先读取结构化 `failureCode`，不再只靠 reason 文本猜测。
- `retryable` 明确字段优先于本地默认分类。
- 预约记录中能保存信号标准失败码，例如 `TRACK_OCCUPIED`、`SWITCH_LOCKED`。

## 2. 当前道岔/进路调度实际行为

现在的自动进路调度不是按站台一站一站发固定路线，而是以下逻辑：

1. 调度在每个评估周期读取列车位置和 MA。
2. `RouteIntentResolver` 计算列车前方 lookahead 范围。
3. 从 `RouteCatalog.candidateRoutesNear(...)` 找附近候选进路。
4. 优先选择与当前列车 `routeId` 相同、正线、距离更近、长度更短的进路。
5. `RouteIntentArbiter` 在多车争用时按优先级选一辆车发 `REQUEST_ROUTE`。
6. 信号收到 routeId 后做联锁检查、道岔转换、锁闭、占用和释放。

也就是说，调度侧现在更像“接近某个进路入口时自动申请一个候选进路”，不是“列车属于某条运营服务，按服务路径依次经过 S01-S02-S03 并申请对应站间进路”。

## 3. 与真实地铁运营方式的差距

真实地铁调度通常不是“哪条路空就走哪条路”。常规载客列车会绑定：

- 交路/车次/服务号。
- 起点站、终点站、方向。
- 停站序列。
- 站台或股道。
- 固定路径或可选折返/绕行路径。
- 每个运行区间对应的进路序列。

备用路径、绕行、折返通常是异常处置或计划内特殊交路，不应该在普通正线运行中随便因为“某条路空”就自动改走。调度可以决定是否改路，但这个决定应该有明确原因，例如故障、冲突长时间不可恢复、人工改路或计划交路要求。

当前差距主要在：

- `dispatch-plan.yaml` 的 service 只有 `direction` 和 `stops`，没有 `pathId`、`routeSequence`、`platformId` 或备选路径。
- `TrainServicePlan` 只表达停站计划，不表达“这趟车应该走哪些进路”。
- 自动进路选择基于位置和候选进路，不基于 service path。
- 配置中的进路粒度较粗，例如 `R_MAIN` 覆盖 T01-T05，不是 S01→S02、S02→S03 这种站间/信号机间小进路。
- 信号侧仍有 `touchRoutes(trains)` 自动建路逻辑，正式闭环模式下可能绕过调度预约语义。

## 4. 调度模块需要补充的功能清单

### P0：结构化联锁反馈消费

状态：已补齐基础能力。

调度需要稳定使用信号侧的标准字段：

- `accepted`
- `failureCode`
- `retryable`
- `rejectReason/rawReason`
- `interlockingState`

调度处理规则：

- `accepted=true`：decision 标记 `ACCEPTED`，reservation 标记 `ACCEPTED`。
- `accepted=false`：reservation 标记 `REJECTED`，保存 `failureCode`、`failureCategory`、`retryable`。
- `retryable=true`：允许在冷却时间后重试同进路，或交给换路策略。
- `retryable=false`：不再盲目重试，提示配置错误、非法命令或人工处理。

### P0：统一命令入口

状态：后端已有统一命令入口，但需要前端和人工操作保持一致。

所有调度到信号的控制都应走：

```http
POST /api/dispatch/commands
```

调度页面不应直接调用信号调试接口建立进路。否则会缺失：

- `DispatchCommand` 操作记录。
- route decision。
- route reservation。
- 失败码展示。
- 重试/取消/超时闭环。

### P0：进路生命周期同步

状态：信号侧已提供事件，调度侧尚未精确消费。

信号侧已有：

```http
GET /api/signal-track/route-events
```

以及 `RouteLifecycleEvent(routeId, trainId, fromStatus, toStatus, detail)`。

调度侧当前主要通过 `syncRouteReservations(interlockingService.states())` 轮询状态，当已接受的 reservation 不再持有联锁资源时标记 `RELEASED`。这能完成基础释放，但还不够精确。

建议补充：

- 消费 `LOCKED/OCCUPIED/RELEASING/RELEASED/CANCELLED/REJECTED/CONFLICTED` 事件。
- 将 reservation 从 `ACCEPTED` 细分或至少记录当前信号状态。
- 用生命周期事件驱动 `releasedAt`、`cancelledAt`、`conflictedAt`。
- 前端显示“已申请、已锁闭、列车占用、释放中、已释放”的全过程。

### P1：服务路径与进路序列

状态：缺失，是道岔/路线调度不像地铁的核心原因。

建议在运行计划中增加路径字段：

```yaml
services:
  - serviceId: SVC-001
    trainId: TR-001
    direction: DOWN
    pathId: PATH-MAIN-DOWN
    routeSequence:
      - routeId: R_S01_S02_DOWN
        fromStation: S01
        toStation: S02
        platformId: P-S01-D
      - routeId: R_S02_S03_DOWN
        fromStation: S02
        toStation: S03
        platformId: P-S02-D
```

调度应按 `routeSequence` 推进游标：

- 当前列车在 S01 发车前，申请 S01→S02 进路。
- 列车进入或释放该进路后，游标推进到 S02→S03。
- 到站停稳、满足停站时间后，再申请下一段出站/区间进路。
- 只有当前计划进路不可用且策略允许时，才进入备选路径。

### P1：按计划进路优先，备选路线显式触发

状态：缺失。

现在逻辑偏向“候选路线排序后选一个”。建议改成：

1. 找到列车绑定的 service。
2. 找到 service 当前 route cursor。
3. 优先申请 plannedRouteId。
4. 如果失败且 `failureCode.retryable=true`，先等待/重试。
5. 如果失败超过阈值或故障不可恢复，再检查 `alternativeRouteIds`。
6. 换路必须生成新的 decision，并记录原因。

这能避免普通载客车因为支线空闲就被调到支线。

### P1：正式闭环模式下禁用信号自动建路兜底

状态：需要和信号侧确认。

`RouteInterlockingService.touchRoutes(trains)` 目前会在列车头部进入某个进路区段时自动建立进路。这对演示很方便，但正式调度闭环里会带来两个问题：

- 进路可能不是由调度 command/reservation 触发。
- 页面看到的 reservation 与实际信号状态可能不一致。

建议增加模式开关：

- 演示模式：允许 touchRoutes 自动建路，保证系统可跑。
- 正式调度模式：只有 `REQUEST_ROUTE/REROUTE` 能建路；未锁闭时信号给零 MA 或 hold。

### P2：进路粒度从粗进路改为站间/信号机间

状态：配置层需要补。

当前 demo 进路：

- `R_MAIN`: T01-T05
- `R_NORTH`: T06-T10
- `R_DEPOT`: T11-T12
- `R_LOOP`: T13-T15
- `R_BRANCH`: T13-T14-T16

这更像整条路径，不像真实联锁里的多个小进路。建议补充更细的 route 定义，例如：

- `R_S01_S02_DOWN`
- `R_S02_S03_DOWN`
- `R_S03_S04_DOWN`
- `R_S04_S05_DOWN`
- `R_S05_S06_DOWN`
- 折返、出入段、支线分别定义独立进路。

调度按站间进路发，信号按进路表控制道岔和 MA。

### P2：路径感知的列车定位

状态：需要车辆/轨道/信号共同确认。

当前列车状态主要使用 `positionMeters` 标量里程。线路有平行支线和重叠里程时，仅靠里程会难以判断列车到底在正线、北侧绕行、折返线还是车辆段。

建议后续引入：

- 当前 segmentId。
- pathId 或 routeId。
- head/tail 所在区段。
- 上下行方向。
- 站台/股道。

调度只消费这些定位结果，不负责车辆物理积分。

## 5. 推荐目标流程

### 正常载客运行

1. 调度加载运行计划，确定每列车 service、stops、pathId、routeSequence。
2. 到计划发车时刻，调度发 `DEPART`。
3. 发车前或列车接近出站信号机时，调度按 route cursor 发 `REQUEST_ROUTE`。
4. 信号检查联锁条件、扳道岔、锁闭进路。
5. 信号返回 `accepted/failureCode/retryable`。
6. 调度更新 decision/reservation。
7. 信号通过 MA 约束车辆运行。
8. 列车占用、出清后，信号释放进路并产生 lifecycle event。
9. 调度推进 route cursor，准备下一站间进路。

### 失败与换路

1. 信号返回 `TRACK_OCCUPIED/ROUTE_CONFLICT/SWITCH_LOCKED` 等 retryable 失败。
2. 调度进入等待或重试，不立即随便换线。
3. 超过阈值后，根据运行策略选择 hold、跳停、绕行或人工确认。
4. 若选择绕行，调度发新的 `REROUTE/REQUEST_ROUTE`，并记录新 decision。

## 6. 与信号侧对接还缺什么

### 6.1 failureCode/retryable 字段格式

本次调度侧已经能解析：

- `details.failureCode`
- `details.retryable`

还需要联调确认每个失败场景都确实返回标准字段，而不是只写在 `rejectReason` 里。

重点场景：

- 进路不存在：`ROUTE_NOT_FOUND`
- 无匹配进路：`NO_MATCHING_ROUTE`
- 区段占用：`TRACK_OCCUPIED`
- 区段故障：`TRACK_FAULT`
- 敌对进路：`ROUTE_CONFLICT`
- 道岔锁闭：`SWITCH_LOCKED`
- 道岔扳动失败：`SWITCH_MOVE_FAILED`
- 进路不可用：`ROUTE_NOT_AVAILABLE`
- 道岔位置矛盾：`SWITCH_POSITION_MISMATCH`
- 不支持命令：`UNSUPPORTED_COMMAND`

### 6.2 route-events 消费方式

信号侧已给 HTTP 轮询接口。调度侧需要确认：

- 事件是否是 drain 语义，即一次读取后清空。
- 多个前端同时访问会不会把调度需要的事件读走。
- 是否应该由后端 Runtime 内部直接消费，而不是前端轮询消费。
- 事件里 `tick` 当前为 0，后续是否能传真实仿真 tick。
- `trainId` 在 `AVAILABLE -> VALIDATING` 阶段是否可能为空，是否需要补 decisionId/reservationId 关联字段。

建议：正式闭环中由后端 runtime 消费事件并同步 reservation，前端只读调度快照。

### 6.3 自动建路 touchRoutes 的边界

需要与信号侧确认：

- 正式调度模式是否关闭自动建路。
- 如果保留自动建路，是否要生成“信号自发建路事件”并让调度补建 reservation。
- 未调度建路时，信号是否对列车保持 MA 截断/零速。

### 6.4 进路粒度

需要与信号侧确认进路表是否能从粗粒度改为站间/信号机间粒度。调度按站台/站间发 route 的前提是信号侧存在相应 routeId。

如果仍只有 `R_MAIN` 这种整条主线进路，调度无法表达“只申请 S02 到 S03”。

### 6.5 routeId 与 from/to 的匹配规则

当前信号支持：

- 直接 `routeId`
- `fromStation/toStation`
- `fromSignal/toSignal`

建议调度正式逻辑优先发明确 `routeId`，`from/to` 只作为人工或兼容入口。否则同一对站点存在多路径时，信号侧“优先 MAIN、最短路径”的选择可能与调度运营计划不一致。

### 6.6 取消进路语义

需要确认：

- `CANCEL_ROUTE` 成功时返回什么 `resultCode`，当前调度侧使用 `ROUTE_CANCELLED`。
- 已 OCCUPIED 的进路不能取消时，是否返回固定失败码，例如 `ROUTE_NOT_AVAILABLE` 或新增 `ROUTE_CANCEL_NOT_ALLOWED`。
- 取消成功后信号是否一定产生 `CANCELLED` 生命周期事件。

## 7. 与车辆侧需要确认的边界

调度不应该直接控制车辆牵引/制动。推荐边界：

- 调度负责运营意图：发车、扣车、限速、缩短/延长停站、申请进路、取消进路、换路决策。
- 信号负责安全约束：联锁、道岔、进路锁闭、MA、信号机开放、站停许可。
- 车辆负责动力学执行：按 MA、限速、ATO/TCMS 状态运行。

需要车辆侧补齐或确认：

- 列车状态中是否稳定提供当前 segmentId/pathId。
- 站台停稳和门状态是否可信，调度才能发下一段出站进路或 DEPART。
- 调度停站调整 `deltaDwellSec` 作用范围为“本次站停一次”，是否满足车辆/ATO 逻辑。

## 8. 建议实施里程碑

### M1：闭环反馈可靠

- 调度保存 `failureCode/retryable`。
- 前端展示失败码和是否可重试。
- 联调所有标准失败场景。

### M2：生命周期状态可靠

- 后端 runtime 消费 `route-events`。
- reservation 能展示锁闭、占用、释放。
- 取消和自动释放都能闭环。

### M3：固定服务路径

- `dispatch-plan.yaml` 增加 `pathId/routeSequence`。
- 调度维护每车 route cursor。
- 自动进路申请改为 planned route first。

### M4：真实站间进路

- 信号配置拆成站间/信号机间进路。
- 调度按站台/站间发 route。
- 支线/折返/出入段作为特殊 path 或 alternative path。

### M5：异常处置策略

- retryable 失败自动等待/重试。
- 非 retryable 失败提示人工或配置错误。
- 支持明确的 reroute 策略，而不是空闲路径自动优先。
