# 服务器部署与运维手册：车辆仿真及相关配置

## 1. 范围

本手册只部署本轮修改的三项仿真服务：

- `9000`：Python + FMPy真实车辆FMU执行服务；
- `9200`：FastAPI供电仿真服务；
- `9300`：Spring Boot车辆运行、PLC 控制输入、控制状态机和批量仿真调度服务。

8080中央系统与MySQL不在本部署包中。中央系统继续使用既有MySQL部署，并把中央配置指向本服务器的9300和9200。本Compose不会创建数据库容器或数据卷，但 P1 上线前必须在既有MySQL执行当前 `database/schema.sql`，以创建 `simulation_run`、`vehicle_control_command_log`等运行证据表并增加 `simulation_run_id`字段。

## 2. 服务器要求

| 项目 | 要求 |
|---|---|
| 操作系统 | x86_64 Linux，建议Ubuntu 22.04/24.04 |
| Docker | Docker Engine 24或更高 |
| Compose | `docker compose` v2，须支持`service_healthy` |
| 资源 | 建议4 vCPU、8 GB内存、20 GB可用磁盘 |
| 出站网络 | 首次构建需访问Docker Hub和Maven/PyPI镜像源 |
| 入站网络 | 向中央服务器IP开放9300、9200；如 PLC/半实物平台直接调用9300 REST入口，另向该设备IP开放9300；9000无需对中央开放 |
| 时间 | 主机配置NTP；容器默认`Asia/Shanghai` |

确认环境：

```bash
uname -m
docker version
docker compose version
git rev-parse --abbrev-ref HEAD
```

应在`elec-and-train`分支或指定的稳定提交上部署，`uname -m`应为`x86_64`。

## 3. 配置

```bash
git clone <项目仓库地址> railway-sim
cd railway-sim
git checkout elec-and-train
cp deploy/server.env.example deploy/server.env
```

必须修改`deploy/server.env`：

```dotenv
CENTRAL_BASE_URL=http://<中央服务器IP>:8080
INTERNAL_BIND_ADDRESS=0.0.0.0
VEHICLE_PHYSICS_MODE=FMU_HTTP
COMPOSE_PROFILES=fmu
```

配置含义：

| 变量 | 默认值 | 说明 |
|---|---|---|
| `CENTRAL_BASE_URL` | 必填 | 9300主动注册车辆时回调既有中央系统 |
| `INTERNAL_BIND_ADDRESS` | `127.0.0.1` | 同机联调保持默认；跨机中央访问时设`0.0.0.0`并配防火墙 |
| `FMU_PORT` | `9000` | FMU内部运维端口，不建议公网开放 |
| `POWER_NETWORK_PORT` | `9200` | 中央供电接口及9300内部调用端口 |
| `VEHICLE_RUNTIME_PORT` | `9300` | 中央车辆运行时接口 |
| `FMU_TIMEOUT_MILLIS` | `80` | 9300等待一次全车FMU批量调用的上限 |
| `POWER_NETWORK_TIMEOUT_MILLIS` | `300` | 9300等待9200的上限 |
| `VEHICLE_PHYSICS_MODE` | `FMU_HTTP` | 正常使用真实FMU；故障回滚使用`JAVA_FALLBACK` |

9000与9300会只读挂载同一个Schema v2 `config/train_params.yaml`。该文件包含118 m六车编组、AW0/AW2/AW3质量、6.5传动比以及52点牵引/制动电机曲线。修改车辆参数后必须同时重启二者，并核验`parameterSetId`和`curveSetId`一致；禁止只替换其中一个容器的参数文件。`config/fmu_mapping.yaml`只挂载到9000，必须与`TrainTractionBrake/2.0.0`匹配。9200只读挂载`config/power_third_rail.yaml`，启动即加载1500 V、5供电分区的真实项目配置；中央后续仍可通过原`/power-network/bootstrap`接口同步权威拓扑。

## 4. 既有中央系统需要设置的连接项

这些是中央系统持有者在其原部署环境中设置的连接配置，不属于本Compose：

```yaml
railway:
  simulation:
    vehicle-runtime:
      mode: EXTERNAL_HTTP
      base-url: http://<仿真服务器IP>:9300
      timeout-millis: 150
      auto-bootstrap: true
    external-power-network:
      mode: EXTERNAL_HTTP
      base-url: http://<仿真服务器IP>:9200
      timeout-millis: 300
      auto-bootstrap: true
vehicle:
  train-params-path: /绝对路径/config/train_params.yaml
```

中央只从该共享参数文件读取车型、参数哈希和118 m车辆长度，用于注册一致性与轨道占用，不执行车辆动力学。中央与9300的参数文件不一致时，运行时注册返回HTTP 409。中央仍使用原MySQL URL和账号，不新建第二个数据库；已有库在部署本轮 P1 前必须执行 `database/2026-07-12-p1-stopping-alarm-recovery.sql`，新建库直接执行最新 `database/schema.sql`。

## 5. 首次部署和检查

```bash
./scripts/deploy-server.sh deploy/server.env
docker compose --env-file deploy/server.env ps
```

脚本会依次执行配置校验、镜像构建、健康依赖启动和三服务一致性检查。启动顺序为9200/9000健康后启动9300。

手工检查：

```bash
curl -fsS http://127.0.0.1:9000/health
curl -fsS http://127.0.0.1:9000/fmu/metadata
curl -fsS http://127.0.0.1:9200/health
curl -fsS http://127.0.0.1:9300/vehicle-runtime/health
python3 scripts/check-deployment.py
```

PLC 联调时先启动/注册列车实例。现场 PLC 的 46 字节输入先由 8080 接收并封装为 RSIM Version 1 聚合帧；9300 的服务间入口为：

```http
POST http://127.0.0.1:9300/vehicle-runtime/peripherals/frame
GET  http://127.0.0.1:9300/vehicle-runtime/peripherals/status
```

该 POST 只接受完整 RSIM 聚合帧，不接受裸 46 字节 PLC 文件，并且只允许 `PLC_INPUT`。前端结构化联调使用 8080 的 `POST /api/vehicle/driver-cabs/{trainId}/plc-input`；信号侧状态和 PLC 显示输出仍从 8080 GET 接口读取。网络屏、信号屏和视景系统均为单向输出显示端，不得向 9300 发送业务输入。

验收时确认：9300为`FMU_HTTP`；9000为`ready=true`；模型版本为`TrainTractionBrake/2.0.0`；9300和9000的`parameterSetId`、`curveSetId`一致；`parameterSchemaVersion=2`、`curvePointCount=52`、`lengthMeters=118.0`；9200显示`nominalVoltage=1500`、`powerSectionCount=5`并返回供电配置哈希。

P1 另需在 8080 检查：快照顶层 `simulationRunId` 非空；`GET /api/simulation-runs/{runId}/summary` 可查；reset 后旧 run 为 `CANCELLED_BY_RESET` 且新 run 为 `CREATED`；9300→9200 `/step` 的 runId 与中央一致。

可单独检查车辆曲线元数据：

```bash
curl -fsS http://127.0.0.1:9300/vehicle-runtime/parameters
curl -fsS http://127.0.0.1:9000/fmu/metadata
```

两端必须同时显示相同的参数哈希和曲线哈希。9000元数据还应显示52点曲线、16台电机、24轴、0.46 m轮径和6.5传动比。

## 6. 日常运维

查看状态和日志：

```bash
docker compose --env-file deploy/server.env ps
docker compose --env-file deploy/server.env logs --tail=200 vehicle-runtime
docker compose --env-file deploy/server.env logs --tail=200 fmu
docker compose --env-file deploy/server.env logs --tail=200 power-network
```

9300健康接口重点字段：

- `fmuBatchLatencyMillis`：最近一次全车FMU批量时延；
- `missedDeadlineCount`：超过100 ms的完整车辆/供电周期累计数；
- `fallbackTrainCount`：当前处于粘滞Java fallback的车辆数；
- `fallbackEventCount`：累计降级车辆事件数；
- `fmiErrorCount`：单车或整批FMU错误累计数；
- `parameterSetId`、`fmuModelVersion`：参数和模型身份。

修改参数或升级镜像：

```bash
git fetch origin
git checkout <已验收提交>
./scripts/stop-server.sh deploy/server.env down
./scripts/deploy-server.sh deploy/server.env
```

## 7. 故障恢复

### 7.1 单列车或9000故障

9300会让受影响列车进入粘滞Java fallback，9000恢复后不会自动热切回。确认9000健康后执行：

```bash
curl -fsS -X POST http://127.0.0.1:9300/vehicle-runtime/physics/instances/resync-all
```

下一100 ms步骤以中央下发的权威速度、位置和累计能量执行`RESYNC`。检查`fallbackTrainCount`回到0、`dataQuality`恢复`GOOD`。

这里的Java fallback由9300执行，仍保持车辆状态机和物理权威在车辆运行服务。若9300整体不可达，8080会将该tick明确判为车辆运行时不可用，不会在中央启用另一套简化动力学；应先恢复9300，再继续仿真。

### 7.2 9200 重启恢复

9200 重启后不会沿用内存运行状态，但会立即从 `config/power_third_rail.yaml` 恢复 1500 V、5 分区基线，因此不应出现 750 V、2 分区过渡态，权威约束查询也不再依赖 8080 先行 bootstrap。保持 9300 和 8080 运行并再次触发中央 tick；8080 仍可重新下发权威拓扑，再由 9300 提交该 tick 的唯一负荷写入。随后检查：

```bash
curl -fsS http://127.0.0.1:8080/api/service-health
curl -fsS http://127.0.0.1:9200/power-network/state
```

9300 与 9200 均须恢复为 `UP`，`RecoveryGate.accepted=true`，runId 与中央一致，`lastAcceptedTick` 等于当前 tick，且所有 hash/version 检查为 true。若任一项不匹配，保持 `RECOVERING`，不得人工改为 UP。

### 7.3 紧急切换Java模型

```bash
cp deploy/server.env deploy/server.env.fmu.bak
# 将COMPOSE_PROFILES改为空，将VEHICLE_PHYSICS_MODE改为JAVA_FALLBACK
./scripts/stop-server.sh deploy/server.env down
./scripts/deploy-server.sh deploy/server.env
```

可参考`deploy/java-fallback.env.example`。此操作不触碰中央或MySQL。

### 7.4 停止服务

```bash
./scripts/stop-server.sh deploy/server.env
# 停止并移除三服务网络：
./scripts/stop-server.sh deploy/server.env down
```

脚本固定先停9300，再停9000和9200。

## 8. 防火墙建议

以UFW为例，`<中央服务器IP>`替换为真实地址：

```bash
sudo ufw allow from <中央服务器IP> to any port 9300 proto tcp
sudo ufw allow from <中央服务器IP> to any port 9200 proto tcp
sudo ufw deny 9000/tcp
```

如果9000仅供Compose内的9300调用，可删除其宿主机端口映射；当前保留`127.0.0.1:9000`是为了健康检查和现场故障演练。

## 9. 正式验收与回滚

```bash
./scripts/test-fmu-acceptance.sh deploy/server.env
```

正式验收会运行20车6000个0.1 s仿真tick，并保存性能和恢复报告。任何阻断项失败时：

1. 保留`docs/真实FMU集成实施计划/验收记录/*.json`和容器日志；
2. 切换`JAVA_FALLBACK`；
3. 回到上一个已验收Git提交和镜像标签；
4. 不迁移、不删除既有MySQL数据。
