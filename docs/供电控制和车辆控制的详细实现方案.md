# 供电控制和车辆控制的详细实现方案

## 1. 依据和范围

本方案只规划方首岳负责的供电与车辆模块实现，不规划调度策略、信号联锁、轨道拓扑算法、综合监控前端页面和客流预测的内部实现。其他模块只在需要消费或提供供电/车辆数据时作为接口边界出现。

## 2. 实现边界

### 2.1 主责内容

| 方向 | 方首岳负责实现 | 不负责实现 |
|---|---|---|
| 供电内部模型 | 供电分区状态、电压电流、带电/失电/欠压、牵引负荷、再生制动吸收、故障状态、检修闭锁状态和供电侧告警语义。 | 调度系统如何制定限速、扣车、折返、清客策略。 |
| 车辆内部模型 | TCMS/ATO 适配、车辆状态缓存、牵引/制动/受流/车门/载客量/故障状态、车辆物理输入输出、FMU fallback 与降级。 | 信号 ATP/ATO 算法本身、进路联锁、轨道占用算法。 |
| 供电-车辆耦合 | 根据列车位置、取流电流、再生功率计算分区负荷和受流点电压；将 `railVoltage`、`powerAvailable` 提供给车辆物理模型。 | 供电故障后的全局运营调整策略。 |
| 数据输出 | 输出供电、车辆、能耗、故障、告警、影响范围和检修预留字段，供综合监控、调度、维修/能耗扩展消费。 | 综合监控页面布局、颜色图例、前端交互实现。 |
| 接口文档和联调 | 定义字段字典、枚举、刷新周期、异常处理、数据质量、验收场景。 | 其他成员模块内部字段的最终解释。 |

### 2.2 本期默认实现深度

1. 供电遥控、检修审批、维修工单闭环和经济成本分析先做状态字段、审计字段和接口预留，不承诺完整业务流。
2. Modelica/FMU 只做车辆牵引、制动、阻力、纵向动力学、牵引功率、取流电流和再生制动功率，不把接触轨供电网、信号、轨道和调度放入 Modelica。
3. 供电模型采用接触轨分区等效模型，不做完整潮流计算。
4. 线路 `.xls/.xlsx` 工作簿只经 `infrastructure` 适配层进入标准对象，供电/车辆模块不直接解析 sheet。

## 5. 总体技术方案

每个仿真步长内保持当前数据流：

```text
SimulationRuntime.tick()
-> TrackService.updateOccupancy()
-> SignalService.calculateAuthorities()
-> PowerService.constraintsForTrains()
-> DispatchService.constraintsForTrains()
-> TcmsAtoAdapterService.buildVehiclePhysicsInput()
-> VehiclePhysicsClient.stepFleet()
-> TrainManager.apply vehicle physics output
-> PowerService.updateFromVehicleOutputs()
-> MonitorService.buildSnapshot()
-> WebSocket push snapshot
```

关键边界：

1. `PowerService` 只输出供电约束和供电状态，不直接操控列车。
2. `TcmsAtoAdapterService` 是唯一把调度、信号、轨道、供电约束转成车辆物理输入的模块。
3. `VehiclePhysicsClient` 是车辆物理端口；当前 Java fallback 保持可运行，真实 FMU 接入不改变上层契约。
4. `TrainManager` 负责多车实体批量推进，不为每辆车创建服务或线程。
5. `MonitorService` 汇聚事件和快照生成告警；综合监控前端只消费接口，不参与车辆/供电计算。

#### 6.1.3 供电计算逻辑

`PowerService.updateFromVehicleOutputs()` 分三步改造：

1. 按供电分区聚合牵引电流、牵引功率、再生功率和列车 ID。
2. 计算接触轨电压和可用功率：

```text
I_section = sum(tractionCurrent) - regenAbsorbedCurrent
U_train = U_substation - I_section * R_path
P_available = max(0, U_train * I_max * supplyModeFactor)
```

3. 根据阈值和故障状态决策：

```text
if maintenanceLocked or breakerOpen:
    status = DEENERGIZED
elif protectionTrip:
    status = TRIPPED
elif voltage < cutoffVoltage:
    status = DEENERGIZED
elif voltage < minimumVoltage:
    status = UNDERVOLTAGE
elif current > overCurrentThreshold:
    status = OVERCURRENT
else:
    status = ENERGIZED
```

再生制动吸收规则：

| 场景 | 处理 |
|---|---|
| 同分区有牵引负荷 | 再生功率优先抵消牵引功率，减少分区净电流。 |
| 同分区无牵引负荷 | 计入 `unabsorbedRegenPowerWatts`，本期默认 `BRAKE_RESISTOR` 消耗。 |
| 分区失电/隔离 | 再生吸收不可用，车辆侧后续应降低再生制动可用比例。 |

| 供电 | `railVoltage`、`powerAvailableWatts`、`energized` | `currentCollectionStatus`、`powerDeratingFactor`、`regenAvailable`。 |
| 调度 | `holdTrain`、速度限制、限速比例 | `dispatchReason`、`commandTraceId`。 |
| 车辆自身 | `loadRate`、能耗累计、门关闭 | 自检状态、牵引/制动可用、故障等级、车门锁闭。 |

牵引命令生成规则：

```text
if door not closed or selfCheck failed:
    tractionCommand = 0
elif brake unavailable or traction unavailable:
    tractionCommand = 0
elif power not energized or powerAvailable <= 0:
    tractionCommand = 0
elif dispatch hold or ATP emergency brake:
    tractionCommand = 0
else:
    tractionCommand = speedMargin based command
```

制动命令生成规则：

```text
if emergencyBrake:
    brakeCommand = 1
elif dispatch hold:
    brakeCommand = service brake until zero speed
elif overspeed:
    brakeCommand = proportional brake
elif maDistance below stopping distance:
    brakeCommand = proportional brake
else:
    brakeCommand = 0
```

#### 6.2.3 车辆物理模型和 FMU

保持 `VehiclePhysicsInput` / `VehiclePhysicsOutput` 为稳定内部契约。

第一阶段先增强 Java fallback：

- 读取 `config/train_params.yaml`，替换当前硬编码质量、牵引、制动、阻力参数。
- 明确再生制动比例、空气制动补足、低压牵引折减。
- 增加 `faultCode` 输出规则：欠压、牵引不可用、制动不可用、FMU 输出异常。

第二步接真实 FMU：

- `fmu-service/modelica/TrainTractionBrake.mo` 导出 FMI 2.0 Co-Simulation FMU。
- `fmu-service/app/train_fmu_instance.py` 用 FMPy/PyFMI 替换 fallback。
- `fmu-service/app/input_mapper.py` 和 `output_mapper.py` 校验变量映射。
- `FmuVehiclePhysicsAdapter` 保持 HTTP `/step-fleet` 批量调用；失败后继续 fallback 并输出 `dataQuality=FALLBACK`。

不做的内容：

- 不在 Modelica 中实现信号 MA、轨道占用、调度策略、接触轨全网潮流。
- 不让信号、轨道、供电、调度模块直接调用 FMU。

#### 6.2.3.1 外部车辆仿真协议适配层

协议对接不改变主循环边界，仍以 `VehiclePhysicsClient.stepFleet()` 作为唯一车辆物理端口，`TcmsAtoAdapterService` 作为唯一车辆控制输入汇聚点。本项目新增 `com.railwaysim.vehicle.external` 适配层：

| 模式 | 配置值 | 实现 | 用途 |
|---|---|---|---|
| 本地默认 | `LOCAL` | `LocalFallbackVehicleAdapter`，兼容旧 `fmu-service-enabled` HTTP FMU 开关。 | 前期自研仿真主路径。 |
| UDP 外部车辆模型 | `EXTERNAL_UDP` | `ExternalUdpVehicleAdapter` + `ExternalUdpPacketCodec`。 | 对接 20ms 高频车辆行驶控制。 |
| RT-LAB API | `EXTERNAL_RTLAB_API` | `RtLabApiVehicleAdapter` + `StubRtLabClient`。 | 对接 500ms 参数写入/监视变量读取；当前先以 stub 保持可测。 |
| 影子比对 | `DUAL_SHADOW` | `ShadowCompareAdapter`。 | 本地模型作为权威输出，外部模型只做误差比对和健康状态验证。 |

配置入口：

```yaml
railway:
  simulation:
    external-simulator:
      mode: LOCAL
      max-trains: 20
      segment-mapping: {}
      udp:
        model-host: 192.168.200.110
        model-port: 23001
        platform-host: 192.168.200.102
        platform-port: 23002
      rt-lab:
        project-path: ""
        stub-enabled: true
```

UDP 协议映射：

| 方向 | 内容 | 实现 |
|---|---|---|
| 平台到模型 | 1-20 车 `command + percent`，小端 double。 | `ExternalUdpPacketCodec.encodeCommands()`。 |
| 模型到平台 | 1-20 车加速度、速度、累计里程，小端 double。 | `ExternalUdpPacketCodec.decodeTelemetry()`。 |
| 牵引 | `tractionCommand > 0` | `command=1`，`percent=round(tractionCommand*100)`。 |
| 制动 | `brakeCommand > 0` | `command=2`，`percent=round(brakeCommand*100)`。 |
| 紧急制动 | `emergencyBrakeCommand=true` | `command=2`，`percent=100`。 |
| 惰行 | 无牵引、无制动 | `command=0`，`percent=0`。 |

RT-LAB API 变量映射：

- 生命周期由 `RtLabClient` 封装：`OpalOpenProject`、`OpalLoad`、`OpalExecute`、`OpalReset`、`OpalCloseProject`、`OpalSetParametersByName`。
- 输入变量按每车 6 个参数展开：`IDn`、`num_segn`、`x0_segn`、`sig_trainn`、`handlen`、`active_Tcn`。
- 路径根命名空间使用 `PowerSystemAndTrainsV1/.../Train_Control/...`；表格中的省略行号不直接照抄，按 1-20 车生成。
- 输出变量预留每车 6 个：列车编号、激活端、方向、加速度、速度、累计里程。

供电边界：

- `PowerService` 仍是本系统供电状态源，继续输出电压、可用功率、降额、失电、跳闸和检修闭锁。
- 外部车辆模型只替代车辆动力学计算，不接管供电分区状态，也不假设协议已有 PSCADA 全点表。
- 供电约束先经 `TcmsAtoAdapterService` 转为牵引/制动/惰行指令，再由协议适配层发送给外部模型。
- 若后续拿到独立供电仿真点表，再新增 `ExternalPowerTelemetryAdapter`，不能把 `PowerSystemAndTrainsV1` 命名空间误判为供电遥测协议。

车轨耦合边界：

- 协议需要 `seg号 + 偏移`，当前由 `ExternalSegmentMapper` 从 `OperationalLineData` 区段和 `segment-mapping` 配置生成。
- 未配置外部段号时使用线路数据 `rawSegmentId`，再退化为区段序号；区段边界和越界位置会被夹到有效 offset。
- 外部 `segNo` 权威定义需要与信号/轨道模块共同确认，车辆模块只消费映射，不硬编码外部段号。

| 正常牵引运行 | 启动仿真，车辆在带电分区运行。 | 车辆速度、位置、牵引功率、取流电流更新；供电分区电流和电压随车辆负荷变化；无告警。 |
| 欠压牵引限制 | 对分区注入 `UNDERVOLTAGE`。 | `PowerSectionState.status=UNDERVOLTAGE`；车辆 `tractionCommand` 或可用功率下降；告警等级 2；受影响列车列表正确。 |
| 分区失电 | 对车辆所在分区注入 `DEENERGIZED` 或 `BREAKER_TRIP`。 | 车辆牵引切除，受流状态异常；供电告警等级 3；综合监控快照包含影响分区和列车。 |
| 再生制动吸收 | 让一列车制动、同分区另一列车牵引。 | 分区 `regenPowerWatts` 增加，牵引负荷被抵消，记录吸收功率。 |
| FMU 服务失败 | 开启外部 FMU 后停止 `fmu-service`。 | 后端发布 `FmuStepFailedEvent` 和 `FmuFallbackActivatedEvent`；车辆继续 fallback 运行；`dataQuality=FALLBACK`。 |
| 外部 UDP 超时 | 切换 `EXTERNAL_UDP`，模型侧无响应。 | 后端降级到本地车辆模型，车辆输出 `EXTERNAL_SIM_FALLBACK`，`dataQuality=FALLBACK`。 |
| RT-LAB stub 影子比对 | 切换 `DUAL_SHADOW`。 | 主输出仍使用本地模型，外部 stub 同步写入 1-20 车变量路径并做误差比对。 |
| 车辆门未锁闭 | 注入 `DOOR_NOT_LOCKED`。 | 车辆禁止牵引，故障等级 3，车辆-信号接口输出门未锁闭。 |
| 制动不可用 | 注入 `BRAKE_UNAVAILABLE`。 | 车辆自检失败，禁止发车或降级，统一告警等级 3。 |
| 检修闭锁 | 设置供电分区 `MAINTENANCE_LOCKED`。 | 分区禁止恢复送电，车辆牵引不可用，操作日志记录闭锁状态。 |

## 12. 完成标准

内部实现阶段完成标准：

1. 车辆在正常供电、欠压、失电、扣车、ATP 制动场景下能持续 tick。
2. 供电分区电压、电流、负荷、可用功率和再生吸收随车辆输出变化。
3. 车辆状态包含 TCMS 必要字段：牵引、制动、受流、车门、载客率、故障等级、能耗。
4. FMU 服务失败不拖垮 Spring Boot，fallback 可继续运行并产生告警。
5. 单元测试覆盖供电计算、TCMS/ATO 适配和车辆物理 fallback。

接口实现阶段完成标准：

1. `docs/API_CONTRACT.md` 与 Java record、前端 TS 类型一致。
2. WebSocket 快照包含方首岳主责的供电、车辆、能耗、告警字段。
3. REST 能查询车辆、供电、能耗和故障状态。
4. 故障注入接口只在仿真模式开放，并记录操作日志。
5. 与调度、信号/轨道、综合监控的联调字段满足 `docs/接口设计分工.md` 中方首岳主责接口族。

## 13. 风险和边界说明

1. 当前运行环路仍是 1D 位置模型；线路工作簿中的复杂拓扑经 `infrastructure` 线性投影后消费，供电/车辆模块不直接处理复杂图拓扑。
2. 当前供电数据只有接触轨分区和电压阈值，PSCADA 全点表、UPS、杂散电流、电能质量先作为字段预留。
3. 当前 Modelica 文件是草案，真实 FMU 导出前 Java/Python fallback 是主运行路径。
4. 维修工单、检修审批、经济分析不在本期方首岳实现范围，只预留字段和只读状态。
5. 供电故障后的限速、扣车、折返、清客由刘倩调度模块决策；方首岳只输出可用供电能力、受影响范围和建议动作。
