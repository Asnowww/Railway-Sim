# 信号轨道模块周报（7月5日 — 7月10日）

## 本周工作概述

本周主要完成了信号轨道模块的 P0 基础功能建设与联调收口：进路联锁十二态状态机、调度-信号进路联调闭环、固定闭塞 MA 计算与站停控制、以及 WP-02~05 的接口与前端面板。同时配合车辆组完成了 9300 离站保护状态（DEPARTING_STATION）和 PLC 输入迁移方案。

---

## 一、进路联锁状态机（Route Interlocking）

### 1.1 十二态迁移实现

完成了从 AVAILABLE 到 RELEASED/CONFLICTED/REJECTED 等十二种状态的合法迁移路径，每种状态迁移前均有前置条件校验：

- **AVAILABLE → VALIDATING**：区段存在性检查、道岔可用性预检
- **VALIDATING → LOCKED**：区段占用/故障逐一检查、敌对进路冲突检测、道岔扳动与锁闭
- **LOCKED → OCCUPIED**：列车进入进路任一区段后自动触发
- **OCCUPIED → RELEASING**：列车离开后自动触发
- **RELEASING → AVAILABLE**：所有道岔解锁、资源释放完成

### 1.2 进路冲突检测

新增 `intersects()` 方法对两条进路的区段集取交集，保证不会在同一区段上建立两条冲突进路。

### 1.3 结构化失败码

定义了 `InterlockingFailureCode` 枚举共 13 种标准失败码（TRACK_OCCUPIED / ROUTE_CONFLICT / SWITCH_LOCKED 等），每种标记 `isRetryable()`。调度模块可据此判断失败后是否要重试或换路。

---

## 二、固定闭塞 MA 计算与站停控制

### 2.1 固定闭塞模式

补充了 `FixedBlockAuthorityCalculator` 的站停速度控制逻辑——站台停靠时 MA 截到站台位置，停站时间到后自动释放。修复了站停释放后速度仍为 0 的问题。

### 2.2 站台停靠时间控制

`resolveStationDwell()` 中实现：
- 列车进入站台窗口且零速后开始累计 dwell ticks
- 达到目标停站秒数后自动释放 MA
- 调度模块可通过 `DispatchConstraint` 的 `deltaDwellSec` 动态调整停站时间

---

## 三、调度-信号进路联调闭环

### 3.1 进路申请接入调度闭环

将 `REQUEST_ROUTE` / `REROUTE` / `CANCEL_ROUTE` 三种命令接入 `handleDepartures` 和 `advanceOneTick`，调度侧生成进路命令后由信号侧 `RouteInterlockingService.applyDispatchCommand()` 消费，返回结构化结果。

### 3.2 按实际列车占用校验

进路建立时不再简单查区段状态，而是调用 `trackService.occupyingTrainIds(segId)` 确认占用列车 ID，仅拒绝其他列车的占用，不拒绝申请列车自己的占用。

## 四、WP-02~05 接口与前端面板

### 4.1 新增端点

| 端点 | 方法 | 用途 |
|------|------|------|
| `/api/signal-track/routes` | GET | 当前所有进路状态 |
| `/api/signal-track/routes/{routeId}/status` | GET | 单条进路详细状态 |
| `/api/signal-track/route-events` | GET | 进路生命周期事件流 |
| `/api/signal-track/faults` | GET/POST | 故障注入与查询 |
| `/api/signal-track/faults/{segmentId}/clear` | POST | 故障清除 |

### 4.2 严格校验

所有写接口（故障注入/清除）增加了 `operator + traceId` 审计字段和区段存在性前置校验。
