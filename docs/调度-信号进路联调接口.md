# 调度-信号进路联调接口

## 目标

本文档定义调度模块与信号模块之间的第一版进路联调协议。当前阶段先跑通最小闭环：

```text
调度生成运营计划
-> 调度向信号发送列车编号和运行路线
-> 信号建立进路、锁闭道岔、生成 MA
-> 信号返回成功或失败原因
-> 调度更新命令、进路预约和 OperationPlan 状态
```

第一版不要求信号理解完整运营计划。`OperationPlan` 是调度侧对象，信号侧只处理安全相关的进路请求。

## 职责边界

调度侧负责：

- 生成和维护 `OperationPlan`
- 选择 `trainId`
- 决定 `routeId`、方向、起点、终点和经由点
- 到计划时间后发起进路请求
- 根据反馈更新计划状态
- 对失败进路执行重试、改路或转人工处理

信号侧负责：

- 根据 `routeId` 建立或取消进路
- 检查轨道占用、故障、敌对进路和道岔条件
- 锁闭/释放道岔
- 生成或约束 MA
- 返回结构化成功或失败反馈

车辆侧负责：

- 按信号 MA 和调度约束运行
- 反馈位置、速度、停站等运行状态

原则：

```text
调度发送运行意图，信号执行安全授权。
信号安全约束优先级高于调度运营优化。
```

## 调度到信号：DispatchRouteRequest

运行时会将调度命令 `REQUEST_ROUTE`、`REROUTE`、`CANCEL_ROUTE` 转换为信号侧请求对象：

```java
RouteInterlockingService.DispatchRouteRequest
```

字段定义：

| 字段 | 必填 | 说明 |
|---|---:|---|
| `commandId` | 是 | 调度命令 ID，用于反馈关联 |
| `commandType` | 是 | `REQUEST_ROUTE`、`REROUTE`、`CANCEL_ROUTE` |
| `trainId` | 是 | 列车 ID |
| `routeId` | 是 | 信号侧固定进路 ID，例如 `R_MAIN`、`R_NORTH` |
| `operationPlanId` | 否 | 调度侧运营计划 ID，例如 `OP-xxxx` |
| `direction` | 否 | 计划方向，建议使用 `UP` / `DOWN` |
| `originPointId` | 否 | 起点，站点或拓扑点 ID |
| `destinationPointId` | 否 | 终点，站点或拓扑点 ID |
| `viaPointIds` | 否 | 经由点列表 |
| `segmentIds` | 否 | 调度计划中的区段列表 |
| `detail` | 否 | 兼容旧接口的进路描述，优先级低于 `routeId` |

第一版约定：

- 信号侧以 `routeId` 作为建路依据。
- `operationPlanId`、起终点、经由点和区段列表主要用于追踪、日志和后续扩展。
- 如果同一 `routeId` 未来需要支持局部进路，再扩展信号侧按 `segmentIds` 建路。

示例：

```json
{
  "commandId": "DC-op-route-OP-001-123456",
  "commandType": "REQUEST_ROUTE",
  "trainId": "T001",
  "operationPlanId": "OP-001",
  "routeId": "R_NORTH",
  "direction": "UP",
  "originPointId": "S02",
  "destinationPointId": "S05",
  "viaPointIds": ["TECH_N", "RENMIN_N", "FIN_N", "EXPO_N"],
  "segmentIds": ["T06", "T07", "T08", "T09", "T10"]
}
```

## 信号到调度：DispatchCommandFeedback

信号处理后通过 `DispatchCommandFeedback.details` 返回结构化信息。

成功示例：

```json
{
  "commandId": "DC-op-route-OP-001-123456",
  "trainId": "T001",
  "commandType": "REQUEST_ROUTE",
  "feedbackSource": "SIGNAL_INTERLOCKING",
  "feedbackStatus": "EFFECT_CONFIRMED",
  "reason": "route established",
  "details": {
    "accepted": true,
    "operationPlanId": "OP-001",
    "routeId": "R_NORTH",
    "resultCode": "ROUTE_ESTABLISHED",
    "failureCode": "ROUTE_ESTABLISHED",
    "retryable": false,
    "interlockingState": "LOCKED",
    "direction": "UP",
    "originPointId": "S02",
    "destinationPointId": "S05",
    "viaPointIds": ["TECH_N", "RENMIN_N", "FIN_N", "EXPO_N"],
    "segmentIds": ["T06", "T07", "T08", "T09", "T10"]
  }
}
```

失败示例：

```json
{
  "commandId": "DC-op-route-OP-001-123456",
  "trainId": "T001",
  "commandType": "REQUEST_ROUTE",
  "feedbackSource": "SIGNAL_INTERLOCKING",
  "feedbackStatus": "SKIPPED",
  "reason": "Route R_NORTH conflicts with established route R_MAIN",
  "details": {
    "accepted": false,
    "operationPlanId": "OP-001",
    "routeId": "R_NORTH",
    "resultCode": "INTERLOCKING_REJECTED",
    "failureCode": "ROUTE_CONFLICT",
    "retryable": true,
    "rawReason": "Route R_NORTH conflicts with established route R_MAIN",
    "interlockingState": "CONFLICTED"
  }
}
```

## 标准失败码

信号侧使用 `InterlockingFailureCode` 返回失败码。调度侧根据 `retryable` 判断后续动作。

| failureCode | retryable | 调度建议 |
|---|---:|---|
| `ROUTE_ESTABLISHED` | 否 | 建路成功，推进计划 |
| `ROUTE_NOT_FOUND` | 否 | 配置错误，转人工 |
| `NO_MATCHING_ROUTE` | 否 | 配置错误或请求不完整，转人工 |
| `TRACK_NOT_FOUND` | 否 | 基础数据不一致，转人工 |
| `TRACK_OCCUPIED` | 是 | 等待冷却后重试 |
| `TRACK_FAULT` | 是 | 可等待，也可触发改路策略 |
| `ROUTE_CONFLICT` | 是 | 排队等待或重试 |
| `SWITCH_LOCKED` | 是 | 等待释放后重试 |
| `SWITCH_MOVE_FAILED` | 是 | 等待或转人工检查道岔 |
| `ROUTE_NOT_AVAILABLE` | 是 | 等待状态迁移后重试 |
| `SWITCH_POSITION_MISMATCH` | 否 | 进路/道岔配置矛盾，转人工 |
| `UNSUPPORTED_COMMAND` | 否 | 接口命令错误，转人工 |
| `INTERNAL_ERROR` | 否 | 信号内部异常，转人工 |

## 当前代码落点

- 调度生成运营计划进路命令：`DispatchService.automaticOperationPlanRouteCommands`
- 总循环转发进路命令：`SimulationRuntime.advanceOneTick`
- 信号进路请求对象：`RouteInterlockingService.DispatchRouteRequest`
- 信号处理入口：`RouteInterlockingService.applyDispatchCommand`
- 调度反馈入口：`DispatchService.acceptFeedback`
- OperationPlan 反馈更新：`DispatchService.updateOperationPlanFromFeedback`
- 进路记录更新：`RouteDispatchRecordStore.updateFromFeedback`

## 第一版联调用例

### 用例 1：正常建路

1. 调度创建 `OperationPlan`
2. 到计划发车时间生成 `REQUEST_ROUTE`
3. 信号收到 `trainId + routeId + operationPlanId`
4. 信号返回 `accepted=true`
5. 调度将计划更新为 `ROUTE_ACCEPTED`

### 用例 2：进路冲突

1. `R_MAIN` 已被其它列车锁闭
2. 调度请求 `R_NORTH` 或冲突路线
3. 信号返回 `failureCode=ROUTE_CONFLICT`、`retryable=true`
4. 调度保留计划，等待冷却后重试

### 用例 3：路线配置错误

1. 调度请求不存在的 `routeId`
2. 信号返回 `failureCode=ROUTE_NOT_FOUND`、`retryable=false`
3. 调度将计划置为 `ROUTE_REJECTED`，前端提示人工处理

### 用例 4：取消进路

1. 调度发送 `CANCEL_ROUTE`
2. 信号释放进路和道岔锁闭
3. 信号返回 `ROUTE_CANCELLED`
4. 调度更新预约状态

## 后续扩展

第一版只要求信号按完整 `routeId` 建路。后续可扩展：

- 支持 `routeId + segmentIds` 建立局部进路
- 信号反馈 `lockedSwitchIds`
- 信号反馈 `authorityReady`
- 信号反馈当前 MA 终点和限速
- 调度根据 `failureCode` 自动选择替代路线
- `OperationPlan` 到点后自动创建车辆并绑定运行任务
