# 8. 车辆控制模块详细设计

## 8.1 模块概述

车辆控制模块负责在中心仿真主循环内维护多车实体状态，将调度、信号/轨道、供电和车辆自身约束汇聚为牵引、制动、惰行等控制输入，并通过车辆物理端口推进列车位置、速度、加速度、牵引功率、制动功率和能耗。

模块当前采用“单车基层智能子系统边界 + 车辆物理端口 + 本地/外部仿真适配”的结构：

| 职责 | 代码/配置 |
|---|---|
| 多车实体管理 | `TrainManager`、`TrainEntity`、`TrainState` |
| 外部车辆控制会话代理 | `ExternalTrainControlSession`、`SignalTrainLifecycleCommand` |
| 外部车辆运行时 | `vehicle-runtime-service`、`VehicleRuntimeInstance`、`VehicleControlQueue`、`VehicleSimulationQueue` |
| 单车基层智能子系统 | `OnboardTrainSubsystemManager`、`OnboardTrainSubsystem`、`OnboardTrainControlInput`、`OnboardTrainControlOutput` |
| 车辆控制节点客户端和注册表 | `OnboardTrainSubsystemClient`、`LocalOnboardTrainSubsystemClient`、`HttpOnboardTrainSubsystemClient`、`OnboardTrainNodeState` |
| 车辆物理端口 | `VehiclePhysicsClient`、`VehiclePhysicsInput`、`VehiclePhysicsOutput` |
| 本地动力学模型 | `SimpleVehicleDynamicsModel`、`config/train_params.yaml` |
| 外部仿真协议适配 | `FmuVehiclePhysicsAdapter`、`com.railwaysim.vehicle.external` |
| 车辆状态输出 | `TrainStateReport`、`RealtimeStateCache`、`VehicleController`、`TrainController` |

设计边界：

1. `vehicle-runtime-service` 是推荐车辆侧运行时入口，车辆仿真系统先创建 `VehicleRuntimeInstance` 并唤醒本车 `VehicleControlQueue`。
2. `OnboardTrainSubsystemManager` 是中央侧 fallback 和纳管视图入口，可按配置选择进程内节点、外部 HTTP 节点或影子比对。
3. 已删除旧控制输入兼容门面，车辆本地控制逻辑由 onboard 边界承接。
4. `EXTERNAL_HTTP` 模式下，`POST /vehicle-runtime/step-fleet` 是车辆控制和动力学权威端口；9300 失败时中央中止当前 tick，不静默产生另一套本地结果。`VehiclePhysicsClient.stepFleet()` 只用于显式 `LOCAL/DUAL_SHADOW` 兼容模式。
5. 车辆模块不实现信号联锁、轨道拓扑决策、调度策略和供电分区状态。
6. 外部车辆模型只替代车辆控制/动力学计算，不越级读取或接管供电设备状态。
7. 外部车辆控制系统作为独立子系统接入；中央系统只维护接入会话、信号网并入状态和电网并入状态，不负责车辆控制系统进程生命周期和内部控制状态机。

## 8.2 功能设计

### 8.2.0 外部车辆控制会话

列车热插拔不表示中央系统创建车辆控制系统实例。真实责任边界为：

1. 车辆仿真系统随列车侧启动一个 `VehicleRuntimeInstance`。
2. 车辆运行时唤醒本车 `VehicleControlQueue`，车辆控制系统实例进入待约束状态。
3. 车辆运行时调用中央 `/api/trains/runtime-registrations`，中央建立或更新本地列车实体和外部控制会话。
4. 信号侧 ADD/DELETE/CLEAR 仍作为兼容协议保留，主要用于旧联调、下线和清空。
5. 中央系统在会话未 `IN_SERVICE` 前禁止牵引，只允许制动/保持安全状态。
6. 信号网并入、电网并入状态通过 `TrainState` 暴露给调度、信号、供电和综合监控。

会话状态：

| 状态 | 含义 |
|---|---|
| `CONNECTING` | 外部车辆控制子系统已声明接入，中央侧尚未完成网络并入。 |
| `SIGNAL_ATTACHING` | 等待信号侧移动授权和安全上下文。 |
| `POWER_ATTACHING` | 等待供电约束可用或确认受流条件。 |
| `ONLINE_STANDBY` | 信号网和电网均已就绪，等待进入运营态。 |
| `IN_SERVICE` | 可参与主循环车辆物理推进。 |
| `SIGNAL_DETACHING` | 退出信号网，停止接受新的运行授权。 |
| `POWER_DETACHING` | 退出电网/受流关系。 |
| `DISCONNECTED` | 中央侧可清理本地列车实体。 |

车辆上线启动步骤：

1. 中央后端启动后，`TrainManager.reset()` 只创建默认演示车或恢复基础仿真态，不代表真实车辆控制子系统已经由中央启动。
2. 车辆仿真系统调用 `vehicle-runtime-service:9300` 的 `POST /vehicle-runtime/trains/launch`，先创建本车 `VehicleRuntimeInstance`。
3. `VehicleRuntimeInstance.launch()` 唤醒该车 `VehicleControlQueue`，使随车控制实例进入可消费 MA、轨道约束和供电约束的状态。
4. `vehicle-runtime-service` 调用中央 `POST /api/trains/runtime-registrations`，中央只建立 `TrainEntity` 状态镜像和 `ExternalTrainControlSession.CONNECTING`，不再反向注册 9300，避免启动环路。
5. 后续 tick 中，信号模块提供 `MovementAuthority`，供电模块提供 `PowerConstraint`，会话依次完成信号网和电网并入。
6. 会话进入 `IN_SERVICE` 后，列车参与 `tickAll()`、外部 `POST /vehicle-runtime/step-fleet` 或本地 fallback、供电负荷回写和快照输出。

`POST /api/trains/lifecycle` 仍保留为中央侧兼容/联调入口，适合旧演示、DELETE/CLEAR 或信号生命周期语义测试；推荐的上线启动入口不再是该接口。

### 8.2.1 多车实体推进

`TrainManager.tickAll()` 在每个仿真 tick 中批量处理当前所有列车：

1. 读取当前 `TrainState`。
2. 按列车 ID 合并信号下发的 `MovementAuthority`、`TrackConstraint` 和 `PowerConstraint`。
3. `EXTERNAL_HTTP` 时统一调用 `vehicle-runtime-service /vehicle-runtime/step-fleet`；失败明确中止 tick。`LOCAL/DUAL_SHADOW` 才通过 `OnboardTrainSubsystemManager + VehiclePhysicsClient` 执行中央兼容链路。
4. 将外部或本地返回的 `VehiclePhysicsOutput` 与 `TrainStateReport` 合并回中央状态镜像。
5. 将 `VehiclePhysicsOutput` 应用回 `TrainEntity`，生成 `TrainStateReport`。
6. 更新实时缓存并发布牵引、制动、再生和车辆物理事件。

### 8.2.2 单车基层智能子系统控制输入汇聚

输入来源：

| 输入 | 来源模块 | 车辆侧作用 |
|---|---|---|
| 速度授权、移动授权终点 | 信号模块 | 限速、ATP 制动、移动授权耗尽保护。 |
| 坡度、曲线半径、车站距离 | 轨道模块 | 阻力、停车距离、进站制动。 |
| 电压、可用功率、受流状态 | 供电模块 | 牵引可用性、降额、失电制动。 |
| 扣车、临时限速、限速比例 | 调度模块，经信号模块折算为 MA/限速 | 车辆不直接消费调度约束。 |
| 门状态、自检、牵引/制动可用、载重 | 车辆自身状态 | 本地安全保护和故障降级。 |
| PLC 方向手柄、主手柄、牵引/制动级位、紧急制动、门和 ATO 请求 | 9300 `DriverCabInputController -> DriverCommandHolder` | 仅作为非安全命令候选；不能覆盖 MA、车门/自检和供电约束。 |

输出控制量：

- `tractionCommand`：0-1 牵引比例。
- `brakeCommand`：0-1 常用制动比例。
- `emergencyBrakeCommand`：紧急制动标志。
- `dynamicsState`：车辆动力学状态机状态。
- `dynamicsConstraintReason`：控制约束原因。

### 8.2.3 车辆动力学计算

本地模型 `SimpleVehicleDynamicsModel` 计算：

- 牵引力、制动力、再生制动力
- 运行阻力、坡度阻力
- 加速度、速度、位置
- 牵引功率、取流电流、再生功率
- 累计消耗电能、累计再生电能
- 车辆物理故障码

外部仿真接入后，仍返回同一个 `VehiclePhysicsOutput`，上层状态和接口不变。

### 8.2.4 外部仿真协议适配

新增 `vehicle-runtime-service` 作为车辆侧外部运行时。它不是前端接口；列车启动时由车辆仿真系统先创建 `VehicleRuntimeInstance`，唤醒本车 `VehicleControlQueue`，随后调用中央注册接口建立 `TrainEntity` 状态镜像。PLC 46 字节输入直接进入 9300，不再通过 8080 或 `step-fleet.driverCommands[]` 转发。后续 tick 中，中央发送列车状态、信号 MA、轨道和调度约束；9300 自行取得 9200 权威供电约束，并把 PLC 命令与安全约束仲裁。返回的 `TrainStateReport` 携带 `decisionSource/inputCommandId/inputTraceId`，8080 据此建立中央决策镜像。

接口迁移评估单独记录在 `docs/车辆运行时接口迁移评估.md`。结论是中央车辆 REST、信号-车辆 REST、综合监控入口和数据库持久化仍保留在中央；外部车辆运行时承接启动实例、唤醒控制队列、同步 tick 和仿真层负荷转发。

通过 `railway.simulation.vehicle-runtime.mode` 切换：

| 模式 | 行为 |
|---|---|
| `LOCAL` | 默认本地链路：`OnboardTrainSubsystemManager + VehiclePhysicsClient`。 |
| `EXTERNAL_HTTP` | 调用 `vehicle-runtime-service:9300`，该服务是生产权威执行者；失败时中止 tick，不在 8080 静默 fallback。 |
| `DUAL_SHADOW` | 本地输出为权威，外部服务只做影子计算和健康监控。 |

当前 P0/P1-1 实现还覆盖：9300 未知列车/非法 PLC 包拒绝、PLC 到新车辆状态 P95 `< 200 ms`、`simulationRunId` 贯穿 8080→9300→9200、最终决策按 run/tick 落库。真实 MySQL 重启回放和完整前端可见延迟仍需环境验收。

通过 `railway.simulation.external-simulator.mode` 切换：

| 模式 | 实现 | 说明 |
|---|---|---|
| `LOCAL` | `LocalFallbackVehicleAdapter` | 默认本地模型；兼容旧 HTTP FMU 服务。 |
| `EXTERNAL_UDP` | `ExternalUdpVehicleAdapter` | 对接 20ms 高频 UDP，1-20 车小端 double 报文。 |
| `EXTERNAL_RTLAB_API` | `RtLabApiVehicleAdapter` | 对接 RT-LAB API 变量路径，当前为 stub。 |
| `DUAL_SHADOW` | `ShadowCompareAdapter` | 本地输出为权威，外部输出只做误差比对。 |

## 8.3 软件结构设计

```text
SimulationRuntime
  -> TrainManager
       -> ExternalTrainControlSession
       -> OnboardTrainSubsystemManager
            -> LocalOnboardTrainSubsystemClient / HttpOnboardTrainSubsystemClient
            -> OnboardTrainNodeRegistry
            -> OnboardTrainSubsystem(trainId)
       -> VehicleRuntimeIntegrationService
            -> LOCAL / EXTERNAL_HTTP / DUAL_SHADOW
            -> vehicle-runtime-service:9300
                 -> POST /vehicle-runtime/trains/launch
                 -> VehicleRuntimeInstance
                      -> VehicleControlQueue
                      -> VehicleSimulationQueue
                 -> 可选推送供电负荷到 power-network-service:9200
       -> VehiclePhysicsClient
            -> FmuVehiclePhysicsAdapter
                 -> LOCAL / HTTP FMU
                 -> ExternalUdpVehicleAdapter
                 -> RtLabApiVehicleAdapter
                 -> ShadowCompareAdapter
       -> TrainEntity.applyPhysicsOutput()
       -> RealtimeStateCache
       -> SimpleEventBus
```

核心类职责：

| 类/组件 | 职责 |
|---|---|
| `TrainEntity` | 单车运行状态、故障注入状态和物理输出应用。 |
| `TrainState` | 对外统一列车状态 record。 |
| `ExternalTrainControlSession` | 外部车辆控制子系统在中央侧的会话、信号网并入和电网并入状态。 |
| `SignalTrainLifecycleCommand` | 信号侧 ADD/DELETE/CLEAR 生命周期协议模型。 |
| `SignalTrainLifecycleCommandCodec` | `0xff 0xf1` 小端生命周期报文编解码。 |
| `OnboardTrainSubsystemManager` | 中央侧车辆控制节点编排器，按配置选择本地、外部 HTTP 或影子模式。 |
| `OnboardTrainSubsystemClient` | 中央到车辆控制节点的客户端接口，隐藏本地/HTTP/后续协议差异。 |
| `OnboardTrainNodeState` | 车辆控制节点注册表快照，记录在线、降级、数据质量和租约时间。 |
| `OnboardTrainSubsystem` | 模拟单辆列车上的基层智能子系统，执行本车自检、保护和 ATO/TCMS 本地决策。 |
| `OnboardTrainControlInput/Output` | 中央与进程内单车子系统之间的接口模型。 |
| `TrainStateReport` | 单车基层智能子系统生成的车辆控制状态报告；外部模式回传 `decisionSource/inputCommandId/inputTraceId`。 |
| `VehicleRuntimeIntegrationService` | 中央侧车辆运行时编排器，负责外部 HTTP、影子模式和本地 fallback。 |
| `vehicle-runtime-service` | 外部车辆运行时服务，负责主动启动车辆仿真实例、唤醒控制队列，并向中央注册状态镜像。 |
| `SimpleVehicleDynamicsModel` | 本地车辆纵向动力学 fallback。 |
| `FmuVehiclePhysicsAdapter` | 车辆物理端口实现和模式分发。 |
| `ExternalVehicleCommandMapper` | 内部牵引/制动输入到外部 `command + percent` 的映射。 |
| `ExternalSegmentMapper` | 线路里程到外部 `segNo + offset` 的映射。 |
| `ExternalUdpPacketCodec` | UDP 小端报文编码/解码。 |
| `RtLabVariablePathMapper` | RT-LAB 1-20 车变量路径生成。 |

## 8.4 业务流程设计

### 8.4.1 启动注册流程

```text
POST vehicle-runtime-service / vehicle-runtime / trains / launch
  -> 创建 VehicleRuntimeInstance
  -> 唤醒 VehicleControlQueue
  -> POST backend / api / trains / runtime-registrations
  -> 中央 TrainManager.registerRuntimeStartedTrain()
  -> 后续 tick 进入中央统一调度
```

`POST /api/trains/lifecycle` 仍可用于中央侧兼容联调，但推荐启动入口是外部车辆运行时的 `launch`。

### 8.4.2 正常运行流程

```text
TrainState
  -> VehicleRuntimeIntegrationService
  -> LOCAL: OnboardTrainSubsystemManager + VehiclePhysicsClient
  -> EXTERNAL_HTTP: vehicle-runtime-service / step-fleet
       -> forwardPowerLoads: POST power-network-service / state/query
  -> VehiclePhysicsOutput + TrainStateReport
  -> TrainEntity.applyPhysicsOutput()
  -> RealtimeStateCache.updateTrainPhysics/updateTrainTcmsState()
  -> WebSocket/REST 输出
```

### 8.4.3 动力学状态机

| 状态 | 触发条件 | 控制输出 |
|---|---|---|
| `SELF_CHECK_BLOCKED` | 门未锁闭、自检失败、牵引不可用 | 切牵引，按速度施加制动。 |
| `SAFETY_BRAKE` | 移动授权耗尽或 ATP 安全制动 | 紧急制动。 |
| `POWER_LOSS` | 失电、受流不可用、可用功率为 0 | 切牵引，常用制动。 |
| `OVERLOAD_DERATED` | 动态载重超限或牵引单元不足 | 降低牵引比例，输出车辆保护原因。 |
| `POWER_DERATED` | 供电降额但仍可受流 | 降低牵引比例。 |
| `MA_BRAKE` | 移动授权距离接近停车距离 | 比例制动。 |
| `STATION_BRAKE` | 进站停车距离不足 | 比例制动。 |
| `STATION_STOPPED` | 进入停车窗口且接近零速 | 保持制动。 |
| `DEPARTING_STATION` | 调度/信号释放停站后，列车尚未离开本站捕获窗口 | 牵引离站，并暂时屏蔽同一站点再次触发停车。 |
| `OVERSPEED_BRAKE` | 超过限速 | 比例制动。 |
| `ACCELERATING` | 目标速度有明显裕量 | 牵引。 |
| `CRUISING` | 接近目标速度 | 小牵引维持。 |
| `COASTING` | 达到目标速度 | 惰行。 |

### 8.4.4 外部仿真降级流程

```text
外部车辆仿真调用
  -> 成功：合并外部 speed/acc/distance 到 VehiclePhysicsOutput
  -> 失败：发布 FmuStepFailedEvent 和 FmuFallbackActivatedEvent
  -> 使用 SimpleVehicleDynamicsModel 回退
  -> faultCode=EXTERNAL_SIM_FALLBACK
  -> TrainState.dataQuality=FALLBACK
```

## 8.5 数据设计

### 8.5.1 车辆输入

`VehiclePhysicsInput`：

| 字段 | 说明 |
|---|---|
| `trainId` | 列车 ID。 |
| `positionMeters/speedMetersPerSecond` | 当前里程和速度。 |
| `trainMassKg` | 车辆质量，含载客率影响。 |
| `tractionCommand/brakeCommand/emergencyBrakeCommand` | 车辆控制指令。 |
| `speedLimitMetersPerSecond/movementAuthorityDistanceMeters` | 信号/轨道约束；调度影响应已由信号折算。 |
| `gradient/curveRadiusMeters` | 线路约束。 |
| `railVoltage/powerAvailableWatts` | 供电约束。 |
| `doorClosed/adhesionCoefficient` | 车辆自身状态和轮轨条件。 |
| `previousEnergyConsumedKwh/previousEnergyRegeneratedKwh` | 能耗累计。 |
| `dynamicsState/dynamicsConstraintReason` | 状态机诊断。 |
| `stationDistanceMeters/stoppingDistanceMeters` | 进站停车诊断。 |

### 8.5.2 车辆输出

`VehiclePhysicsOutput`：

| 字段 | 说明 |
|---|---|
| `newPositionMeters/newSpeedMetersPerSecond` | 新位置和速度。 |
| `accelerationMetersPerSecondSquared` | 加速度。 |
| `tractionForceNewtons/brakeForceNewtons/regenBrakeForceNewtons` | 牵引、制动和再生制动力。 |
| `tractionPowerWatts/railCurrentAmps/regenPowerWatts` | 供电耦合数据。 |
| `energyConsumedKwh/energyRegeneratedKwh` | 能耗累计。 |
| `faultCode` | `OK`、`LOW_VOLTAGE`、`CURRENT_COLLECTION_LOST`、`ATP_BRAKE`、`FMU_STEP_FAILED`、`EXTERNAL_SIM_FALLBACK` 等。 |

### 8.5.3 对外车辆状态

`TrainState` 主要字段：

| 字段 | 说明 |
|---|---|
| `controlSessionState` | 外部车辆控制会话状态。 |
| `signalNetworkStatus/powerNetworkStatus` | 信号网、电网并入状态。 |
| `controlSessionReason` | 会话状态原因。 |
| `linkId/direction` | 协议 link ID 和上下行方向。 |
| `operationMode` | `ATO`、`ATP_BRAKE`、`DEGRADED` 等。 |
| `zeroSpeed` | 零速状态。 |
| `doorState` | 车门状态，当前主要为 `CLOSED_LOCKED` 或故障注入后的 `OPEN`。 |
| `tractionState/brakeState/currentCollectionStatus` | TCMS 牵引、制动、受流状态。 |
| `tractionAvailable/brakeAvailable/selfCheckStatus` | 自检和可用性。 |
| `loadMassKg/overloadStatus` | 协议载重和车辆超载状态。 |
| `availableTractionCount/availableBrakeCount` | 可用牵引/制动单元数量。 |
| `vehicleProtectionReason` | 车辆本地保护原因，供信号侧消费。 |
| `faultLevel/availableOperationMode/dataQuality` | 故障等级、可用运行模式和数据质量。 |
| `dynamicsState/dynamicsConstraintReason` | 动力学状态和约束原因。 |
| `tractionPowerWatts/railCurrentAmps/regenPowerWatts` | 车辆-供电耦合输出。 |

`VehicleSignalStatus` 在 `TrainState` 基础上额外提供 `driverConsoleState`，用于信号系统的司机台状态投影。拆分部署中 PLC 原始输入不在 8080 执行；8080 有中央/信号侧 `DriverCabStateSnapshot` 时优先展示，否则从最终 `TrainState` 派生可观测状态。真实 PLC 命令是否被执行，必须以 9300 报告和中央 `VehicleControlDecision` 为准，不以展示投影反推。

### 8.5.4 外部协议数据

| 模型 | 字段 |
|---|---|
| `ExternalTrainCommand` | `trainNo`、`command`、`percent`、`segNo`、`offset`、`direction`、`activeCab`。 |
| `ExternalTrainTelemetry` | `trainNo`、`activeCab`、`direction`、`acceleration`、`speed`、`cumulativeDistance`。 |
| `ExternalSimulatorHealth` | `mode`、`heartbeatStatus`、`lastPacketAt`、`latencyMillis`、`dataQuality`。 |
| `SignalCabDisplayState` | `currentDrivingMode`、`maximumAvailableDrivingMode`、`doorEnable`、`doorControlMode`、`tractionBrakeInfo`、`departureInfo`、`turnbackInfo`、`speedLimitMetersPerSecond`、`emergencyBrake`、`distanceToNextStationMeters`。 |
| `DriverCabPlcInputPacket` | 司机台 PLC 到车辆控制系统的钥匙、门模式、ATO 启动、折返、方向手柄、主手柄和级位。 |
| `DriverCabPlcOutputPacket` | 车辆控制系统到司机台 PLC 的指示灯、门关好、网络故障、ATO/折返可用和激活状态。 |

### 8.5.5 运行主键与持久化证据

- `DispatchService.simulationRunId` 是唯一运行 ID 来源，`SimulationRunContext` 仅镜像当前 `runId/tick`，不生成第二个 ID。
- `SimulationSnapshot`、9300 `VehicleRuntimeStepRequest`、9200 `PowerNetworkStepRequest` 使用同一 `simulationRunId`。
- `vehicle_control_command_log` 按 `(simulation_run_id,tick,train_id)` 保存最终决策，并保存 `command_id/trace_id/decision_source/reason_code`。
- `train_physics_snapshot`、`train_energy_record`、`power_section_record`、`track_occupancy_record`、`signal_state_record` 和 `fmu_fault_log` 均携带 `simulation_run_id`。
- reset 只清临时车辆/控制态；旧 run 记 `CANCELLED_BY_RESET`，历史决策和快照保留供只读回放。

## 8.6 接口设计

### 8.6.1 对调度模块

车辆模块不直接消费 `DispatchConstraint`。调度系统提交扣车、临时限速、速度比例限制后，由信号模块计算 `MovementAuthority` 和速度限制，再通过信号-车辆接口交付车辆。

### 8.6.2 对信号/轨道模块

输入 `MovementAuthority` 和 `TrackConstraint`，用于移动授权、限速、坡度、曲线半径、车站距离和停车保护。车辆模块输出 `zeroSpeed`、门锁闭、牵引/制动可用、受流状态、动态载重、超载状态和 `vehicleProtectionReason` 等字段供信号/轨道联调。

### 8.6.3 对供电模块

输入`PowerConstraint`，输出`VehiclePhysicsOutput`中的`railCurrentAmps`、`tractionPowerWatts`、`regenPowerWatts`。非拆分部署的`LOCAL`、`DUAL_SHADOW`模式由中央供电模块根据这些输出更新分区负荷；拆分部署的`EXTERNAL_HTTP + forwardPowerLoads=true`固定由外部车辆运行时把分区负荷写入供电仿真，中央供电模块只拉取供电状态并做电压偏差监控。写入所有权不因FMU单车或整批降级而切换，避免中央与9300双写。

### 8.6.4 REST 和 WebSocket

REST：

```http
GET /api/trains
GET /api/trains/{trainId}
GET /api/trains/{trainId}/energy
GET /api/trains/{trainId}/faults
GET /api/vehicle/maintenance-states
POST /api/trains/runtime-registrations
POST /api/trains/lifecycle
POST /api/trains/{trainId}/faults
POST /api/trains/{trainId}/faults/clear
GET /api/simulation-runs/{runId}/train-snapshots
GET /api/simulation-runs/{runId}/control-decisions
```

WebSocket 快照中的 `trains[]` 复用 `TrainState` 字段。

`POST /api/trains/runtime-registrations` 是 `vehicle-runtime-service` 主动注册中央状态镜像的服务间入口。`POST /api/trains/lifecycle` 对齐信号生命周期协议 ADD/DELETE/CLEAR，仅作为中央侧兼容/联调入口。两个接口都不替代信号系统下发移动授权，也不暴露车辆内部牵引/制动控制参数给调度系统。

典型 ADD 请求：

```json
{
  "action": "ADD",
  "trains": [
    { "trainNo": 3, "linkId": 12, "offsetMeters": 640.0, "direction": "DOWN" }
  ],
  "reason": "external train control attached",
  "operator": "signal",
  "confirmToken": "SIMULATION_CONFIRM",
  "traceId": "signal-train-003"
}
```

返回值是中央侧已创建或已存在的 `TrainState` 列表。新车通常先处于 `CONNECTING`，需要后续 tick 才会根据 MA 和供电约束推进到 `IN_SERVICE`。

司机台 PLC 临时联调入口（方案 A — 输入侧在 9300，输出侧在 8080）：

```http
POST http://localhost:9300/api/vehicle/driver-cabs/{trainId}/plc-input
GET  http://localhost:8080/api/vehicle/driver-cabs/{trainId}/state
GET  http://localhost:8080/api/vehicle/driver-cabs/{trainId}/plc-output
```

- `POST plc-input` 由外部测试脚本调用，使用 `application/octet-stream` 提交 46 字节 PLC 输入报文到 `vehicle-runtime-service:9300`。非法枚举码和越界百分比（0–100 范围外）返回 `DriverCommandAcceptance{accepted:false, reasonCode:"DECODE_FAILED"}`，不再静默截断。合法报文返回 `{accepted:true, commandId, reasonCode:"ACCEPTED", receivedAt, expiresAt}`。
- 9300 必须先存在已启动的列车实例；未知列车返回 `404/UNKNOWN_TRAIN`，命令不存入 Holder。
- `GET state/plc-output` 由 `backend:8080` 提供。`plc-output` 根据 `TrainState` 和信号命令显示状态生成协议 7.2 的 26 字节报文。
- 真实硬件接入时应替换为车辆控制系统侧 TCP 适配器（9300），REST 仅用于前期仿真和自动化测试。

### 8.6.5 外部仿真协议

UDP：

- 模型侧：`192.168.200.110:23001`
- 平台侧：`192.168.200.102:23002`
- 字节序：小端
- 平台到模型：1-20 车 `command + percent`
- 模型到平台：1-20 车加速度、速度、累计里程

RT-LAB API：

- 输入路径根：`PowerSystemAndTrainsV1/.../Train_Control/...`
- 每车输入变量：列车编号、指令、`segNo`、偏移、方向、激活端

## 8.7 核心算法或控制逻辑设计

### 8.8.1 控制优先级

车辆控制决策按安全优先：

```text
紧急制动
  > MA/ATP 安全约束
  > 车门/钥匙/自检/制动可用性
  > 供电失电、受流不可用和降额约束
  > MANUAL 模式下的 PLC 人工命令
  > AUTO/INTELLIGENT 自动命令
  > 巡航/惰行基线
```

`MANUAL/AUTO/INTELLIGENT` 只决定非安全命令来源，不会改变前四层安全约束的优先级。

### 8.8.2 牵引和制动生成

牵引规则：

```text
if blocked by safety/power/door/self-check:
    tractionCommand = 0
elif power derated:
    tractionCommand = tractionForSpeedMargin * powerDeratingFactor
elif speedMargin large:
    tractionCommand = clamp(speedMargin / targetBand, 0, 1)
elif near target:
    tractionCommand = min(command, 0.25)
else:
    tractionCommand = 0
```

制动规则：

```text
if emergencyBrake:
    brakeCommand = 1
elif signalHold or powerLoss:
    brakeCommand = speed > 0.1 ? serviceBrake : holdingBrake
elif distance limit reached:
    brakeCommand = proportionalBrake
elif overspeed:
    brakeCommand = clamp(overspeed / 3.0, 0.2, 1)
else:
    brakeCommand = 0
```

### 8.8.3 车辆动力学

```text
F_traction = maxTractionForce * tractionCommand * powerFactor * adhesionFactor
F_brake = emergency ? maxEmergencyBrakeForce : maxServiceBrakeForce * brakeCommand
F_resistance = DavisA + DavisB * v + DavisC * v^2
F_gradient = mass * g * gradient
F_net = F_traction - F_brake - F_resistance - F_gradient
a = clamp(F_net / mass, -1.3, 1.0)
v_new = max(0, v + a * dt)
x_new = x + (v + v_new) / 2 * dt
```

### 8.8.4 外部指令映射

| 内部输入 | 外部协议 |
|---|---|
| `tractionCommand > 0` | `command=1`，`percent=round(tractionCommand*100)` |
| `brakeCommand > 0` | `command=2`，`percent=round(brakeCommand*100)` |
| `emergencyBrakeCommand=true` | `command=2`，`percent=100` |
| 无牵引无制动 | `command=0`，`percent=0` |

方向码预留：`UP=0x55`、`DOWN=0xaa`、`UNKNOWN=0xff`。

`segNo + offset` 由 `ExternalSegmentMapper` 根据 `OperationalLineData` 和 `external-simulator.segment-mapping` 生成；外部段号最终需与信号/轨道共同确认。

### 8.8.5 停车结果评价

`TrainStopEvaluationService` 消费 9300 控制后已更新的中央 `TrainState`，从 `STATION_BRAKE` 或 `MA_BRAKE` 开始建立一次停车 episode，不从全 run 累计平稳性指标。当列车进入 `STATION_STOPPED`、速度不高于 `zero-speed-meters-per-second`，且停稳时间达到 `stable-seconds` 时，每次进站只生成一条 `TrainStopResult`。

- 站台目标使用站点中心里程；停车过程出现 `MA_BRAKE/MA_DISTANCE_LIMIT` 时，使用 `position + movementAuthorityDistance` 形成 `MOVEMENT_AUTHORITY` 目标。
- 签名误差固定为 `actualPosition-targetPosition`，正值表示越过目标方向；同时保存绝对误差和越站布尔值。
- 停车 episode 内累计最大减速度、最大 jerk、制动状态切换次数和是否使用紧急制动。
- 按实际状态记录 `APPROACH/BRAKE_1/BRAKE_2/FINE_STOP/HOLD/OVERRUN` 阶段历史，只增加证据，不改写安全仲裁结果。
- 结果以 `simulationRunId/trainId/stationId/episodeStartTick` 组成稳定 ID，写入 `train_stop_result`，并可通过 run 历史 API 查询。

中央本地兼容控制与 9300 已使用同一 `STOPPING_V1` 默认值：服务制动减速度、停车窗口、进站缓冲、有效减速度上下限和零速阈值均配置化。MA 终点更近时建立显式 `StoppingTarget` 并发布 `TARGET_OVERRIDDEN_BY_MA`。`stop-statistics` 按场景汇总 10 次 MAE/P95/方差/成功率和失败原因。尚未完成的是 AW0/AW2/AW3、坡度、失电、MA 缩短等真实业务场景的连续 10 次验收报告。

## 8.8 异常处理设计

| 异常 | 处理 |
|---|---|
| 车门未锁闭 | 禁止牵引，状态为 `SELF_CHECK_BLOCKED`。 |
| 牵引不可用 | 切牵引，故障等级提升。 |
| 制动不可用 | 自检失败，禁止发车或降级运行。 |
| 移动授权耗尽 | 紧急制动，状态为 `SAFETY_BRAKE`。 |
| 供电失电 | 切牵引，常用制动，受流状态 `LOST`。 |
| 供电欠压 | 降额牵引，受流状态 `LOW_VOLTAGE`。 |
| HTTP FMU 失败 | 本地 fallback，`faultCode=FMU_STEP_FAILED`，`dataQuality=FALLBACK`。 |
| UDP/API 外部仿真失败 | 本地 fallback，`faultCode=EXTERNAL_SIM_FALLBACK`，`dataQuality=FALLBACK`。 |
| 外部协议缺车或报文长度不足 | 丢弃本次外部输出，按 fallback 推进。 |
| 外部车辆控制节点超时 | `OnboardTrainSubsystemManager` 回落进程内节点，`OnboardTrainNodeState.dataQuality=FALLBACK`。 |

## 8.9 安全性设计

1. 车辆故障注入写接口必须带 `SIMULATION_CONFIRM`，并写操作日志。
2. 外部 UDP/API 模式默认关闭，默认 `LOCAL`，避免开发环境误连内网仿真机。
3. RT-LAB 真实依赖不进入主业务逻辑，先通过接口和 stub 隔离。
4. 外部仿真失败不会阻塞主循环，必须降级到本地模型。
5. 车辆模块不直接接收调度越级参数，不把底层仿真参数暴露给调度系统控制。

## 8.11 开发备注

1. 车辆物理端口保持稳定，上层 `TrainManager` 和快照接口不感知本地、FMU、UDP 或 RT-LAB 的差异。
2. 前期自研仿真仍为默认模式；外部仿真建议先使用 `DUAL_SHADOW` 验证误差，再切换为主用。
3. 车轨 `segNo/offset` 属于跨模块接口，车辆模块只提供映射入口，不写死外部段号。
4. 当前外部协议没有独立供电 PSCADA 点表，车辆模型不接管供电分区状态。
5. 后续若新增车辆故障、自检或维护记录表，需要同步更新 `/database` 留档。
6. 云库联调时，实时接口成功不等价于持久化 schema 已同步；若 `SimulationPersistenceService` 报 `train_physics_snapshot` 写入失败，应先执行 `/database` 中对应迁移。
