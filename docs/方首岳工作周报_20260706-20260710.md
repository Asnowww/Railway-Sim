本周完成了车辆与供电仿真从“中央后端内置简化模型”向“可拆分、可替换、可部署、可验收的真实 FMU 车电闭环”的阶段性演进。工作覆盖车辆动力学、线路数据、供电仿真、外部车辆运行时、信号—车辆接口、司机台与机房局域网协议、真实 FMU 构建与运行、三服务部署及自动化验收。

`8080 中央系统 → 9300 车辆运行服务 → 9000 FMPy/FMU 服务 → 9200 权威供电仿真服务`

## 四、主要完成工作

### 4.1 车辆动力学与线路数据基础

1. 建立车辆物理可替换接口，完成 `VehiclePhysicsClient`、`FmuVehiclePhysicsAdapter`、Java 简化动力学模型与 TCMS/ATO 适配链路。
2. 将车辆推进纳入统一仿真 tick，形成从中央运行时到列车状态更新的基础闭环。
3. 接入 `database/线路数据(1).xls`，解析线路约束并构建显式 `TrainDynamicsState` 状态机。
4. 支持线路数据落库及数据源切换，补齐动力学快照、诊断字段和数据库迁移。
5. 在状态机中覆盖自检阻断、等待发车、牵引、惰行、制动、站停等运行阶段，并加入门控、制动可用性和线路限制。

### 4.2 车辆—供电二阶段与外部运行时

1. 完成车辆载重计算、超载保护、受流状态及供电约束接入。
2. 完成车辆—信号接口族、生命周期命令和状态回传闭环。
3. 拆分车辆控制子系统边界，建立外部车辆运行时服务及中央接入适配层。
4. 加固外部运行时启动与注册流程：仅在远端成功后标记 bootstrap 完成；重复/旧 tick 不再污染生命周期状态；路径与请求体列车编号不一致时返回明确错误。
5. 将接口审计落库异步化，降低主仿真链路受数据库 I/O 的影响。

### 4.3 供电仿真与车电耦合

1. 重构供电控制，接入独立供电仿真服务脚手架，并修复外部供电拓扑初始化。
2. 优化供电拓扑，按真实配置建立 1500 V、5 个供电分区的权威供电仿真。
3. 明确中央控制与权威供电服务的职责边界，避免中央与外部服务重复写入负荷。
4. 实现分区牵引负荷、网压约束、可用功率、再生预算、已吸收及未吸收回馈计算。
5. 完成同段列车耦合、异段隔离、欠压/失电保护和一周期延迟的闭环语义。

### 4.4 信号、司机台与机房局域网适配

1. 统一信号—车辆—司机台协议字段，完善列车生命周期和内容帧编解码。
2. 实现信号视觉状态回填与指定列车 UDP 导出。
3. 实现司机台 PLC 适配层，建立司机台输入、车辆状态和显示数据之间的转换边界。
4. 实现 localnet 协议适配器，并依据实施计划补齐点表元数据、信号帧分发顺序、回放接口和测试。
5. 修复列车停站后离站复捕获问题，避免列车离站后被旧站台条件再次捕获。

### 4.5 真实 FMU 集成 WP0～WP8

1. **WP0～WP1：接口与参数基线。** 冻结跨服务 JSON、变量、单位、功率及生命周期契约；9300 与 9000 统一读取 `train_params.yaml`，使用一致的参数集哈希。
2. **WP2：车辆纵向动力学模型。** 以 Modelica 实现牵引、制动、再生、Davis 阻力、坡道、黏着、欠压、门控和能量积分。
3. **WP3：可复现构建。** 使用固定 OpenModelica 1.27.0 镜像导出 FMI 2.0 Co-Simulation FMU，并校验 Linux 二进制和变量清单。
4. **WP4：真实 FMPy 服务。** 9000 使用 FMPy 0.3.30 和持久 `FMU2Slave` 管理一车一实例，支持 INIT、STEP、RESET、RESYNC、删除、全量复位、批次预检和单实例错误隔离。
5. **WP5～WP6：车辆供电闭环。** 9300 按“全车 Prepare—单次批量 Execute—统一 Apply”推进车队；一次 tick 仅调用一次 9000 和一次 9200，实现车辆物理与供电约束闭环。
6. **WP7～WP8：部署与验收。** 完成 9000/9200/9300 的 Docker Compose 编排、健康依赖、故障恢复、参数一致性检查、性能与长稳验收自动化。
