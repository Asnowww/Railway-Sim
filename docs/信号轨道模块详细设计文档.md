## 1. 概述

信号轨道模块负责铁路仿真系统的核心运行时逻辑：**轨道拓扑建模 → 列车定位与占用染色 → 移动授权(MA)计算 → 进路联锁管理 → 调度命令执行与反馈**。本阶段主要完成了 M9 真实线路接入、上下行双线独立运行、调度发车闭环、以及三层命令反馈系统。

## 2. M9 真实线路数据接入

### 2.1 数据来源

线路数据来自 Excel 文件《视景系统公里标最新数据+站台位置20260703.xlsx》，包含：

| 表格 | 内容 | 行数 |
|------|------|------|
| Sheet1 上行视景边号公里标 | 13 条信号机闭塞区间 | 13 |
| Sheet2 下行视景边号公里标 | 12 条信号机闭塞区间 | 12 |
| Sheet3 站台位置 | 13 站 × 2 股道 = 26 行 | 26 |

### 2.2 线路规格

```
线路名称：北京地铁9号线（郭公庄 ↔ 国家图书馆）
总里程：16489.02m
站点：13 站（S101 郭公庄 → S113 国家图书馆）
运营模式：双线双向
  - 上行（up）：郭公庄 → 国家图书馆，里程递增 0→16489m
  - 下行（down）：国家图书馆 → 郭公庄，里程递减 16489→0m
```

### 2.3 YAML 配置文件

线路数据存储在 `config/line-m9.yaml`，由转换脚本 `scripts/convert_m9_xlsx.py` 从 Excel 自动生成。文件结构：

```yaml
line:
  id: m9
  name: 北京地铁9号线
  length_meters: 16489.02

stations:        # 13 站，每站含 up/down 双向站台
nodes:           # NU00-NU13(上行14节点) + ND00-ND12(下行13节点)
segments:        # U段13条 + D段12条 + 渡线2条
switches:        # SW_GGZ(郭公庄道岔) + SW_LIB(国图道岔)
signals:         # 28 个信号机(SIG-U* + SIG-D*)
routes:          # R_UP + R_DOWN + R_TB_LIB + R_TB_GGZ
```

## 3. 区段切分与拓扑构建

### 3.1 问题

原始 M9 数据中，信号机之间的闭塞区间长度分布极不均匀：

```
两端（站场区）：密集
  U02  0.0 -  215.5  (215m)
  U10  215 -  232    (17m)
  ...
中间（正线区间）：稀疏
  U20   511 - 5146   (4635m)  ← 包含 3 个站
  U23  5146 - 8352   (3205m)
  U34  8632 - 14119  (5487m)  ← 包含 4 个站
```

前端态势图按等比例里程渲染，长段导致一个红色色块跨越多站，视觉效果差。

### 3.2 解决方案

按站台中心公里标将 8 个长段（上下行各 4 个）切分为子段：

| 原段 | 长度 | 切分后 | 子段数 |
|------|------|--------|--------|
| U20 | 4635m | U20a/b/c/d | 4 |
| U23 | 3205m | U23a/b/c | 3 |
| U34 | 5487m | U34a/b/c/d/e | 5 |
| U42 | 1807m | U42a/b/c | 3 |
| D17 | 4763m | D17a/b/c/d | 4 |
| D21 | 3071m | D21a/b/c | 3 |
| D28 | 5352m | D28a/b/c/d/e | 5 |
| D36 | 2163m | D36a/b/c | 3 |

**结果**：上行 13→24 段，下行 12→23 段。切分点对应真实站台中心，新增 22 个站间节点。信号 `segment_id` 自动映射到对应子段，进路 `axle_section_ids` 同步展开。

### 3.3 拓扑邻居关系

`YamlLineDataLoader` 自动计算两条邻居链：

| 邻居类型 | 计算规则 | 用途 |
|----------|---------|------|
| forwardNeighbor | other.from == this.to | 进路连通性校验、前向路径搜索 |
| sideNeighbor | other.from == this.from && other.to != this.to | 分叉（道岔）识别 |

## 4. 上下行双线独立运行

### 4.1 问题

M9 上下行里程在 0→16489m 完全重叠——两套区段共享同一个 1D 坐标空间。如果不加轨道过滤，会出现：

- UP 列车被 DOWN 轨上的占用截断 MA
- DOWN 列车的预留溢到 UP 轨，前端一大片红色
- 站台查找在上下行同公里标位置误判方向

### 4.2 同轨前向邻居图 (kmForwardMap)

`TrackService.kmForwardMap()`：基于公里标邻接（`next.start ≈ this.end`）且同轨道（`next.track == this.track`），不依赖 `from→to` 节点方向。

```java
// TrackService.java
public synchronized Map<String, List<String>> kmForwardMap() {
    for (TrackSegmentState seg : segments) {
        String segTrack = seg.track() != null ? seg.track() : "main";
        List<String> next = segments.stream()
            .filter(o -> Math.abs(o.startMeters() - seg.endMeters()) < 0.1)
            .filter(o -> segTrack.equals(o.track()))
            .map(TrackSegmentState::id).toList();
        map.put(seg.id(), next);
    }
}
```

**作用**：`resolveTopologyObstacle` 和 `collectReservedAlongActiveTopology` 统一使用 `kmForwardMap`，上下行各自沿自己的轨道前向搜索。

### 4.3 轨道显式绑定

发车时根据方向显式绑定列车到对应轨道：

```java
// SimulationRuntime.java - handleDepartures()
String track = ExternalTrainDirection.DOWN.equals(direction) ? "down" : "up";
trackService.assignTrainToTrack(trainId, offsetMeters, track);
```

```java
// TrackService.java
public synchronized void assignTrainToTrack(String trainId, double positionMeters, String track) {
    segments.stream()
        .filter(s -> positionMeters >= s.startMeters() && positionMeters < s.endMeters())
        .filter(s -> track.equals(s.track()))
        .findFirst()
        .ifPresent(s -> assignTrainToSegment(trainId, s.id()));
}
```

### 4.4 异轨 MA 互不阻挡

`SignalService.calculateAuthorities()` 中，线性排序的后车如果是异轨列车，不构成前后阻挡关系：

```java
TrackSegmentState selfSeg = trackService.segmentForTrain(train);
TrackSegmentState nextSeg = trackService.segmentForTrain(nextTrain);
boolean differentTrack = selfSeg != null && nextSeg != null
    && selfSeg.track() != null && nextSeg.track() != null
    && !selfSeg.track().equals(nextSeg.track());
if (!differentTrack) {
    // 同轨才检查前车阻挡
}
```

### 4.5 跨轨障碍搜索即停

`resolveTopologyObstacle` 中，沿 `kmForwardMap` 搜索时，遇到不同轨道立即停止：

```java
if (!curTrack.equals(selfTrack) && !"main".equals(curTrack)) {
    return Double.POSITIVE_INFINITY;
}
```

### 4.6 预留段严格同轨过滤

`collectReservedAlongActiveTopology` 中，渡线/异轨即停：

```java
if (!nextTrack.equals(track)) {
    break; // 不溢到对向轨道
}
```

## 5. 调度→信号→车辆发车闭环

### 5.1 问题

调度计划（`dispatch-plan.yaml`）定义了 4 条运行线：SVC-001/002(UP) 和 SVC-003/004(DOWN)。但之前存在三个阻塞点：

1. **TR-001/002 硬编码创建**：`TrainManager.reset()` 中调用 `addInitialTrain()`，绕过调度计划
2. **DOWN 门禁**：`handleDepartures` 中检查 `toPos < fromPos` → 跳过下行发车（引擎只支持递增里程）
3. **下行门禁条件**：`offsetMeters > lineLen * 0.5` → 下行车 km=0 不满足 → 不绑定轨道 → 随机跳到 up 轨

### 5.2 修复

| 阻塞点 | 位置 | 修复 |
|--------|------|------|
| 初始车 | `TrainManager.reset()` | 删除 `addInitialTrain` 调用，全部由调度发车 |
| DOWN 门禁 | `SimulationRuntime.handleDepartures()` | 删除 `toPos < fromPos` 检查 |
| 轨道绑定 | `SimulationRuntime.handleDepartures()` | 无条件 `assignTrainToTrack`（UP→up, DOWN→down）|

### 5.3 发车流程

```
点击 Reset → dispatchService.reset(simStart=now)
→ Clock 推动 tick → evaluate()（首帧 30s 窗口内）
→ automaticDepartureCommands:
    SVC-001 (offset=0s)  → DEPART TR-001 (UP, km=0)
    SVC-003 (offset=0s)  → DEPART TR-003 (DOWN, km=0)
    SVC-002 (offset=300s)→ DEPART TR-002 (UP, 5min后)
    SVC-004 (offset=300s)→ DEPART TR-004 (DOWN, 5min后)
→ handleDepartures → addTrain → assignTrainToTrack
→ touchRoutes → R_UP / R_DOWN → MA 计算 → 车辆运行
```

## 6. 占用/预留/联锁联动

### 6.1 区段状态机

```
FREE(灰) ──MA预留──→ RESERVED(黄) ──车到达──→ OCCUPIED(红) ──车离开──→ FREE
                                                    ↑
                                              故障注入 → FAULT(橙)
```

优先级：FAULT > OCCUPIED > RESERVED > FREE

### 6.2 overlaps 修复

原实现 `head > segStart && tail < segEnd`（严格包含），导致车尾刚离开段就被判定为还在重叠。

修复为正确的区间交集：`tail < segEnd && head > segStart`。

### 6.3 终点站释放

列车到达终点站（首站或末站 ±120m）且零速后，不再标记占用：

```java
boolean atTerminalStation = hasStations && train.zeroSpeed()
    && (Math.abs(train.positionMeters() - firstStationPos) <= 120
        || Math.abs(train.positionMeters() - lastStationPos) <= 120);
if (atTerminalStation) continue; // 不标 OCCUPIED
```

### 6.4 MAIN 进路多车追踪 hold 解除

`touchRoutes` 中新增：被 hold 的列车若已在已建立的 MAIN 进路(OCCUPIED)上，自动清除 hold：

```java
if (route.status() == RouteStatus.OCCUPIED
    && isMainTypeRoute(route.routeId())
    && resolvedSegmentIds(route).contains(trainSeg.id())) {
    clearRouteHold(train.id());
}
```

### 6.5 进路站台制动保护

`resolveStationDwell` 中：仅对高速接近站台的列车（v > 2.0m/s）做长距离制动保护，慢速/零速正常进站。避免 M9 长段中过早截断 MA 导致列车刹车停在站台窗外。

## 7. 三层命令反馈闭环

### 7.1 架构

```
调度命令 CMD
  → 信号接收 accepted/rejected    ← POST /api/signal-track/command-receipt
  → 信号应用 applied/not applied  ← generateTimeCommandFeedback (每 tick)
  → MA/限速/进路约束说明          ← checkDispatchCompletion
  → 运行状态变化
  → 间隔改善确认
  → 调度侧关闭扰动/确认效果       ← POST /api/dispatch/command-feedback
```

### 7.2 新增端点

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/signal-track/command-receipt` | POST | 信号侧命令接收回执 |
| `/api/dispatch/command-feedback` | POST | 信号/运行控制统一反馈 |

### 7.3 自动反馈生成

`SimulationRuntime.generateTimeCommandFeedback()` 每 tick 自动为每个有 `sourceCommandIds` 的 `DispatchConstraint` 生成反馈，包含：

```json
{
  "accepted": true,
  "effectiveSpeedMps": 7.8,
  "movementAuthorityEndMeters": 1280.0,
  "maDistanceMeters": 450.0,
  "speedFactor": 0.65,
  "speedLimitMps": 8.0,
  "currentSegmentId": "U20a",
  "endSegmentId": "U20b",
  "constraintSource": "FRONT_TRAIN",
  "constraintReason": "speedFactor=0.65",
  "limitBreakdown": {
    "finalSpeedMps": 8.0,
    "dispatchFactor": 0.65
  }
}
```

约束来源自动识别：

| 来源 | 判定条件 |
|------|---------|
| STATION_DWELL | holdTrain + MA原因含"站台" |
| ROUTE_CONFLICT | reasonCode含ROUTE |
| FRONT_TRAIN | MA原因含"前车" |
| MA_LIMITED | reasonCode=MA_LIMITED |
| TRACK_FAULT | MA原因含"故障" |
| SIGNAL_MA | 默认其他 |

### 7.4 前端消费

`GET /api/dispatch/commands` 轮询，每条命令的 `status` 从 `PENDING` → `APPLIED` → `EFFECT_CONFIRMED`。前端调试面板 (`DispatchLoopDebugView.vue`) 逐条展示命令的约束来源、有效限速、MA 距离等详情。
