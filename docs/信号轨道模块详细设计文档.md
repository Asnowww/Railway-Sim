## 1. 模块概述

信号轨道模块是铁路仿真系统的**核心运行时引擎**，负责将原始线路拓扑数据转化为列车可运行的安全授权环境。它的职责链条可以概括为：

```
线路数据 → 轨道拓扑 → 区段状态 → 进路联锁 → 移动授权 → 车辆运行
   ↑                                                          ↓
   └────────── 调度命令接收 / 反馈回推 ────────────────────────┘
```

本模块直接操作的数据结构包括：

| 数据结构 | 职责 |
|----------|------|
| `TrackSegmentState` | 区段占用/预留/故障状态 |
| `RouteState` | 进路六态生命周期 |
| `MovementAuthority` | 每列车的安全移动授权 |
| `DispatchCommandFeedback` | 调度命令的回执与效果反馈 |

上游依赖调度模块的 `DispatchService`（命令生成）和 `DispatchPlanFile`（时刻表）。下游驱动 `TrainManager`（车辆实体）和 `VehicleRuntimeIntegrationService`（9300 物理引擎）。

### 2.3 Tick 循环中的执行顺序

```
1. updateOccupancy(trains)           ← 清除上一 tick 的占用
2. constraintsForTrains(trains)      ← 计算轨道约束(段限速)
3. calculateAuthorities(trains, ...) ← 核心：MA + 预留 + 信号
   ├─ touchRoutes(trains)           ← 自动建进路/hold解除
   ├─ resolveStationDwell(train)    ← 站台控制
   ├─ resolveTopologyObstacle(train)← 移动闭塞障碍搜索
   └─ collectReservedAlongActiveTopology ← 预留段标记
4. dispatchService.evaluate(...)     ← 调度生成新命令
5. interlockingService.applyCommand  ← 消费进路命令
6. handleDepartures(...)            ← 消费发车命令
7. dispatchService.constraintsForTrains ← 调度约束(speedBias)
8. calculateAuthorities(trains, ..., dispatchConstraints)
   ← 第二遍信号：含调度约束的 MA
9. tickAll → 9300 物理步进
10. checkDispatchCompletion / generateTimeCommandFeedback
    ← 信号→调度反馈
11. buildSnapshot → WebSocket 推送
```

## 4. 区段切分与拓扑构建

### 4.1 问题

原始 M9 数据中，信号机之间的闭塞区间长度不均匀。站场端（郭公庄、国家图书馆附近）信号机密集——200m 左右一段；正线区间信号机稀疏——最大间距 5.5km，一个段跨越 3-4 个车站，前端等比例里程渲染下一条红色块盖过多个站。

### 4.2 切分方案

将 8 个长段（上下行各 4 个）按站台中心公里标切分为子段：

| 原段 | 原长度 | 切分后 | 子段数 | 典型段长 |
|------|--------|--------|--------|---------|
| U20 | 4635m | U20a/b/c/d | 4 | 788-1655m |
| U23 | 3205m | U23a/b/c | 3 | 153-1794m |
| U34 | 5487m | U34a/b/c/d/e | 5 | 152-1910m |
| U42 | 1808m | U42a/b/c | 3 | 102-1093m |
| D17 | 4763m | D17a/b/c/d | 4 | 788-1787m |
| D21 | 3071m | D21a/b/c | 3 | 167-1779m |
| D28 | 5352m | D28a/b/c/d/e | 5 | 69-1912m |
| D36 | 2164m | D36a/b/c | 3 | 93-1096m |

**结果**：上行 13→24 段，下行 12→23 段。新增 22 个站间节点（ND/NU 系列）。信号 `segment_id` 自动映射到对应子段，进路 `axle_section_ids` 同步展开。

### 4.3 拓扑邻居关系

`YamlLineDataLoader` 在加载 YAML 后自动计算两条邻居图：

| 邻居类型 | 计算规则 | 用途 |
|----------|---------|------|
| `forwardNeighbor` | `other.from == this.to` | 路径搜索、进路连通性校验 |
| `sideNeighbor` | `other.from == this.from && other.to != this.to` | 道岔分叉识别 |

生成的 `OperationalLineData.TrackSegmentDefinition` 包含 `forwardNeighborSegmentIds` 和 `sideNeighborSegmentIds` 两个列表，供运行时 `TrackService.forwardNeighborMap()` 直接查询。

## 5. 移动闭塞 (Moving Block) 算法

### 5.1 核心算法

`SignalService.calculateAuthorities()` 是移动闭塞计算的总入口。对每列车依次处理：

```
列车按公里标排序
  → 取线性前车间距限制 nextTrainTailLimit
  → 沿拓扑前向搜索 resolveTopologyObstacle
  → 取进路联锁限制 interlockingLimit
  → 取故障区段限制 faultLimit
  → 取站台停靠限制 stationLimit
  → authorityEnd = min(以上全部)
  → 安全制动限速 = sqrt(2 * decel * maDistance)
  → 预留段标记
```

### 5.2 resolveTopologyObstacle — 拓扑障碍搜索

沿区段的前向邻居链逐段搜索，遇到三种情况立即返回障碍：

1. **占用段（OCCUPIED）且不是自己所在段** → 返回 `seg.startMeters - safetyGap`
2. **故障段（FAULT）** → 返回 `seg.startMeters - safetyGap`
3. **异轨段（不同 track 标签）** → 返回 `Double.POSITIVE_INFINITY`（即无限制）

使用 `kmForwardMap` 而非 `forwardNeighborMap` 做前向搜索——前者基于公里标邻接，上下行通用。

### 5.3 关键修复

| 修复 | 位置 | 说明 |
|------|------|------|
| overlaps 区间交集 | `TrackService.overlaps()` | `tail < segEnd && head > segStart`（原是严格包含，导致走过不释放） |
| 异轨障碍即停 | `resolveTopologyObstacle()` | 搜索到 `crossover`/异轨段立即返回 INFINITY |
| 同轨预留不过界 | `collectReservedAlongActiveTopology()` | 渡线即停，预留不溢到对向轨道 |
| MA 互不阻挡 | `calculateAuthorities()` | 线性排序时检查前后车是否同轨，异轨不算前车 |

### 5.4 安全制动距离

```
safeBrakingSpeed = sqrt(2 * DEFAULT_BRAKING_DECELERATION * maDistance)
DEFAULT_BRAKING_DECELERATION = 0.8 m/s²
```

在站台进站保护中额外检查：仅当列车速度 > 2.0m/s 且距离站台超过安全制动距离时才不做 MA 截断（低速进站的直接截到站台位置）。

## 6. 上下行双线独立运行

### 6.1 问题域

M9 线路的上下行段在 0→16489m 公里标空间上完全重叠——同一位标存在两条区段（如 U02 和 D03 都在 0→215m）。如果不做轨道隔离，会出现：

- UP 列车被 DOWN 轨上的占用截断 MA
- DOWN 列车的预留溢到 UP 轨，前端一大片红色
- 占用的进路匹配到错误轨道，列车被 hold 锁死

### 6.2 同轨前向邻居图 (kmForwardMap)

`TrackService.kmForwardMap()` 不同于 `forwardNeighborMap`：

| | forwardNeighborMap | kmForwardMap |
|---|-------------------|-------------|
| 计算依据 | YAML `from→to` 节点方向 | 公里标数值邻接 |
| 对下行段 | 反向（D03→ND01←ND00 链断裂） | 正向（按 km 递增） |
| 轨道过滤 | 无 | `sameTrack` 过滤 |
| 适用场景 | 进路验证 | MA 搜索 + 预留扩散 |

```java
// TrackService.kmForwardMap() 核心逻辑
for (TrackSegmentState seg : segments) {
    String segTrack = seg.track();
    List<String> next = segments.stream()
        .filter(o -> Math.abs(o.startMeters() - seg.endMeters()) < 0.1)
        .filter(o -> segTrack.equals(o.track()))
        .map(TrackSegmentState::id).toList();
    map.put(seg.id(), next);
}
```

### 6.3 轨道绑定策略

**列车创建时**（`handleDepartures`）根据方向显式绑定到对应轨道，避免 `segmentForTrain` 回退到错误轨道：

```java
String track = "DOWN".equals(direction) ? "down" : "up";
trackService.assignTrainToTrack(trainId, offsetMeters, track);
```

**段回退时**（`segmentForTrain` 的 `resolveNextSegment`）候选段按轨道偏好排序：

```
优先级：prevTrack(列车历史轨道) > main > up > down > other
```

### 6.4 四层轨道过滤叠加

| 层次 | 位置 | 规则 |
|------|------|------|
| 段定位 | `segmentAt()` | main > up > down |
| 段绑定回退 | `segmentForTrain()` fallback | prevTrack > main > up > down |
| MA 障碍搜索 | `resolveTopologyObstacle()` | 异轨 → INFINITY |
| 预留扩散 | `collectReservedAlongActiveTopology()` | 异轨 → break |

## 7. 进路联锁状态机

### 7.1 六态模型

```
AVAILABLE ──establishRoute──→ VALIDATING
                                   │
                    ┌──────────────┤
                    ▼              ▼
               LOCKED          REJECTED (冲突/不可用)
                    │
                    ▼
               OCCUPIED
                    │
                    ▼
               RELEASING
                    │
                    ▼
               ──→ AVAILABLE (cycle)
```

### 7.2 建立进路的检查顺序

1. **状态检查**：进路必须在 AVAILABLE 或 RELEASING（可完成释放）状态
2. **共享检查**：如进路已被占用且 `trainId` 匹配 → 返回成功（无需重建）
3. **MAIN 共享**：正线进路 OCCUPIED 时允许多车追踪，后车无需重建
4. **区段检查**：依次检查每个子段——段存在、无 FAULT、无其他车 OCCUPIED、无无主占用
5. **道岔检查**：不要求同一道岔同时 NORMAL 和 REVERSE；未被锁闭
6. **冲突检查**：`intersects()` 判定与已建立进路是否有共享区段
7. **扳道岔**：按 `switchRequirements` 逐一扳动
8. **锁闭**：用 `withLocked` 转入 LOCKED 状态

### 7.3 MAIN 进路多车追踪 + hold 解除

`touchRoutes` 在每个 tick 中执行：

1. 清理到期的 CONFLICTED → AVAILABLE
2. 前车优先建路（按公里标倒序）
3. **hold 自动解除**：列车如果站在 MAIN 进路(OCCUPIED)上，但被 `routeHold` 锁着 → 清除 hold

```java
if (route.status() == RouteStatus.OCCUPIED
    && isMainTypeRoute(route.routeId())
    && resolvedSegmentIds(route).contains(trainSeg.id())) {
    clearRouteHold(train.id());
}
```

### 7.4 进路冲突诊断反馈

`RouteDispatchResult` 现在包含冲突详情：

```java
public record RouteDispatchResult(
    boolean accepted, String rejectReason,
    InterlockingFailureCode failureCode, boolean retryable,
    String conflictRouteId, List<String> conflictSegmentIds
)
```

调度侧通过 `conflictRouteId` 和 `conflictSegmentIds` 可以精准定位冲突来源，用于重试决策或换路建议。

### 7.5 失败码枚举

`InterlockingFailureCode` 定义了 13 种标准失败码，每种标记了是否可重试：

| 码 | 含义 | 可重试 |
|----|------|--------|
| ROUTE_ESTABLISHED | 成功 | - |
| ROUTE_NOT_FOUND | 进路不存在 | 否 |
| TRACK_OCCUPIED | 区段被占用 | 是 |
| TRACK_FAULT | 区段故障 | 是 |
| ROUTE_CONFLICT | 敌对进路 | 是 |
| SWITCH_LOCKED | 道岔锁闭 | 是 |
| SWITCH_MOVE_FAILED | 道岔移动失败 | 是 |
| ROUTE_NOT_AVAILABLE | 进路不在可用状态 | 是 |
| SWITCH_POSITION_MISMATCH | 道岔矛盾 | 否 |
| TRACK_NOT_FOUND | 区段不存在 | 否 |
| UNSUPPORTED_COMMAND | 不支持的命令 | 否 |
| NO_MATCHING_ROUTE | 无匹配进路 | 否 |
| INTERNAL_ERROR | 内部错误 | 否 |

## 8. 调度→信号→车辆发车闭环

### 8.1 发车流程

```
dispatch-plan.yaml
  → DispatchService.evaluate() 每 tick 检查
    → automaticDepartureCommands 生成 DEPART
      → handleDepartures 解析 trainNo/linkId/offset/direction
        → trainManager.addTrain() 创建 TrainEntity
          → assignTrainToTrack 绑定轨道 (up/down)
            → touchRoutes 自动建进路 (R_UP/R_DOWN)
              → calculateAuthorities 计算 MA
                → 9300 物理步进 → 车辆运行
```

## 9. 调度命令三层反馈闭环

### 9.1 三层模型

```
调度下发命令 (PENDING)
  └─→ 信号侧接收回执 (ACCEPTED/REJECTED)  ← POST /api/signal-track/command-receipt
       └─→ 应用反馈 (APPLIED)              ← generateTimeCommandFeedback (每 tick)
            └─→ 效果确认 (EFFECT_CONFIRMED/TIMEOUT)  ← checkDispatchCompletion
```

### 9.2 新增端点

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/signal-track/command-receipt` | POST | 信号侧命令接收回执，支持批量 |
| `/api/dispatch/command-feedback` | POST | 信号→调度统一反馈接口 |
| `/api/signal-track/dispatch-publications` | GET | 信号侧已缓存的调度计划发布记录 |
| `/api/signal-track/dispatch-publications/latest` | GET | 最近一次发布记录 |
| `/api/signal-track/route-events` | GET | 当前 tick 的进路生命周期事件 |

### 9.3 自动反馈生成

`SimulationRuntime.generateTimeCommandFeedback()` 每 tick 执行一次：

1. 遍历所有 `DispatchConstraint`，找有 `sourceCommandIds` 的条目
2. 对每个命令取对应列车的 MA 信息
3. 生成反馈详情：有效限速、MA 距离、约束来源、限速分解
4. 调用 `dispatchService.acceptFeedback()` 推送

约束来源自动推断：

```java
private String deriveConstraintSource(MovementAuthority ma, DispatchConstraint c) {
    if (c.holdTrain()) {
        if (ma.reason().contains("站台")) return "STATION_DWELL";
        if (ma.reason().contains("进路")) return "ROUTE_CONFLICT";
        return "HOLD";
    }
    if (ma.reasonCode().equals("MA_LIMITED")) return "MA_LIMITED";
    if (ma.reason().contains("前车")) return "FRONT_TRAIN";
    if (ma.reason().contains("站台")) return "STATION_DWELL";
    return "SIGNAL_MA";
}
```

### 9.4 Publication Ack

`SignalDispatchPlanRegistry.acceptAndValidate()` 在调度发布计划时逐条校验每个 entry 的 routeId：

```java
for (Entry entry : publication.entries()) {
    var state = interlockingService.state(entry.routeId());
    if (state == null) rejection = "ROUTE_NOT_FOUND";
    else if (state.holdsInterlockingResources()
        && !entry.trainId().equals(state.establishedByTrainId()))
        rejection = "ROUTE_CONFLICT:" + state.name();
    // ...
}
```

返回带 per-entry `ACCEPTED/REJECTED` 状态和 `acceptedCount/rejectedCount` 的快照。

## 10. 占用管理与区段状态机

### 10.1 区段状态机

```
FREE(灰色) ──MA预留──→ RESERVED(黄色) ──车到达──→ OCCUPIED(红色)
     ▲                                                  │
     └──────────── 车离开 ──────────────────────────────┘

                                                故障注入
                                              OCCUPIED → FAULT(橙色)
```

优先级：FAULT > OCCUPIED > RESERVED > FREE

### 10.2 updateOccupancy 算法

每 tick 执行：

1. **清除上 tick 的 OCCUPIED**：`OCCUPIED → FREE`，但如果同时被预留则 `→ RESERVED`
2. **逐车标占用**：对每列车，取 `segmentForTrain` 绑定段 + 前历史段，判定车身区间和段区间是否重叠（`overlaps`），重叠则标 `OCCUPIED`
3. **终点站释放**：终点站已停列车不再标记占用

### 10.3 终点站释放

有两个释放触发条件（满足任一即释放）：

1. **有站数据时**：列车零速且与首站或末站距离 ≤ 120m
2. **无站数据时**（测试/演示线路）：列车零速且位置 ≥ lineEnd - 10m

```java
boolean atTerminalStation = hasStations && train.zeroSpeed()
    && (Math.abs(train.positionMeters() - firstStationPos) <= 120
        || Math.abs(train.positionMeters() - lastStationPos) <= 120);
if (atTerminalStation) continue; // 不标 OCCUPIED
```

## 11. 站台停靠控制

### 11.1 resolveStationDwell 算法

对每列车，根据位置找"前方最近站"，按以下逻辑返回 `maEndAt`：

```
1. 找前方最近站（UP→前向 min，DOWN→镜像 max）
2. 若调度已释放该站 → 不截 MA
3. 若列车在站台窗口内且零速 → 累计 dwell ticks
4. 若 dwell 未完成 → MA 截到站台位置
5. 若 dwell 完成 → 释放 MA，清除计数
```

站停秒数默认为 `DEFAULT_DWELL_SECONDS = 5s`，可由调度 `deltaDwellSec` 动态调整。

### 11.2 DOWN 站台查找

下行列车使用镜像公里标查找站台，确保站序从 S113→S101 逆序匹配：

```java
if (isDown && lineLen > 0) {
    double effectiveHead = lineLen - head;
    nextStation = stations.stream()
        .filter(s -> s.centerMeters() <= effectiveHead + WINDOW)
        .max(Comparator.comparingDouble(s -> s.centerMeters()))
        .orElse(null);
}
```

### 11.3 进站制动保护

仅对高速接近站台的列车做长距离保护——防止 M9 长段中过早截断 MA 导致列车刹车停在站窗外：

```java
if (distanceToStation > 0 && train.speedMetersPerSecond() > 2.0) {
    double safeBraking = (speed²) / (2 * 0.8) + 60.0;
    if (distanceToStation > safeBraking) return result; // 安全，不截
}
```

慢速/零速的列车直接允许进站截断。
