# 7. 供电控制模块详细设计

## 7.1 模块概述

供电控制模块负责在中心仿真主循环内维护牵引供电分区状态，向车辆控制模块提供可用电压、可用功率、受流可用性、再生制动吸收能力和故障约束。模块以接触轨分区等效模型为当前实现口径，不承担真实 PSCADA 全点表解析、不直接下发运营调度命令，也不直接控制列车牵引。

模块当前落点：

| 职责 | 代码/配置 |
|---|---|
| 静态供电拓扑和阈值 | `config/power_third_rail.yaml`、`OperationalPowerData`、`PowerConfigLoader` |
| 实时供电状态 | `PowerSectionState` |
| 车辆约束输出 | `PowerConstraint` |
| 供电计算和状态机 | `PowerService` |
| 外部供电仿真接入 | `PowerIntegrationService`、`power-network-service:9200` |
| REST 查询和故障注入 | `PowerController` |
| 快照缓存和告警事件 | `RealtimeStateCache`、`SimpleEventBus`、`MonitorService` |

设计边界：

1. 供电模块只输出约束和状态，不越级控制调度扣车、折返、清客。
2. 车辆牵引切除、制动或惰行由 `OnboardTrainSubsystemManager` 调用本车基层智能子系统后根据 `PowerConstraint` 决策。
3. 中央供电控制模块仍是前端和其他中央模块的唯一入口；外部供电仿真只作为设备状态、电压遥测和风险指标来源。
4. 仅 `EXTERNAL_HTTP` 权威外部车辆运行时健康时可把车辆取电负荷推送到 `power-network-service:9200`，中央 `PowerIntegrationService` 只拉取供电状态，避免同一 tick 重复注入负荷。

## 7.2 功能设计

### 7.2.1 供电分区状态监测

模块按配置文件加载牵引变电所、馈线、接触轨分区、断路器、供电方式和检修状态，运行时为每个分区维护：

- 接触轨电压 `voltage`
- 净牵引电流 `current`
- 牵引负荷 `loadWatts`
- 再生制动功率 `regenPowerWatts`
- 已吸收再生功率 `absorbedRegenPowerWatts`
- 未吸收再生功率 `unabsorbedRegenPowerWatts`
- 可用牵引功率 `availablePowerWatts`
- 断路器、保护、检修闭锁和数据质量状态

### 7.2.2 车辆供电约束输出

`constraintsForTrains()` 根据列车位置匹配供电分区，并输出 `PowerConstraint`：

| 字段 | 含义 | 车辆侧用途 |
|---|---|---|
| `sectionId` | 列车所在供电分区 | 状态诊断、影响范围 |
| `railVoltage` | 接触轨电压 | 低压/失电判断 |
| `powerAvailableWatts` | 可用牵引功率 | 限制牵引输出 |
| `energized` | 分区是否可受流 | 切除牵引 |
| `powerDeratingFactor` | 降额比例 | 供电降额牵引 |
| `currentCollectionAvailable` | 受流可用 | TCMS 受流状态 |
| `regenAvailable` | 再生吸收可用 | 再生制动策略 |
| `constraintReason` | 约束原因 | 状态机诊断 |

### 7.2.3 车辆负荷回写

车辆物理输出完成后，`updateFromVehicleOutputs()` 按分区聚合车辆电流、牵引功率、再生功率和受影响列车列表，重新计算分区电压、可用功率和保护状态，并更新实时缓存。

非拆分部署的`LOCAL`、`DUAL_SHADOW`模式下，中央可把聚合后的`sectionLoads`通过`/power-network/state/query`发送给外部供电仿真；拆分部署固定由`vehicle-runtime-service`通过`/power-network/step`写入，中央只调用`/power-network/state`读取结果。写入权按部署配置确定，不因单车FMU降级而动态切换，因此“列车状态变化 -> 牵引/制动功率变化 -> 电网负荷变化 -> 供电参数波动”始终只有一个负荷写入方。

### 7.2.4 故障注入和检修闭锁

仿真演示和联调阶段支持以下内部/REST 操作：

| 操作 | 行为 |
|---|---|
| `injectPowerFault(sectionId, UNDERVOLTAGE)` | 分区进入欠压，车辆牵引降额。 |
| `injectPowerFault(sectionId, DEENERGIZED)` | 分区失电，车辆受流不可用。 |
| `injectPowerFault(sectionId, BREAKER_TRIP)` | 断路器跳闸，分区退出供电。 |
| `injectPowerFault(sectionId, OVERCURRENT)` | 分区过流，保护状态为过流。 |
| `setMaintenanceLock(sectionId, LOCKED_OUT/GROUNDED)` | 设置检修闭锁或接地，禁止恢复送电。 |
| `clearPowerFault(sectionId)` | 清除注入故障，但不绕过检修闭锁状态。 |

## 7.3 软件结构设计

```text
StaticInfrastructureCatalog
  -> OperationalPowerData
  -> PowerService
       -> constraintsForTrains()
       -> updateFromVehicleOutputs()
       -> injectPowerFault()/clearPowerFault()
       -> setMaintenanceLock()/setSupplyMode()
       -> PowerIntegrationService
            -> vehicle-runtime healthy: GET power-network/state
            -> fallback/local: POST power-network/state/query
  -> RealtimeStateCache
  -> SimpleEventBus
  -> PowerController / MonitorService
```

核心类职责：

| 类/组件 | 职责 |
|---|---|
| `OperationalPowerData` | 静态供电配置模型，包含额定电压、最低电压、最大电流、过流阈值、再生吸收策略和分区定义。 |
| `PowerSectionState` | 分区实时状态 record，是 REST/WebSocket/监控的统一数据源。 |
| `PowerConstraint` | 供电到车辆的最小约束对象，不携带调度动作。 |
| `PowerService` | 供电状态机、负荷聚合、故障注入、事件发布。 |
| `PowerIntegrationService` | 与外部供电仿真同步设备状态；根据车辆运行时是否为 `EXTERNAL_HTTP` 权威写入方决定拉取状态还是主动提交分区负荷。 |
| `PowerSectionEvent` | 供电故障、恢复和检修操作记录。 |
| `PowerController` | 查询供电分区、能耗、事件和仿真故障注入接口。 |

## 7.4 业务流程设计

### 7.4.1 正常仿真流程

```text
列车状态
  -> PowerService.constraintsForTrains()
  -> 车辆运行时 LOCAL 或 vehicle-runtime-service:9300
  -> PowerService.updateFromVehicleOutputs()
  -> PowerIntegrationService
       -> EXTERNAL_HTTP权威车辆运行时健康: GET power-network/state
       -> fallback/local: POST power-network/state/query
  -> RealtimeStateCache.updatePowerSections()
  -> MonitorService 生成告警和快照
```

### 7.4.2 供电状态转换

```text
ENERGIZED
  -> UNDERVOLTAGE        电压低于 minimumVoltage 或注入欠压
  -> OVERCURRENT         净电流超过 overCurrentThreshold 或注入过流
  -> TRIPPED             断路器跳闸或注入 BREAKER_TRIP
  -> DEENERGIZED         电压低于 cutoffVoltage、断路器 OPEN 或注入失电
  -> MAINTENANCE_LOCKED  检修闭锁或接地

UNDERVOLTAGE/OVERCURRENT
  -> ENERGIZED           故障清除且电压/电流恢复
  -> DEENERGIZED/TRIPPED 保护升级
```

### 7.4.3 故障注入流程

```text
REST 故障注入
  -> 参数和 confirmToken 校验
  -> PowerService.injectPowerFault()
  -> 发布 PowerFaultStateChangedEvent
  -> 写入 PowerSectionEvent 和操作日志
  -> refreshSectionState()
  -> 下一 tick 通过 PowerConstraint 影响车辆
```

## 7.5 数据设计

### 7.5.1 静态配置

`OperationalPowerData` 字段：

| 字段 | 单位/枚举 | 说明 |
|---|---|---|
| `nominalVoltage` | V | 标称接触轨电压。 |
| `minimumVoltage` | V | 欠压阈值。 |
| `cutoffVoltage` | V | 失电/受流不可用阈值。 |
| `maxTractionCurrentAmps` | A | 分区最大牵引电流。 |
| `overCurrentThresholdAmps` | A | 过流保护阈值。 |
| `currentToVoltageDrop` | V/A | 简化压降系数。 |
| `sameSectionAbsorbFirst` | boolean | 再生制动是否优先同分区吸收。 |
| `unabsorbedRegenMode` | enum | 未吸收再生能量处理方式。 |
| `sections[]` | list | 分区定义。 |

`PowerSectionDefinition` 字段包括 `id`、`name`、`substationId`、`feederId`、`startMeters`、`endMeters`、`substationVoltage`、`breakerStatus`、`supplyMode`、`maintenanceState`、`lockoutState`、`resistanceOhmPerMeter`。

### 7.5.2 实时状态

`PowerSectionState` 是分区状态权威对象：

| 字段 | 数据类型 | 说明 |
|---|---|---|
| `id/name` | string | 分区标识和名称。 |
| `substationId/feederId` | string | 牵引变电所和馈线。 |
| `startMeters/endMeters` | number | 分区里程范围。 |
| `voltage/current` | number | 当前电压和净电流。 |
| `status` | string | `ENERGIZED`、`UNDERVOLTAGE`、`OVERCURRENT`、`TRIPPED`、`DEENERGIZED`、`MAINTENANCE_LOCKED`。 |
| `availablePowerWatts` | number | 车辆可用牵引功率。 |
| `supplyMode` | string | `DOUBLE_END`、`SINGLE_END`、`CROSS_FEED`、`OUTAGE`。 |
| `isolatorStatus` | string | 分区边界隔离开关汇总状态。 |
| `substationAvailability` | string | 关联变电所可用性。 |
| `breakerStatus` | string | `CLOSED`、`OPEN`、`TRIPPED`。 |
| `protectionState` | string | `NORMAL`、`UNDERVOLTAGE`、`OVERCURRENT`、`TRIPPED`。 |
| `maintenanceState/lockoutState` | string | 检修和闭锁状态。 |
| `externalDataQuality` | string | 外部供电仿真状态质量，失联降级时为 `FALLBACK`。 |
| `externalVoltage/externalCurrent/externalLoadWatts` | number | 外部供电仿真按区段负荷回算的电压、电流和负荷。 |
| `voltageDeviation/voltageDeviationPercent` | number | 外部电压相对中央本地电压的偏差。 |
| `voltageComparisonStatus` | string | `NO_EXTERNAL_DATA`、`MATCHED`、`DEVIATED`、`DIVERGED`。 |
| `externalSupportReason` | string | 外部仿真对供电支撑方式或降级原因的说明。 |
| `strayCurrentRiskLevel/strayCurrentRiskReason` | string | 杂散电流风险等级和原因。 |
| `affectedTrainIds` | list | 当前受该分区影响的列车。 |
| `dataQuality` | string | `GOOD`、`STALE`、`FALLBACK`、`INVALID`。 |

`voltage/current/loadWatts/availablePowerWatts` 仍由中央本地约束计算生成，并继续驱动车辆牵引约束；外部 `external*` 字段用于供电控制监控、偏差判断和前端联动演示。

### 7.5.3 数据库留档

当前供电相关数据库表和留档原则：

| 表/文件 | 用途 |
|---|---|
| `power_section_snapshot` | 持久化供电分区快照。 |
| `power_operation_log` | 供电故障注入、恢复、检修闭锁操作日志。 |

## 7.6 接口设计

### 7.6.1 对车辆模块

内部接口：`PowerService.constraintsForTrains(List<TrainState>) -> List<PowerConstraint>`。

刷新节奏：随仿真 tick 调用，不通过 REST 进入热循环。

车辆侧消费规则：

| 供电状态 | 车辆控制响应 |
|---|---|
| `ENERGIZED` | 正常牵引/制动。 |
| `UNDERVOLTAGE` | 牵引降额，车辆状态显示低压。 |
| `OVERCURRENT` | 降额牵引，触发供电告警。 |
| `DEENERGIZED` | 牵引切除，受流丢失，按速度施加制动。 |
| `TRIPPED` | 牵引切除，故障等级提升。 |
| `MAINTENANCE_LOCKED` | 禁止送电，车辆不可受流。 |

### 7.6.2 对综合监控和前端

REST：

```http
GET /api/power/sections
GET /api/power/sections/{sectionId}
GET /api/power/sections/{sectionId}/events
GET /api/power/energy
GET /api/power/maintenance-locks
```

WebSocket 快照通过 `powerSections[]` 输出同一组 `PowerSectionState` 字段。

### 7.6.3 仿真控制接口

```http
POST /api/power/sections/{sectionId}/faults
POST /api/power/sections/{sectionId}/faults/clear
```

写接口仅用于仿真演示和联调，必须携带 `confirmToken=SIMULATION_CONFIRM`，并记录操作日志。

## 7.7 核心算法或控制逻辑设计

### 7.7.1 分区映射

列车位置 `positionMeters` 落入 `[startMeters, endMeters)` 的供电分区。若位置超过所有分区，使用最后一个分区作为保护性 fallback，保证仿真不中断。

### 7.7.2 负荷聚合

对每个分区聚合：

```text
I_load = sum(output.railCurrentAmps)
P_traction = sum(output.tractionPowerWatts)
P_regen = sum(output.regenPowerWatts)
affectedTrainIds = outputs in section
```

### 7.7.3 再生制动吸收

```text
P_absorbed = min(P_traction, P_regen)   when sameSectionAbsorbFirst=true
P_unabsorbed = max(0, P_regen - P_absorbed)
I_absorbed = P_absorbed / U_substation
I_net = max(0, I_load - I_absorbed)
```

### 7.7.4 电压和可用功率

```text
R_section = resistanceOhmPerMeter * sectionLength
U_section = max(0, U_substation - I_net * (currentToVoltageDrop + R_section))
P_available = U_section * maxTractionCurrentAmps * deratingFactor(status) * supplyModeFactor(supplyMode)
```

降额规则：

| 条件 | 系数 |
|---|---|
| `UNDERVOLTAGE` | `0.5` |
| `OVERCURRENT` | `0.8` |
| `SINGLE_END` | `0.65` |
| `CROSS_FEED` | `0.75` |
| `OUTAGE` 或不可受流 | `0` |

## 7.8 异常处理设计

| 异常 | 处理 |
|---|---|
| 供电配置文件缺失或解析失败 | 启动阶段失败，避免以错误拓扑运行。 |
| 分区 ID 不存在 | REST 查询/写入抛出业务异常。 |
| 失电或跳闸 | `availablePowerWatts=0`，车辆下一 tick 切除牵引。 |
| 检修闭锁未解除 | 清故障不恢复送电，必须先解除闭锁。 |
| 再生功率无法吸收 | 计入 `unabsorbedRegenPowerWatts`，按配置模式处理。 |
| 数据质量异常 | 输出 `dataQuality`，由监控层生成告警。 |

恢复策略：

1. 故障注入类异常通过 `clearPowerFault()` 清除。
2. 检修闭锁类异常通过 `setMaintenanceLock()` 恢复到 `NONE`。
3. 电压/过流类异常可在负荷下降后自动回到 `ENERGIZED`。

## 7.9 安全性设计

1. 写接口仅面向仿真模式，必须使用确认 token。
2. 所有供电故障注入、清除和检修闭锁变更写入操作日志，包含操作人、原因和 traceId。
3. 供电模块不保存真实 PSCADA 凭据，不在仓库中提交外部系统账号。
4. REST 写接口不得直接调用调度命令，避免越权控制运营策略。
5. 外部供电接口接入前必须先定义数据质量、心跳、超时和回退策略。

## 7.10 测试设计

| 测试 | 覆盖点 | 预期 |
|---|---|---|
| `PowerServiceTests` 正常负荷 | 分区映射、电压压降、可用功率 | 状态为 `ENERGIZED`，功率随负荷变化。 |
| 欠压注入 | `UNDERVOLTAGE` | 车辆约束降额，保护状态为欠压。 |
| 失电/跳闸注入 | `DEENERGIZED`、`BREAKER_TRIP` | 可用功率为 0，车辆受流不可用。 |
| 过流场景 | 电流超过阈值 | 状态为 `OVERCURRENT`，触发告警事件。 |
| 再生吸收 | 同分区牵引和制动共存 | 牵引负荷优先吸收再生功率。 |
| 检修闭锁 | `LOCKED_OUT/GROUNDED` | 清除故障不绕过闭锁。 |
| REST 故障注入 | confirmToken 和日志 | 状态立即可见，操作可审计。 |

## 7.11 开发备注

1. 当前供电模型是分区级等效模型，不做完整牵引网潮流和 PSCADA 点表仿真。
2. 供电模块对车辆只输出 `PowerConstraint`，车辆如何制动、惰行或降额由 `OnboardTrainSubsystemManager` 所调用的车辆控制节点处理。
3. 外部仿真协议中的 `PowerSystemAndTrainsV1/...` 暂按 RT-LAB 车辆/综合动力学模型命名空间处理，不作为供电遥测接口。
4. 后续新增数据库结构时，必须在 `/database` 下留档；本次文档补充未改变数据库。
