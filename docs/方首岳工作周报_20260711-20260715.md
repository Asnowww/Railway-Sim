## 四、主要完成工作

### 4.1 车辆参数和动力学模型升级

1. 将车辆参数升级为 Schema v2，统一 9300 与 9000 的参数来源、版本和哈希校验。
2. 补齐 118 m 六节编组、AW0/AW2/AW3 质量、6.5 传动比及 52 点牵引/制动电机曲线，供 Java fallback 与 Modelica FMU 共用。
3. 更新 Modelica 模型和闭环验证，使车辆牵引、制动、能耗、低压保护和曲线约束使用同一组参数语义。
4. 增加 `DEPARTING_STATION` 离站保护状态：调度/信号释放停站后，列车在离站窗口内不再被同一站台重新捕获，消除“刚起步又回到站停”的状态机问题。

### 4.2 P0/P1 控制、停车和告警能力

1. 完成 P0 阻断项收口，将 PLC 输入迁移至 9300 处理；中央仅负责转发和业务编排，不再直接修改车辆权威状态。
2. 实现停车结果采集、历史查询和运行批次关联；平稳性指标从制动开始阶段累计，避免只按最终停车帧评价。
3. 实现告警去重、确认、自动清除、再次发生及持久化，并修正清除供电故障、解除检修闭锁时误产生新告警的问题。
4. 增加车辆运行批次滚动切换、服务健康记录和恢复门槛，为停止、恢复、重启查询提供可追溯的状态基础。

### 4.3 列车状态权威迁移至 9300

1. 新增 `TrainStateHolder`，由 9300 持有每列车的位置、速度、牵引力、能耗、TCMS 诊断、车站追踪等可变状态；8080 的 `TrainEntity` 调整为镜像。
2. 9300 建立自主时钟、手动控制和本车控制队列，中央向 9300 下发 MA、轨道、调度及运行批次约束，不再传递完整列车状态。
3. 将 `EXTERNAL_HTTP` 固化为生产主模式，清理中央重复的车辆物理与控制路径；9300 不可用时中央中止当前 tick，而不再静默回退为另一套中央物理结果。
4. 8080 从当前线路 YAML/工作簿生成只读车站目标，9300 仅消费静态拓扑进行停车和到站判断，不创建或修改基础设施。

### 4.4 司机台、现场协议和前端真值接入

1. 按现场协议接通司机台与车辆运行时：PLC 原始输入由 8080 封装并转发至 9300，9300 成为命令仲裁和车辆状态变更的唯一执行端。
2. 完善 PLC 驾驶控制、司机台报文解码、网络屏/信号屏/视景输出方向及跨服务状态同步。
3. 前端改为消费后端权威状态和运行会话，修复演示数据残留与自主运行会话不稳定问题。
4. 更新车辆—信号接口、机房局域系统协议和控制详细设计，使接口字段、端口方向、失败策略与当前权威边界一致。

### 4.5 多速率 FMU 与车电闭环

1. 建立“100 ms TCMS 控制周期 + 5 次 20 ms FMU 物理子步”的多速率闭环：控制量和安全约束在 5 个子步内保持，最终状态在 100 ms 边界回写。
2. 保持 9300 为车辆控制、状态和物理的唯一权威端；9000 仅承担 FMU 实例运行；9200 仅承担分区供电约束和电网计算。
3. 统一供电默认拓扑，服务启动直接加载 1500 V、5 个供电分区基线，避免出现 750 V、2 分区的过渡拓扑。
4. 修复 FMU 零速过零和站停释放相关问题，保证零速附近的牵引/制动切换及终点距离展示与调度约束一致。

### 4.6 千车通信优化与能耗展示

1. 为 9300 增加多个 9000 FMU 服务分片的稳定选路、并发调用和实例生命周期同步门禁；删除、清空、重置、重同步与步进不会在物理子步中途释放实例。
2. 引入紧凑的中间子步协议与二进制编码，减少 1000 车多子步通信中的序列化开销。
3. 将车辆牵引、再生、受流和供电约束的能耗字段统一至前端展示，便于观察分区负荷和车电耦合结果。
4. 新增 1000 车容量测试脚本、机器可读验收报告和部署配置，建立后续扩容优化的基准。

### 4.7 7 月 14 日补充：外部仿真系统接口与逻辑适配

1. 补齐机房局域网司机台 TCP 适配：持续接收并重同步 46 字节 PLC 输入帧，按状态变化回写 26 字节 PLC 显示帧，并保留最近司机台快照供监控与显示读取。
2. 建立浏览器司机台与物理 PLC 的双控制源仲裁：非权威输入仍可作为显示镜像保留，只有当前选定的控制源才会转发至 9300，避免双源同时驱动车辆。
3. 完善外部仿真手动桥：提供外部仿真桥接配置入口，按 UDP 接收外部列车遥测，并将 ATO 或司机台控制量转换为外部列车命令；遥测再映射回中央侧列车状态和显示数据。
4. 将精确停车点从“站中心”调整为“对应股道站台末端”，向车辆侧下发带符号的停车控制距离；在 Java fallback 与 FMU 物理模式中统一执行低速蠕行、贴靠和对位归零逻辑。
5. 联动修正停站释放、零 MA 停站和终点距离展示，使外部仿真、信号 MA、车辆状态机和调度监控使用同一套位置、站台和控制语义。

### 5.2 容量验收

| 场景 | 结果 | FMU 批次 p95 | 9300 完整周期 p95 | HTTP 往返 p95 | 结论 |
| --- | --- | ---: | ---: | ---: | --- |
| 100 车 × 100 tick | 通过 | 27.000 ms | 27.000 ms | 35.636 ms | 无 fallback、无 FMI 错误、无负速度，满足当前性能门槛 |
| 1000 车 × 100 tick | 功能通过、性能未通过 | 103.000 ms | 107.000 ms | 153.989 ms | 实例隔离、完整车集与物理输出正确，但超过 60/80 ms p95 门槛 |

1000 车场景中，每个 TCMS 周期固定执行 5 个 20 ms FMU 子步；当前正式结果证明功能链路和分片策略可用，但不能宣称在本机 Docker Desktop 的 `linux/amd64` 兼容环境中达到 100 ms 硬实时。该结论按验收报告原样保留，没有以“均值低于 100 ms”替代 p95 门槛。

## 六、遗留问题与下期建议

1. **千车端到端时延未达标。** 当前 1000 车 p95 为 107 ms（9300）和 103 ms（FMU），应优先压缩 9300 回传 8080 的完整权威状态，采用版本化二进制或增量快照，而不是继续增加单进程线程数。
2. **真实 MySQL 环境验收待闭合。** 运行批次、停车结果、告警记录已具备代码与定向验证基础，仍需完成真实数据库迁移、断库 503、重启回放和查询链路验收。
3. **权威迁移需持续联调。** `EXTERNAL_HTTP` 已成为主链，后续应持续验证 8080、9300、9000、9200 的启动顺序、故障边界、列车创建/删除和 `RESET/RESYNC`，避免镜像与权威状态漂移。
4. **容量测试需独立流水线。** 保留 100 车作为日常性能基线，将 1000 车分片、协议和端到端时延验证纳入独立容量流水线与目标部署环境。
