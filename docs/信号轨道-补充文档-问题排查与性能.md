## A. 经典问题排查指南

### A.1 "车全堵住了，MA 全是前车限速"

**现象**：所有列车停在原地，Authority 的 reason 全是"前车限速"。

**根因**：`resolveTopologyObstacle` 沿 forward neighbor 链搜索时跨到了另一条轨道。M9 上下行在 0→16489m 完全重叠，搜索从 up 轨跳到了 down 轨上的列车，返回了一个极近的障碍点，MA 被截成零。

**排查**：在 `resolveTopologyObstacle` 里打日志看 `selfTrack` 和每个 `curTrack`。如果出现 `curTrack != selfTrack` 且不是 `crossover`，就是问题所在。

**修复**：加轨道过滤——`curTrack != selfTrack && !"main".equals(curTrack)` → `return INFINITY`。

### A.2 "走过的区段不释放，全线通红"

**现象**：列车开过后，走过的区段仍然 OCCUPIED。

**根因**：`overlaps()` 判断条件原来是 `head > segStart && tail < segEnd`，这是严格包含。当车尾刚离开段时，`tail` 已经大于 `segEnd`？不对——车尾指的是 `position - length`，车尾在 km 较小的方向。正确逻辑应该是"车体区间 [tail, head] 与区段 [segStart, segEnd) 有交集"。

**修复**：`tail < segEnd && head > segStart`。

### A.3 "列车被进路锁死，动不了"

**现象**：列车位置不动，MA.reason = "等待进路建立(ROUTE_NOT_LOCKED)"。

**根因**：`handleDepartures` 中申请进路被拒，调用了 `holdTrainUntilRouteEstablished`，但后续的 `touchRoutes` 从未清除这个 hold。

**修复**：在 `touchRoutes` 中加自动清除逻辑：如果列车在已建立的 MAIN 进路(OCCUPIED)上 → `clearRouteHold`。

### A.4 "下行车发不出来"

**现象**：调度计划有 SVC-003(DOWN)，但从未看到 DEPART 命令。

**根因**：`handleDepartures` 中有 `toPos < fromPos` 检查（S113=16105m → S101=372m，372 < 16105 → SKIP）。

**修复**：删除这个门禁。

### A.5 "下行车跑到上行轨道上"

**现象**：TR-003 方向标记为 DOWN，pos 在 0→16489，但 seg 显示是 U02(up 轨)。

**根因**：`assignTrainToTrack` 只在 `offsetMeters > L * 0.5` 时执行，下行车 offset=0 → 不执行 → `segmentForTrain` 回退时选择了 up 轨（因为 `main > up` 优先级在 km 0 处有 D03 和 U02，选了 U02）。

**修复**：`assignTrainToTrack` 改为无条件执行。

### A.6 "新车创建后站台查找错误"

**现象**：下行车从 km 0 出发，第一站正确应该是 S113(16105m)，但列车的 MA 被截到 S101(372m)。

**根因**：`resolveStationDwell` 用 `min(center ≥ head)` 找站，km 0 往右第一个站是 S101(372m)。

**修复**：下行列车用 `effectiveHead = L - head` 做镜像，找 `max(center ≤ effectiveHead)` → S113(16105)。

## B. 性能分析

### B.1 Tick 循环耗时分布（典型 4 车场景）

| 阶段 | 耗时 (μs) | 占比 |
|------|----------|------|
| updateOccupancy | ~15 | 2% |
| calculateAuthorities | ~200 | 27% |
| dispatchService.evaluate | ~150 | 20% |
| 进路命令处理 | ~50 | 7% |
| 9300 tickAll（物理步进） | ~200 | 27% |
| buildSnapshot + 推送 | ~100 | 14% |
| 其他（持久化/反馈/事件） | ~30 | 4% |

**瓶颈点**：`calculateAuthorities` 和 9300 RPC 约占 54%。16 车场景下 `calculateAuthorities` 会涨到 ~800μs（每车遍历全部段）。

### B.2 优化方向

1. **段查找缓存**：`findSegment()` 是 O(n) 线性扫描，可改成 HashMap
2. **批量 MA 计算**：当前是逐车遍历全部段，可改成先按轨道分组再计算
3. **9300 并行化**：多车物理步进可考虑分批并行 RPC

## C. 配置参考

### C.1 application.yml 核心参数

```yaml
railway:
  simulation:
    tick-millis: 100          # 每 tick = 100ms 仿真时间
    push-interval-millis: 1000 # 快照推送间隔
    safety-gap-meters: 120     # 移动闭塞安全距离
    block-mode: MOVING         # 闭塞模式：MOVING/FIXED
    line-data-path: ../config/line-m9.yaml
    scheduler-delay-millis: 200 # ClockScheduler 每 200ms 推 50 tick
```

### C.2 dispatch-plan.yaml 核心字段

```yaml
periods:
  - type: PEAK
    intervalSec: 180    # 高峰发车间隔
    dwellSec: 30        # 站停时长

circulations:
  - id: CIRC-001
    rollingStockId: STOCK-001  # 车辆编组

services:
  - serviceId: SVC-001
    trainId: TR-001
    direction: UP       # UP/DOWN
    stops:
      - { stationId: S101, arrivalOffsetSec: 0, departureOffsetSec: 15 }
```

## D. 常见开发工作流

### D.1 新增线路

1. 准备 Excel 或手动写 YAML
2. 确保 `stations → nodes → segments → switches → signals → routes` 六段完整
3. 修改 `application.yml` 的 `line-data-path`
4. 若有新的进路类型，检查 `YamlLineDataLoader.validateRoutes` 是否要跳过
5. `mvn clean spring-boot:run` 启动验证

### D.2 新增进路

1. 在 `config/line-m9.yaml` 的 `routes:` 下添加进路定义
2. 确保 `axle_section_ids` 中的段 ID 全部在 `segments` 中存在
3. 确保 `start_signal` 和 `end_signal` 在 `signals` 中定义
4. 对于含渡线（crossover）的 BRANCH 进路，确保 `type: BRANCH` 以便验证跳过

### D.3 调试 MA 计算

```bash
# 查看当前 MA 快照
curl -s localhost:8080/api/simulation/snapshot | python -c "
import json,sys; d=json.load(sys.stdin)
for a in d['authorities']:
    print(f'{a[\"trainId\"]}: end={a[\"authorityEndMeters\"]:.0f}m '
          f'spd={a[\"speedLimitMetersPerSecond\"]:.1f}m/s reason={a[\"reason\"][:60]}')"
```

### D.4 调试进路状态

```bash
# 进路列表
curl -s localhost:8080/api/signal-track/routes | python -m json.tool

# 进路事件
curl -s localhost:8080/api/signal-track/route-events | python -m json.tool
```
