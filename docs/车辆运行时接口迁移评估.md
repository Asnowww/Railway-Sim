# 车辆运行时接口迁移评估

## 目标边界

本评估用于约束车辆仿真外置后的接口迁移范围。中央系统仍是前端、综合监控、调度、信号、轨道和供电控制模块的唯一业务入口；**列车状态权威已迁移至 `vehicle-runtime-service:9300`**。

列车上线由 `vehicle-runtime-service` 先启动仿真实例、唤醒随车控制实例、创建 `TrainStateHolder`（权威状态持有者），再向中央注册状态镜像。中央在后续仿真 tick 中向 9300 发送约束（无列车状态），9300 从本地 `TrainStateHolder` 读取状态、计算、返回完整 `TrainStateSnapshot[]`，中央更新镜像。

目标链路固定为：

```text
POST /vehicle-runtime/trains/launch
  -> 创建 VehicleRuntimeInstance + TrainStateHolder
  -> 唤醒 VehicleControlQueue
  -> POST /api/trains/runtime-registrations
  -> 中央建立 TrainEntity 状态镜像（权威在 9300）
  -> 前端/PLC 输入先到8080协议网关，保持PLC 46字节载荷并封装RSIM聚合帧
  -> POST /vehicle-runtime/peripherals/frame（9300只接受PLC_INPUT）→ DriverCommandHolder
  -> 或 POST /vehicle-runtime/trains/{id}/manual-control
  -> 中央 tick: 发送约束（无 trains）→ POST /vehicle-runtime/step-fleet
  -> 9300 从 TrainStateHolder 读取状态 → 控制决策 → FMU 物理计算
  -> 9300 返回 { trainOutputs, trainReports, trainStates(权威), events }
  -> 中央 TrainManager.updateMirrorState() 更新镜像
  -> 9300 事件桥接到中央 SimpleEventBus
  -> 中央 REST / WebSocket 对外发布
```

### 9300 自主模式（新增）

9300 可脱离中央独立运行，用于调试或教学演示：

```text
POST /vehicle-runtime/autonomous/enable
  -> VehicleRuntimeTickClock @Scheduled 启动
  -> 每 100ms: 生成默认约束 → 从 TrainStateHolder 读取 → 控制决策 → FMU → 更新状态
  -> 独立推进无需中央参与
POST /vehicle-runtime/autonomous/disable → 关闭自主模式
POST /vehicle-runtime/tick → 手动单步推进
```

### 新增接口

| 方法 | 路径 | 说明 |
|---|---|---|
| `POST` | `/vehicle-runtime/trains/{trainId}/manual-control` | 手动控制单列车（牵引/制动/紧急制动/方向/门） |
| `GET` | `/vehicle-runtime/trains/{trainId}/state` | 获取单列车权威状态快照 |
| `GET` | `/vehicle-runtime/trains/state` | 获取所有列车权威状态快照 |
| `POST` | `/vehicle-runtime/autonomous/enable` | 开启自主 tick 模式 |
| `POST` | `/vehicle-runtime/autonomous/disable` | 关闭自主 tick 模式 |
| `POST` | `/vehicle-runtime/tick` | 手动单步推进 |
| `POST` | `/vehicle-runtime/peripherals/frame` | 统一外设聚合入口；当前只接受 RSIM Version 1 的 `PLC_INPUT` |
| `GET` | `/vehicle-runtime/peripherals/status` | 查询聚合协议版本、允许通道和收拒帧计数 |

## 数据库依赖评估

| 依赖 | 当前中央用途 | 是否复制到 `vehicle-runtime-service` | 结论 |
|---|---|---|---|
| `StaticInfrastructureCatalog` / 线路数据 | 中央加载线路、区段、站台、供电分区映射，并生成 `TrackConstraint`、`PowerConstraint`。 | 否 | 外部车辆运行时只消费中央下发的列车状态和约束，不能自行读取或解释中央数据库。 |
| `operation_log` / 故障日志 | 中央记录 REST 操作、车辆故障、供电操作审计。 | 否 | 审计仍属于中央接口层；危险操作必须先同步写入 `run_id/tick/trace_id`，失败返回503。外部运行时只保留运行事件用于联调诊断。 |
| `train_physics_snapshot` / 仿真持久化 | 中央持久化车辆状态、能耗、动力学诊断。 | 否 | 外部运行时返回 `VehiclePhysicsOutput` 和带 `decisionSource/inputCommandId/inputTraceId` 的 `TrainStateReport`，由中央按 `simulationRunId/tick` 统一落库。 |
| `simulation_run` / `vehicle_control_command_log` | 中央保存运行生命周期与最终决策。 | 否 | 9300/9200只传递并校验同一 `simulationRunId`，历史查询仍由8080提供。 |
| 车辆运行时实例队列 | 新增服务内每车控制队列和仿真队列。 | 暂不落库 | 首期按内存队列实现，服务重启后由车辆仿真系统重新 `launch`，再向中央注册镜像。 |

结论：当前没有需要复制到新 Spring Boot 的数据库依赖。若后续要求外部车辆运行时具备断点续跑、跨进程租约恢复或独立审计，再新增独立 schema，而不是复用中央业务表。

## 车辆相关接口迁移表

| 接口/模型 | 当前位置 | 迁移决策 | 原因 |
|---|---|---|---|
| `GET /api/trains`、`GET /api/trains/{id}` | 中央 `TrainController` | 保留中央 | 前端和综合监控读取中央状态镜像，不直接访问车辆运行时。 |
| `POST /api/trains/runtime-registrations` | 中央 `TrainController` | 新增服务间入口 | `vehicle-runtime-service` 启动实例后调用，中央只建立状态镜像和审计，不反向拉起 9300。 |
| `POST /api/trains/lifecycle` | 中央 `TrainController` | 保留中央兼容入口 | 用于旧演示、DELETE/CLEAR 和信号生命周期语义测试；推荐上线入口改为 `POST /vehicle-runtime/trains/launch`。 |
| `POST /api/trains/{id}/faults` | 中央 `TrainController` | 保留中央 | 故障注入需要中央审计和跨模块影响，不下放给外部车辆服务。 |
| `GET /api/vehicle/onboard-subsystems` | 中央 `VehicleController` | ✅ 已移除 | 对应 `OnboardTrainSubsystemManager` 已删除。查询 9300 `GET /vehicle-runtime/instances`。 |
| `GET /api/vehicle/runtime-health` | 中央 `VehicleController` | 新增中央代理 | 供前端查看外部车辆运行时健康、实例队列和 fallback，不直连 `9300`。 |
| `GET /api/signal/vehicles/statuses` | 中央 `SignalVehicleInterfaceController` | 保留中央 | 信号系统消费中央列车状态镜像，不访问外部车辆仿真。 |
| `GET /api/signal/vehicles/commands` | 中央 `SignalVehicleInterfaceController` | 保留中央 | MA/限速仍由中央信号模块计算后进入车辆约束链路。 |
| `POST /api/vehicle/driver-cabs/{trainId}/plc-input` | 8080 `DriverCabController` | 协议网关 | 接收前端结构化 JSON，编码为 46 字节 PLC 报文后封装 RSIM 聚合帧；不在中央保存或执行控制。 |
| `POST /vehicle-runtime/peripherals/frame` | 9300 `PeripheralAggregationController` | 新增服务间聚合入口 | 只接受 `PLC_INPUT`；输出/屏幕/视景通道返回 `400 CHANNEL_DIRECTION_INVALID`。 |
| `GET /api/vehicle/driver-cabs/{trainId}/state` / `plc-output` | 中央 `DriverCabController` | 保留8080只读/显示 | 信号侧状态投影和26字节PLC显示输出仍需要中央MA、门使能和网络状态。 |
| `POST /api/signal/vehicles/telemetry` | 中央 `SignalVehicleInterfaceController` | 保留中央 | 外部运营遥测属于中央接入数据源，不等同车辆运行时控制。 |
| `TrackService.updateOccupancy(...)` | 中央轨道模块 | 保留中央 | 轨道占用按中央 `TrainState` 镜像计算，车辆运行时不直接修改轨道状态。 |
| `TrackConstraint` | 中央轨道模块 -> 车辆运行时 | 作为 `step-fleet` 入参迁移 | 轨道限速、坡度、曲线半径和站距由中央算出后下发给外部车辆运行时。 |
| `/api/simulation/snapshot` | 中央 `SimulationController` | 保留中央 | 快照聚合车辆、轨道、信号、供电和运行时健康；前端不直接访问 `9300`。 |
| `/ws/simulation` | 中央 WebSocket | 保留中央 | WebSocket 继续发布中央统一快照，新增 `vehicleRuntime` 字段只反映外部运行时状态。 |
| `VehicleRuntimeStepRequest/Response` | 中央 + `vehicle-runtime-service` | 已迁移完成 | **不再承载 `trains[]`**：9300 从 `TrainStateHolder` 读取权威状态。中央只发送 MA、轨道/调度/供电约束和 `simulationRunId`。响应新增 `trainStates[]`（权威状态快照）和 `events[]`（事件列表），中央用于镜像更新和事件桥接。`driverCommands[]`仅保留兼容。 |
| `VehiclePhysicsClient` | 中央内部 seam | ✅ 已移除 | 9300失败时中止 tick，不由8080静默接管物理；FMU fallback必须由9300内部执行。 |
| `external-simulator` / UDP / RT-LAB API | 中央旧适配 | 暂保留兼容 | 旧 FMU/RT-LAB 原型不删除，后续可并入外部车辆运行时。 |

## 供电仿真联动迁移

`EXTERNAL_HTTP` 且 `forwardPowerLoads=true` 时，写入权固定在9300：

```text
TrainManager.tickAll()
  -> POST /vehicle-runtime/step-fleet
  -> vehicle-runtime-service 聚合 PowerConstraint.sectionId + VehiclePhysicsOutput
  -> POST /power-network/step (simulationRunId + tick + 完整分区负荷)
  -> PowerIntegrationService GET /power-network/state
```

中央车辆运行时的 `LOCAL`、`DUAL_SHADOW` 模式已经移除，不再提供中央兼容路径。下列历史流程不再适用：

```text
TrainManager.tickAll()
  -> 本地 VehiclePhysicsClient 输出
  -> PowerService.updateFromVehicleOutputs()
  -> PowerIntegrationService POST /power-network/state/query
```

`DUAL_SHADOW` 不向供电仿真写负荷，因为该模式下中央本地车辆输出仍是权威结果。

9300/FMU降级不会将9200写入权切回8080。如9300整体不可达，当前tick明确失败，恢复服务后再继续。

## 已验证与未完整逻辑测试

已验证：

- `vehicle-runtime-service` 单元测试覆盖实例队列、重复 tick 拒绝、失电制动和分区负荷转发。
- 中央/车辆运行时定向测试覆盖 PLC唯一入口、未知列车拒绝、最终决策溯源、`simulationRunId`传递和供电固定写入权。
- 前端构建、供电仿真 Python 自测和 `git diff --check` 已通过。

未完整逻辑测试：

- 未进行 `backend + vehicle-runtime-service + power-network-service` 三服务长时间联动压测。
- 未接入真实 FMU/RT-LAB 车辆物理模型。
- 未接入真实 PSCADA 点表或潮流求解器。
- 仍需在真实MySQL执行 P1 schema，验证中央重启后可按同一runId查询决策、车辆、供电和故障时间线。9300实例队列本身仍由外部车辆仿真系统重新launch，不复制中央业务表。
