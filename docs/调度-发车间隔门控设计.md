# 调度-发车间隔门控设计

## 1. 目标

当前调度侧已经能按照运行图生成 `DEPART` 命令，也能在运行过程中计算站点发车间隔和进行间隔调节。缺口在于：当仿真循环启动较晚、恢复后一次性跨过多个计划发车点，或多趟同方向列车在同一个评估周期都已到计划时刻时，调度会把所有已到点的服务一次性放出，导致始发间隔被压缩。

本次发车门控只处理正式运行图自动发车，即 `FORMAL_SERVICE_PLAN -> DEPART`。人工命令、线路运行计划 `REQUEST_ROUTE` 和既有进路申请闭环保持不变。

## 2. 数据来源

发车门控使用调度侧已经持有的数据：

| 数据 | 来源 | 用途 |
|---|---|---|
| `TrainServicePlan` | `dispatch-plan.yaml` | 确认服务、方向、计划发车时间、始发站 |
| `CurrentRunPlan.departureIntervalSec` | 当前运行时段 | 作为目标发车间隔 |
| `departureCommandIdByService` | 调度内存状态 | 判断前车服务是否已经放行 |
| `DispatchCommand.createdAt` | 已生成 `DEPART` 命令 | 记录前车实际调度放行时间 |
| `issuedServiceIds` | 调度内存状态 | 避免重复发车 |

后续如果信号轨道侧提供更稳定的站台占用、列车当前区段和实际出清事件，可进一步把门控从“计划前车已放行”升级为“前车已离开始发站/出清保护区段”。

## 3. 门控规则

### 3.1 适用范围

仅适用于自动运行图发车：

```text
DispatchService.automaticDepartureCommands(...)
```

不拦截：

- 人工 `POST /api/dispatch/commands` 下发的 `DEPART`。
- 线路运行计划到点生成的 `REQUEST_ROUTE`。
- 自动进路申请 `REQUEST_ROUTE`。

### 3.2 同方向顺序

调度按 `direction + originStationId` 分组判断前车。对于同一方向、同一始发站的服务：

1. 按计划发车时刻排序。
2. 第一趟到点后可发。
3. 后续服务必须等上一趟同组服务已经生成 `DEPART`，且距离上一趟调度放行时间达到目标间隔。

这里的“上一趟”来自运行图服务顺序，而不是实时位置排序。这样可以在维持四条粗粒度进路的前提下，先保证始发放行节奏稳定。

### 3.3 目标间隔

默认目标间隔使用当前时段 `departureIntervalSec`，例如高峰 180 秒、平峰 300 秒。

为了允许现场演示或故障恢复，可配置门控比例：

```yaml
railway:
  dispatch:
    departure-headway-gate-enabled: true
    departure-headway-gate-ratio: 1.0
```

实际门控间隔：

```text
requiredHeadwaySec = currentPlan.departureIntervalSec * departureHeadwayGateRatio
```

比例默认为 `1.0`，即严格按运行图目标间隔放行。

### 3.4 延迟处理

如果某服务已经到计划发车时刻，但不满足门控：

- 不生成 `DEPART`。
- 不写入 `issuedServiceIds`。
- 下一个调度评估周期继续判断。

满足门控后再生成 `DEPART`，payload 中的 `dispatchDelaySec` 会自然记录实际放行相对计划时刻的延迟。

## 4. 状态展示

当前 `ServicePlanView.departureStatus` 仍沿用：

- `PLANNED`：尚未生成 `DEPART`，包括未到点或被门控暂缓。
- `PENDING/SENT/APPLIED/...`：已生成命令后的状态。

后续如果前端需要区分“未到点”和“门控等待”，可以在 `ServicePlanView` 增加 `departureHoldReason/nextEligibleDepartureAt`。本次先不扩展快照字段，避免前端接口变更扩大。

## 5. 与信号轨道侧边界

本次门控不要求信号轨道侧拆细进路。维持四条粗粒度进路时，调度只保证“发车节奏”：

- 调度决定某趟车何时放行。
- 信号仍负责进路可用性、联锁安全、MA 和限速。
- 如果信号拒绝进路或未给授权，车辆仍由信号安全约束保持。

建议后续信号轨道侧补充：

- `TrainState.currentSegmentId` 或 `trackSegments[].occupyingTrainIds`，用于按区段判断前车位置。
- 始发站/折返站出清事件，用于把门控条件升级为“前车已离站/出清”。
- 站台占用和门状态，用于避免后车在站台未释放时发车。

## 6. 验收场景

| 场景 | 初始条件 | 预期 |
|---|---|---|
| 首车到点 | 当前无同方向已发服务 | 立即生成 `DEPART` |
| 多车同时过点 | 同方向多趟服务都已到计划时刻 | 同一评估周期只放行首个未发服务 |
| 间隔未满足 | 前车刚生成 `DEPART`，后车已到计划时刻 | 后车保持 `PLANNED` |
| 间隔满足 | 前车放行时间已超过目标间隔 | 后车生成 `DEPART`，记录 `dispatchDelaySec` |
| 门控关闭 | `departure-headway-gate-enabled=false` | 恢复原逻辑，所有到点服务可生成 `DEPART` |
