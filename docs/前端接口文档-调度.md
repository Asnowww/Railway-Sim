# 调度模块 — 前端接口文档

> 刘倩 | 2026-07-13 | 面向调度页重构与调度-信号闭环联调

---

## 〇、关于这份文档

> 本文档面向前端重构使用，重点说明调度页需要调用的 REST 接口、WebSocket 数据来源、字段结构、闭环关联方式和当前待补字段。
>
> 调度侧不维护轨道拓扑，也不允许前端在调度页新增/编辑路线。调度页只能使用信号/轨道侧已经存在的 `routeId`、区段、道岔、MA 和进路状态。
>
> 调度闭环的展示目标不是“堆接口数据”，而是把同一次调度从计划、命令、进路、授权、列车状态到效果确认串成一条执行证据链。

## 一、调度页推荐数据获取流程

```text
1. 页面初始化
   -> GET /api/dispatch/plan
   -> GET /api/dispatch/status
   -> GET /api/dispatch/operation-route/templates
   -> GET /api/dispatch/operation-plans
   -> GET /api/signal-track/dispatch-publications/latest（若 404，则表示尚未发布）

2. 仿真运行期间
   -> 通过 ws://localhost:8080/ws/simulation 接收 SimulationSnapshot
   -> 周期性或刷新按钮调用 GET /api/dispatch/status
   -> 必要时调用 GET /api/signal-track/route-events 获取进路生命周期事件

3. 手动演示/联调
   -> POST /api/dispatch/disturbances/demo 注入调度扰动
   -> POST /api/dispatch/commands 下发调度命令
   -> POST /api/dispatch/operation-plans 创建线路运行计划
   -> POST /api/dispatch/signal-publications 发布计划给信号端

4. 闭环展示
   -> 用 trainId / routeId / commandId / reservationId / operationPlanId / publicationId 做关联
   -> 优先使用 commandId、reservationId 强关联；trainId + routeId 只作为兜底关联
```

## 二、数据来源总览

### 2.1 调度侧 REST

基础前缀：

```http
http://localhost:8080/api/dispatch
```

| 方法 | URL | 说明 | 前端用途 |
|---|---|---|---|
| GET | `/api/dispatch/plan` | 完整运行图计划 | 间隔时刻表、线路计划、站点/区段基础信息 |
| GET | `/api/dispatch/plan/current` | 当前时段运行计划 | 当前模式、目标间隔、默认停站 |
| GET | `/api/dispatch/status` | 调度快照 | 调度首页主要数据源 |
| GET | `/api/dispatch/disturbances` | 扰动列表 | 调度关注、扰动列表 |
| POST | `/api/dispatch/disturbances/demo` | 注入演示扰动 | 演示调度效果 |
| GET | `/api/dispatch/commands` | 调度命令历史 | 命令时间线、闭环状态 |
| POST | `/api/dispatch/commands` | 手工下发调度命令 | 扣车、限速、进路申请等演示 |
| POST | `/api/dispatch/commands/{commandId}/cancel` | 取消命令 | 取消临时限速、进路申请等 |
| GET | `/api/dispatch/route/list` | 信号侧进路列表 | 只读展示可用 routeId |
| POST | `/api/dispatch/route/establish` | 快捷申请进路 | 调试入口，不建议作为正式页面主入口 |
| GET | `/api/dispatch/operation-route/templates` | 线路运行计划可选路线模板 | 创建线路运行计划的路线选择来源 |
| POST | `/api/dispatch/operation-route/preview` | 预览点序列匹配路线 | 点选站点后预览候选路线 |
| GET | `/api/dispatch/operation-plans` | 已创建线路运行计划 | 线路计划列表 |
| POST | `/api/dispatch/operation-plans` | 创建线路运行计划 | 创建“某车某时刻走某既有路线” |
| POST | `/api/dispatch/operation-plans/{planId}/cancel` | 取消线路运行计划 | 取消未执行计划 |
| POST | `/api/dispatch/signal-publications` | 发布调度计划到信号端 | 计划接收闭环 |
| POST | `/api/dispatch/headway/adjust` | 手动间隔调节命令 | 间隔调度演示 |
| GET | `/api/dispatch/station-records` | 站点到发记录 | 到发偏差、站停效果确认 |

### 2.2 信号轨道侧反馈 REST

调度页需要消费部分信号接口，但只作为反馈展示，不在调度页做信号配置。

| 方法 | URL | 说明 | 调度页用途 |
|---|---|---|---|
| GET | `/api/signal-track/dispatch-publications` | 信号侧已接收的全部调度计划发布记录 | 计划发布历史 |
| GET | `/api/signal-track/dispatch-publications/latest` | 最近一次调度计划发布 | 计划接收状态 |
| GET | `/api/signal-track/route-events` | 进路生命周期事件 | 进路执行时间线 |
| GET | `/api/signal-track/routes` | 当前所有进路状态 | 调度反馈详情 |
| GET | `/api/signal-track/routes/{routeId}/status` | 单条进路状态 | 进路详情弹窗 |

注意：当前 `/api/signal-track/route-events` 是 drain 型接口，读取后会清空本批事件。前端重构时如果多个页面都要消费进路事件，建议后续改为非破坏式 `sinceSeq` 查询；当前阶段可先由调度页单点消费，或只作为调试入口。

### 2.3 WebSocket

```text
ws://localhost:8080/ws/simulation
消息格式：{ "type": "snapshot", "payload": SimulationSnapshot }
```

`SimulationSnapshot` 是实时运行状态来源，调度页至少要使用：

| 字段 | 说明 | 调度页用途 |
|---|---|---|
| `trains[]` | 列车实时状态 | 速度、位置、当前站、状态、停站时间 |
| `authorities[]` | 每车移动授权 MA | 授权终点、有效限速、当前区段 |
| `trackSegments[]` | 区段状态 | 冲突/故障原因详情 |
| `routeStates[]` | 进路状态 | 信号反馈概要 |
| `signalStates[]` | 信号机状态 | 红/黄/绿原因展示 |
| `switchStates[]` | 道岔状态 | 进路失败/锁闭详情 |

## 三、核心页面结构建议

调度页建议按以下 5 个业务区组织，不再展示完整拓扑图，也不提供路线新增/编辑。

| 页面区块 | 主要数据源 | 作用 |
|---|---|---|
| 运行概览 | `/dispatch/status` | 当前模式、目标间隔、平均偏差、命令闭环 |
| 演示触发 | `/dispatch/disturbances/demo`、`/dispatch/commands` | 手动触发扰动和调度命令，服务演示 |
| 间隔时刻表 | `/dispatch/plan`、`/dispatch/status`、WebSocket trains | 展示计划到发偏移、当前列车位置/状态 |
| 线路运行计划 | `/dispatch/operation-route/templates`、`/dispatch/operation-plans` | 创建/展示调度线路计划，只能选择既有 routeId |
| 信号执行反馈 | `/signal-track/dispatch-publications/latest`、`/signal-track/route-events`、WebSocket authorities | 展示计划接收、进路生命周期、MA/限速和效果证据 |

## 四、运行图与时刻表接口

### 4.1 获取完整运行图

```http
GET /api/dispatch/plan
```

响应类型：

```typescript
interface RunPlanResponse {
  planId: string
  lineId: string
  periods: RunPlanPeriod[]
  stations: DispatchPlanStation[]
  segments: DispatchPlanSegment[]
  circulations: CirculationPlan[]
  services: TrainServicePlan[]
}

interface RunPlanPeriod {
  periodType: 'PEAK' | 'FLAT' | 'OFF_PEAK' | string
  start: string
  end: string
  departureIntervalSec: number
  defaultDwellTimeSec: number
}

interface DispatchPlanStation {
  id: string
  positionMeters: number
  platformCapacity: number
}

interface DispatchPlanSegment {
  id: string
  startMeters: number
  endMeters: number
  speedLimitMps: number
}

interface TrainServicePlan {
  serviceId: string
  circulationId: string
  trainId: string
  trainNo: number
  linkId: number
  offsetMeters: number
  direction: 'UP' | 'DOWN' | string
  stops: PlannedStop[]
}

interface PlannedStop {
  stationId: string
  arrivalOffsetSec: number
  departureOffsetSec: number
}
```

前端展示建议：

- `periods` 用于当前模式说明：高峰/平峰/低谷、发车间隔、默认停站。
- `services[].stops` 用于“间隔时刻表”，按 `origin.departureOffsetSec` 排序。
- `segments[].speedLimitMps` 是基础限速，不等于实时有效限速。实时有效限速以 `MovementAuthority.speedLimitMetersPerSecond` 为准。

### 4.2 获取当前计划

```http
GET /api/dispatch/plan/current
```

响应字段来自 `CurrentRunPlan`，用于当前运行模式卡片：

```typescript
interface CurrentRunPlan {
  planId: string
  lineId: string
  runMode: string
  departureIntervalSec: number
  defaultDwellTimeSec: number
  resolvedAt: string
}
```

## 五、调度快照接口

### 5.1 获取调度快照

```http
GET /api/dispatch/status
```

响应类型：

```typescript
interface DispatchSnapshot {
  runMode: string
  planId: string
  targetHeadwaySeconds: number
  defaultDwellSeconds: number
  services: DispatchServicePlanView[]
  stationHeadways: DispatchStationHeadway[]
  interventionActive: boolean
  trainProfiles: DispatchTrainProfile[]
  openDisturbances: DispatchDisturbance[]
  activeCommands: DispatchCommandView[]
  routeDispatchActive: boolean
  routeDecisions: DispatchRouteDecision[]
  routeReservations: DispatchRouteReservation[]
  operationPlans: OperationPlanView[]
}
```

前端主要用法：

| 字段 | 页面用途 |
|---|---|
| `runMode` | 当前模式 KPI |
| `targetHeadwaySeconds` | 目标间隔 |
| `services[]` | 正式发车计划状态 |
| `stationHeadways[]` | 站点间隔观测 |
| `trainProfiles[]` | 调度效果判断：间隔偏差、策略、晚点 |
| `openDisturbances[]` | 调度关注和扰动列表 |
| `activeCommands[]` | 命令时间线、命令闭环 |
| `routeDecisions[]` | 调度侧进路决策 |
| `routeReservations[]` | 调度侧进路预约状态 |
| `operationPlans[]` | 线路运行计划列表 |

### 5.2 调度效果相关类型

```typescript
interface DispatchTrainProfile {
  trainId: string
  regulatedTrainId: string
  frontTrainId: string | null
  headwayActualSeconds: number | null
  headwayErrorSeconds: number | null
  headwayDeviationSeconds: number
  headwayState: string
  headwayAction: string
  regulationAction: 'CATCH_UP' | 'SLOW_DOWN' | 'NORMAL' | 'OBSERVE' | string
  regulationReason: string
  dwellDeviationSeconds: number
  departureDelaySeconds: number
}
```

推荐展示：

- `headwayActualSeconds / targetHeadwaySeconds`：实际/目标间隔。
- `headwayErrorSeconds`：剩余偏差，适合作为“效果确认”主指标。
- `regulationAction`：调度策略，显示为追赶/放慢/正常/观察。
- `departureDelaySeconds`：发车晚点。

### 5.3 进路预约与反馈字段

```typescript
interface DispatchRouteReservation {
  reservationId: string
  trainId: string
  routeId: string
  decisionId: string
  state: string
  commandId: string
  rejectReason: string | null
  failureCode: string | null
  failureCategory: string | null
  retryable: boolean
  retryCount: number
  expiresAt: string | null
  nextRetryAt: string | null
  timedOutAt: string | null
  cancelCommandId: string | null
}
```

前端展示建议：

| 字段 | 展示方式 |
|---|---|
| `state` | 预约状态徽标，如 REQUESTED / ACCEPTED / REJECTED / RELEASED / TIMEOUT |
| `failureCode` | 失败码，优先展示中文映射 |
| `failureCategory` | 失败类别，用于分组和颜色 |
| `retryable` | 是否可重试 |
| `retryCount` | 已重试次数 |
| `rejectReason` | 详细原因 |

信号侧已定义标准 `failureCode`：

```text
ROUTE_ESTABLISHED
ROUTE_NOT_FOUND
TRACK_OCCUPIED
TRACK_FAULT
ROUTE_CONFLICT
SWITCH_LOCKED
SWITCH_MOVE_FAILED
ROUTE_NOT_AVAILABLE
SWITCH_POSITION_MISMATCH
TRACK_NOT_FOUND
NO_MATCHING_ROUTE
UNSUPPORTED_COMMAND
INTERNAL_ERROR
```

建议中文映射：

| failureCode | 中文 | 页面建议 |
|---|---|---|
| `ROUTE_ESTABLISHED` | 进路已建立 | 成功 |
| `ROUTE_NOT_FOUND` | 进路不存在 | 配置错误，转人工 |
| `TRACK_OCCUPIED` | 区段占用 | 等待或重试 |
| `TRACK_FAULT` | 区段故障 | 等待恢复或改路 |
| `ROUTE_CONFLICT` | 进路冲突 | 排队或重试 |
| `SWITCH_LOCKED` | 道岔锁闭 | 等待释放 |
| `SWITCH_MOVE_FAILED` | 道岔转换失败 | 转人工检查 |
| `ROUTE_NOT_AVAILABLE` | 进路不可用 | 等待状态迁移 |
| `SWITCH_POSITION_MISMATCH` | 道岔位置不匹配 | 配置矛盾，转人工 |
| `TRACK_NOT_FOUND` | 区段不存在 | 基础数据不一致 |
| `NO_MATCHING_ROUTE` | 无匹配进路 | 请求不完整或配置错误 |
| `UNSUPPORTED_COMMAND` | 不支持的命令 | 接口命令错误 |
| `INTERNAL_ERROR` | 信号内部错误 | 转人工 |

待信号侧补充字段：

```typescript
interface FutureRouteFailureDetail {
  conflictSegmentIds?: string[]
  conflictTrainIds?: string[]
  conflictRouteIds?: string[]
  affectedSwitchIds?: string[]
}
```

这些字段补齐后，前端应优先显示“卡在哪个资源”，而不是只显示失败码。

## 六、调度命令接口

### 6.1 获取命令列表

```http
GET /api/dispatch/commands
```

响应类型：

```typescript
interface DispatchCommandView {
  id: string
  trainId: string
  regulatedTrainId?: string | null
  commandType: string
  status: string
  reason: string
  regulationAction?: string | null
  payload?: Record<string, unknown>
  createdAt?: string
  appliedAt?: string | null
}
```

推荐状态映射：

| status | 中文 |
|---|---|
| `PENDING` | 待下发 |
| `SENT` | 已下发 |
| `APPLIED` | 执行中 |
| `EFFECT_CONFIRMED` | 效果已确认 |
| `SKIPPED` | 已拒绝/跳过 |
| `TIMEOUT` | 超时 |
| `CANCELLED` | 已取消 |
| `EXPIRED` | 已结束 |
| `RELEASED` | 已释放 |

### 6.2 下发调度命令

```http
POST /api/dispatch/commands
Content-Type: application/json
```

请求体：

```typescript
interface ManualCommandRequest {
  trainId: string
  commandType: string
  detail?: string
  targetHeadwaySec?: number
  speedBiasRatio?: number
  deltaDwellSec?: number
  routeId?: string
  payload?: Record<string, unknown>
}
```

示例：临时限速

```json
{
  "trainId": "TR-001",
  "commandType": "SPEED_LIMIT",
  "detail": "8.0",
  "payload": {
    "speedLimitMps": 8.0,
    "regulationSource": "DISPATCH_WORKSTATION_DEMO"
  }
}
```

示例：申请进路

```json
{
  "trainId": "TR-001",
  "commandType": "REQUEST_ROUTE",
  "routeId": "R_MAIN",
  "payload": {
    "source": "MANUAL_ROUTE_REQUEST"
  }
}
```

支持的主要 `commandType`：

| commandType | 说明 | 主要字段 |
|---|---|---|
| `REQUEST_ROUTE` | 申请进路 | `routeId` |
| `REROUTE` | 重排进路 | `routeId` |
| `CANCEL_ROUTE` | 取消进路 | `routeId`、`reservationId` |
| `DEPART` | 发车 | 通常由正式计划自动生成 |
| `HOLD` / `HOLD_TRAIN` | 扣车 | `trainId` |
| `SPEED_LIMIT` / `TEMP_SPEED_LIMIT` | 临时限速 | `detail` 或 `payload.speedLimitMps` |
| `SPEED_BIAS` | 速度偏置 | `speedBiasRatio` |
| `HEADWAY_ADJUST` | 间隔调节 | `targetHeadwaySec`、`regulationAction` |
| `EXTEND_DWELL` | 延长停站 | `deltaDwellSec` |
| `SHORTEN_DWELL` | 缩短停站 | `deltaDwellSec` |

### 6.3 取消命令

```http
POST /api/dispatch/commands/{commandId}/cancel
```

响应：

```json
{
  "accepted": true,
  "commandId": "DC-xxx"
}
```

## 八、线路运行计划接口

### 8.1 获取既有路线模板

```http
GET /api/dispatch/operation-route/templates
```

响应：

```typescript
interface OperationRouteTemplate {
  routeId: string
  name: string
  typeCode: string
  pointIds: string[]
  stationIds: string[]
  segmentIds: string[]
}
```

前端约束：

- 这里只能展示和选择已有模板，不能在调度页新增 route。
- routeId 必须来自信号/轨道拓扑已有进路，例如 `R_MAIN`、`R_NORTH`、`R_DEPOT`、`R_LOOP`、`R_BRANCH`。
- 如果后端模板和信号拓扑不一致，最终发布到信号端时仍以信号 route catalog 校验结果为准。

```json
{
  "operator": "dispatch-workstation",
  "effectiveFrom": "2026-07-13T09:00:00Z"
}
```

如果请求体为空，后端默认：

| 字段 | 默认值 |
|---|---|
| `operator` | `dispatch` |
| `effectiveFrom` | 当前仿真 run 的 `simulationStart` |

### 9.2 发布响应

```typescript
interface SignalDispatchPlanPublication {
  publicationId: string
  simulationRunId: string
  dispatchPlanId: string
  lineId: string
  effectiveFrom: string
  publishedAt: string
  operator: string
  status: 'ACCEPTED' | 'PARTIAL_ACCEPTED' | 'REJECTED' | string
  acceptedCount: number
  rejectedCount: number
  entries: SignalDispatchPlanPublicationEntry[]
}

interface SignalDispatchPlanPublicationEntry {
  entryId: string
  sourceType: 'SERVICE_PLAN' | 'OPERATION_PLAN' | string
  sourceId: string
  trainId: string
  routeId: string
  routeName: string
  direction: 'UP' | 'DOWN' | string
  originPointId: string | null
  destinationPointId: string | null
  viaPointIds: string[]
  stationIds: string[]
  segmentIds: string[]
  plannedDepartureAt: string | null
  status: 'ACCEPTED' | 'REJECTED' | string
  rejectReason: string | null
}
```

当前发布规则：

| sourceType | 来源 | routeId 规则 |
|---|---|---|
| `SERVICE_PLAN` | 正式发车计划 `TrainServicePlan` | 当前固定发布到既有主线 `R_MAIN` |
| `OPERATION_PLAN` | 线路运行计划 `OperationPlan` | 使用创建计划时选择的既有 `routeId` |

当前调度侧发布时会校验 `routeId` 是否存在于信号/轨道拓扑的 `RouteCatalog`。如果不存在，该 entry 会返回：

```json
{
  "status": "REJECTED",
  "rejectReason": "routeId R_UNKNOWN is not provided by signal route catalog"
}
```

### 9.3 信号侧发布记录查询

```http
GET /api/signal-track/dispatch-publications
GET /api/signal-track/dispatch-publications/latest
```

`latest` 如果当前还没有发布记录，会返回 HTTP 404。前端应展示为“尚未发布计划”，不要作为错误弹窗。

### 9.4 待信号侧补充：publication ack

信号侧下一步计划补逐条接收回执。前端字段可预留：

```typescript
interface FuturePublicationAckEntry {
  entryId: string
  trainId: string
  routeId: string
  status: 'ACCEPTED' | 'REJECTED'
  rejectReason: string | null
  failureCode?: string
  retryable?: boolean
}
```

注意边界：

- 计划发布 ack 主要判断计划字段是否合法、routeId 是否存在、计划是否可接收。
- 当前瞬时区段占用不建议在 publication ack 阶段直接拒绝未来计划。
- 到点执行时的现场资源冲突，应由 `REQUEST_ROUTE` 结构化反馈体现。

## 十、信号执行反馈接入

调度页中的“信号执行反馈”建议拆成 4 个小区块。

### 10.1 计划接收

数据源：

```http

```

进路生命周期状态：

```text
AVAILABLE
VALIDATING
SETTING_SWITCHES
LOCKED
OCCUPIED
RELEASING
RELEASED
CONFLICTED
REJECTED
CANCELLED
```

```text
TR-001 / R_MAIN
REQUESTED -> VALIDATING -> SETTING_SWITCHES -> LOCKED -> OCCUPIED -> RELEASED
```

当前关联规则：

| 优先级 | 关联字段 |
|---:|---|
| 1 | `commandId` |
| 2 | `reservationId` |
| 3 | `trainId + routeId + 时间窗口` |


### 10.4 命令效果确认

数据源：

```text
DispatchSnapshot.activeCommands[]
DispatchSnapshot.trainProfiles[]
SimulationSnapshot.authorities[]
SimulationSnapshot.trains[]
DispatchSnapshot.routeReservations[]
```

推荐效果判断：

| commandType | 确认依据 |
|---|---|
| `REQUEST_ROUTE` | 对应 reservation 进入 ACCEPTED/RELEASED，或 route event 到 LOCKED/OCCUPIED/RELEASED |
| `SPEED_LIMIT` | MA 有效限速小于等于目标限速，且列车速度逐步降到目标附近 |
| `HEADWAY_ADJUST` | `headwayErrorSeconds` 回到容差范围 |
| `EXTEND_DWELL` | 列车在站停留时间增加，命令进入 APPLIED/EFFECT_CONFIRMED |
| `SHORTEN_DWELL` | 列车提前释放停站并发车，命令进入 APPLIED/EFFECT_CONFIRMED |
| `HOLD` | 列车保持停车，速度为 0，命令进入 APPLIED |

## 十二、前端闭环视图模型建议

前端重构时建议不要直接把各接口原始数据平铺展示，而是聚合为“调度执行链”。

```typescript

  commandId?: string

  decisionId?: string

  operationPlanId?: string
  publicationId?: string

  authorityStatus: '无授权' | '已授权' | '受限授权'
```

## 十五、前端重构注意事项

1. 调度页不要再放完整路线拓扑图，避免与首页/信号轨道页重复。
2. 调度页不要提供新增路线能力，只能选择信号/轨道已有 `routeId`。
3. “线路计划”不是画轨道，而是展示计划车、计划时刻、既有 routeId、起终点和状态。
4. “信号执行反馈”应优先展示执行证据链：计划接收、进路状态、MA/限速、效果确认。
5. `route-events` 当前是 drain 型接口，重构时不要多个组件同时轮询。
6. 实时限速以 MA 为准，计划里的 `speedLimitMps` 只是区段基础限速。
7. `publication ack`、`speedLimitSources`、冲突资源明细目前是待补字段，前端可预留结构但不要当作已稳定字段强依赖。

## 十六、最小联调验收场景

| 场景 | 操作 | 预期展示 |
|---|---|---|
| 发布计划 | 点击“发布计划”或调用 `/dispatch/signal-publications` | 显示 publicationId、接收数、拒绝数 |
| 正常建路 | 创建 operation plan，tick 到计划时间 | route reservation 进入 ACCEPTED，route event 到 LOCKED/OCCUPIED |
| 进路冲突 | 前方区段占用时申请同一路线 | 显示 failureCode、rejectReason、retryable |
| 临时限速 | 下发 `SPEED_LIMIT=8.0` | MA 有效限速变为 8.0，列车速度逐步下降 |
| 间隔过短 | 注入 TOO_SHORT 扰动 | 显示策略、命令、剩余偏差逐步恢复 |
| 停站延长 | 注入 DWELL_EXTENDED | 命令进入 APPLIED/EFFECT_CONFIRMED，站停时间增加 |
