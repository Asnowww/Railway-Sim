## 一、M9 真实线路数据接入

### 1.1 长段按站间距切分

M9 原始数据中信号机闭塞区间长度分布极端（端部 17m~215m，中部 4.6km~5.5km）。编写 Python 脚本将 8 个长段（上下行各 4 个：U20/U23/U34/U42 + D17/D21/D28/D36）按站台中心公里标切分为子段。

| 指标 | 切分前 | 切分后 |
|------|--------|--------|
| 上行段数 | 13 | 24 |
| 下行段数 | 12 | 23 |
| 最大段长 | 5487m | 1912m |
| 典型段长 | - | 800~1800m |
| 新增节点 | - | 22 个 |

进路 `axle_section_ids`、信号 `segment_id` 自动映射到对应子段。

### 1.2 环线里程探索

尝试了多种下行方向实现方案——from↔to 交换、LD 段追加、环路 km 偏移——最终确认物理引擎不支持递减公里标是核心瓶颈，当前方案为在快照层做 DOWN 位置镜像（`L - position`），内部全部使用物理递增 km。环路方案的技术储备已写入 `scripts/fold_m9_loop.py`。

## 二、上下行双线独立运行

### 2.1 问题背景

M9 上下行在 0→16489m 公里标空间上完全重叠。同一位置存在 up 和 down 两套区段（如 U02 和 D03 都在 0→215m），不隔离会导致 MA 跨轨截断、预留跨轨溢出。

### 2.2 kmForwardMap

新增 `TrackService.kmForwardMap()`——基于公里标数值邻接（`next.start ≈ this.end`）加轨道过滤（`next.track == this.track`），不依赖 YAML 的 from→to 节点方向。`resolveTopologyObstacle` 和 `collectReservedAlongActiveTopology` 全部切换到 `kmForwardMap`。

### 2.3 四层轨道过滤

| 层次 | 位置 | 规则 |
|------|------|------|
| 段定位 | `segmentAt()` | main > up > down |
| 段绑定回退 | `segmentForTrain()` fallback | prevTrack > main > up > down |
| MA 障碍搜索 | `resolveTopologyObstacle()` | 异轨 → INFINITY |
| 预留扩散 | `collectReservedAlongActiveTopology()` | 异轨 → break |

### 2.4 关键修复

- **拓扑障碍跨轨**：搜索到异轨段立即返回无穷大，不再沿渡线跳到对向轨道
- **预留跨轨溢出**：同轨过滤 + 渡线即停，彻底解决一大片红色问题
- **MA 同轨前后车判定**：`calculateAuthorities()` 线性排序时检查 `differentTrack`，异轨列车不构成前后阻挡关系
- **overlaps 区间判断**：从 `head > segStart && tail < segEnd` 改为 `tail < segEnd && head > segStart`，列车驶过立即释放占用

## 三、调度→信号→车辆发车闭环

### 3.1 三个阻塞点的修复

| 阻塞点 | 位置 | 修复 |
|--------|------|------|
| 硬编码初始车 | `TrainManager.reset()` | 删除 `addInitialTrain("TR-001/002")` 调用 |
| DOWN 里程门禁 | `handleDepartures()` | 删除 `toPos < fromPos → SKIP` 检查 |
| 轨道绑定条件 | `handleDepartures()` | 从 `offset > L/2` 改为无条件绑定 |

### 3.2 发车流程

```
dispatch-plan.yaml → DispatchService.evaluate
  → automaticDepartureCommands (30s 窗口)
    → handleDepartures (trainNo/linkId/offset/direction)
      → trainManager.addTrain
        → assignTrainToTrack (UP→"up", DOWN→"down")
          → touchRoutes → R_UP/R_DOWN → MA → 车辆运行
```

### 3.3 折返终点释放

`updateOccupancy()` 新增终点站列车释放逻辑：列车到达首站或末站 ±120m 且零速后不再标记占用。

### 3.4 发车密度配置

调度计划配置为 16 车、50s 间隔全上行走廊发车，支持高峰密度演示。

## 四、调度命令三层反馈闭环

### 4.1 三层模型

```
调度命令 (PENDING)
  └→ 信号接收回执    ← POST /api/signal-track/command-receipt
       └→ 应用反馈   ← generateTimeCommandFeedback (每 tick)
            └→ 效果确认 ← checkDispatchCompletion
```

### 4.2 新增端点

| 端点 | 用途 |
|------|------|
| `POST /api/signal-track/command-receipt` | 信号侧命令接收回执，支持批量 |
| `POST /api/dispatch/command-feedback` | 调度侧统一反馈入口 |
| `GET /api/signal-track/dispatch-publications` | 调度计划发布记录查询 |

### 4.3 Publication Ack

`SignalDispatchPlanRegistry.acceptAndValidate()` 在调度发布计划时逐条校验每个 entry 的 routeId 是否存在且可用，返回带 per-entry `ACCEPTED/REJECTED` 状态和 `acceptedCount/rejectedCount` 的验证快照。

### 4.4 进路冲突诊断

`RouteDispatchResult` 新增 `conflictRouteId` 和 `conflictSegmentIds` 字段。进路被拒时，调度侧可直接定位冲突来源进路和冲突区段。

## 五、站台停靠与进站制动保护

### 5.1 DOWN 站台查找

下行列车使用 `effectiveHead = lineLen - head` 镜像公里标查找站台，确保站序从 S113 反向匹配到 S101。

### 5.2 制动保护优化

仅对高速（> 2.0 m/s）接近站台的列车做长距离制动保护，避免 M9 长段中过早截断 MA 导致刹车停在站窗外。慢速/零速直接允许截断进站。

## 六、MAIN 进路多车追踪

`touchRoutes` 新增 hold 自动清除逻辑：列车如果已在已建立的 MAIN 进路（OCCUPIED 状态）上，但被 `routeHold` 锁着，自动清除 hold，允许后车跟随进入同一正线进路。
