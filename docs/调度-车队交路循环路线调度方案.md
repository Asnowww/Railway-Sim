# 调度-车队交路循环路线调度方案

## 1. 背景与目标

当前路线调度已经具备单条运营进路计划能力：调度端读取信号轨道侧已有 route/template，生成 `OperationPlan`，到达计划发车时刻后下发 `REQUEST_ROUTE`，并通过 `RouteReservation` 跟踪信号接受、拒绝、超时和释放状态。

下一步目标不是继续让调度端手动创建零散路线，而是形成“车队交路循环计划”：

1. 车辆对象由信号轨道侧或车辆侧创建，调度侧只同步可用车辆。
2. 调度侧按车辆起点给每辆车分配固定交路模板。
3. 每辆车按 `R_UP -> R_TB_LIB -> R_DOWN -> R_TB_GGZ` 或从下行起点旋转后的顺序循环运行。
4. 每个 route leg 到点后复用现有 `OperationPlan -> REQUEST_ROUTE -> RouteReservation` 闭环。
5. 完成设定循环次数后，调度不再派发下一段 route，车辆进入下线休息状态。

## 2. 术语划分

### 2.1 Route

`Route` 是信号轨道侧已有进路或路线。调度端不能创建轨道拓扑，也不应新增信号 route，只能引用信号端已经发布的 routeId。

M9 交路循环需要的 route：

| routeId | 含义 | 起终点 |
| --- | --- | --- |
| `R_UP` | 上行正线 | 郭公庄 `S101` -> 国家图书馆 `S113` |
| `R_TB_LIB` | 国家图书馆折返 | 上行转下行 |
| `R_DOWN` | 下行正线 | 国家图书馆 `S113` -> 郭公庄 `S101` |
| `R_TB_GGZ` | 郭公庄折返 | 下行转上行 |

### 2.2 CirculationLeg

`CirculationLeg` 是调度侧交路中的一个执行片段。它引用一个已有 route，并记录当前 leg 的计划发车、执行状态、命令 ID 和拒绝原因。

### 2.3 TrainCirculationPlan

`TrainCirculationPlan` 是调度侧分配给单辆车的交路循环计划。它包含：

- 车辆 ID。
- 起点端点。
- 循环目标次数。
- 当前已完成次数。
- 当前正在执行的 leg。
- 每个 leg 的状态。

## 3. 交路模板

系统内置一个 M9 环形模板：

```text
M9_LOOP = [
  R_UP,
  R_TB_LIB,
  R_DOWN,
  R_TB_GGZ
]
```

不同起点只是从模板不同位置开始：

| 起点 | 起始 leg | 执行顺序 |
| --- | --- | --- |
| 郭公庄 `S101` | `R_UP` | `R_UP -> R_TB_LIB -> R_DOWN -> R_TB_GGZ` |
| 国家图书馆 `S113` | `R_DOWN` | `R_DOWN -> R_TB_GGZ -> R_UP -> R_TB_LIB` |

一个完整循环定义为跑完四个 leg。`cycleTarget = 3` 表示该车完整跑 3 圈后下线休息。

## 4. 车辆分配规则

调度侧从当前仿真快照中读取轨道侧已创建车辆，按当前位置或当前站点识别起点：

1. `currentStationId = S101`：分配郭公庄起始交路。
2. `currentStationId = S113`：分配国家图书馆起始交路。
3. 没有站点时，可按线路两端里程兜底识别。
4. 已经分配交路且未取消/休息的车辆不重复分配。
5. 故障等级较高的车辆不自动分配。

同一方向内，车辆按目标发车间隔排队：

```text
郭公庄队列：TR-001, TR-003 ...
国家图书馆队列：TR-002, TR-004 ...
```

计划发车时刻：

```text
plannedStart = 当前仿真时间 + leadSeconds + 同方向序号 * headwaySeconds
```

## 5. 执行闭环

每个 leg 到计划发车时间后，调度侧生成 `REQUEST_ROUTE` 指令，payload 中携带：

- `source = CIRCULATION_PLAN`
- `circulationId`
- `legId`
- `cycleIndex`
- `legIndex`
- `routeId`
- `plannedDepartureAt`
- `dispatchDelaySec`

信号侧返回反馈后，调度侧通过现有 `RouteReservation` 记录：

- `REQUESTED`
- `ACCEPTED`
- `REJECTED`
- `TIMEOUT`
- `RELEASED`
- `CANCELLED`

交路层根据 `OperationPlan` 和 `RouteReservation` 推进 leg：

1. `REQUEST_ROUTE` 已生成：leg 进入 `ROUTE_REQUESTED`。
2. reservation 被接受：leg 进入 `ROUTE_ACCEPTED`。
3. reservation 被拒绝：leg 进入 `ROUTE_REJECTED`，记录 `rejectReason`。
4. reservation 释放或 route 不再持有资源：leg 进入 `COMPLETED`，进入下一个 leg。
5. 每完成四个 leg，`cycleCompleted + 1`。
6. `cycleCompleted >= cycleTarget` 后，交路进入 `RESTING`，不再生成新 route。

## 6. 状态设计

### 6.1 CirculationPlan 状态

| 状态 | 含义 |
| --- | --- |
| `ASSIGNED` | 已分配交路，等待首段发车 |
| `IN_SERVICE` | 已开始执行循环 |
| `WAITING_ROUTE` | 当前 leg 已申请或等待信号反馈 |
| `RESTING` | 循环次数完成，车辆休息 |
| `CANCELLED` | 人工取消 |
| `BLOCKED` | 当前 leg 被拒绝且暂不能继续 |

### 6.2 CirculationLeg 状态

| 状态 | 含义 |
| --- | --- |
| `PLANNED` | 未到发车时刻 |
| `ROUTE_REQUESTED` | 已生成进路申请 |
| `ROUTE_ACCEPTED` | 信号已接受 |
| `ROUTE_REJECTED` | 信号拒绝 |
| `COMPLETED` | 当前 leg 完成 |
| `SKIPPED` | 所属计划取消 |

## 7. 与现有模块关系

本方案采用最小改动方式：

```text
TrainCirculationPlan
  -> 生成/维护多个 OperationPlan leg
  -> OperationPlan 到点生成 REQUEST_ROUTE
  -> RouteReservation 跟踪信号反馈
  -> TrainCirculationPlan 汇总圈数与当前 leg
```

也就是说：

- 不推翻现有 `OperationPlan`。
- 不修改信号侧 route catalog 维护方式。
- 不影响时间间隔调度的本车命令。
- 交路计划只负责“下一段 route 是什么、何时申请、跑几圈后休息”。

## 8. 前后端接口

新增接口建议：

```http
GET  /api/dispatch/circulation-plans
POST /api/dispatch/circulation-plans/auto
POST /api/dispatch/circulation-plans/{circulationId}/cancel
```

`POST /api/dispatch/circulation-plans/auto` 请求：

```json
{
  "cycleTarget": 3,
  "headwaySeconds": 300,
  "leadSeconds": 30
}
```

返回 `TrainCirculationPlan[]`。

`DispatchSnapshot` 中增加 `circulationPlans`，前端可在调度闭环页展示：

```text
车辆 | 起点 | 当前路线 | 圈数 | 状态 | 下一发车 | 最新反馈
```

## 9. 前端展示建议

路线调度页面不再突出拓扑绘制，而展示车队交路表：

| 车辆 | 起点 | 当前 leg | 圈数 | 状态 | 下一动作 |
| --- | --- | --- | --- | --- | --- |
| TR-001 | 郭公庄 | R_UP | 0/3 | IN_SERVICE | 到国图后折返 |
| TR-002 | 国家图书馆 | R_DOWN | 0/3 | IN_SERVICE | 到郭公庄后折返 |
| TR-003 | 郭公庄 | 等待 R_UP | 0/3 | ASSIGNED | 300s 后发车 |
| TR-004 | 国家图书馆 | 等待 R_DOWN | 0/3 | ASSIGNED | 300s 后发车 |

点击车辆后展示 leg 序列：

```text
[完成] R_UP
[进行] R_TB_LIB
[等待] R_DOWN
[等待] R_TB_GGZ
```

## 10. 后续增强点

1. 由信号侧提供明确到站、折返完成、下线确认事件，减少调度侧用 route reservation 推断。
2. 增加 `REST/OFFLINE` 命令，让循环完成后的休息状态真正通知车辆侧。
3. 增加异常重排：某个 route 被拒绝后，从候选 route 中选择替代路线并生成 `REROUTE`。
4. 增加路线调度专属 KPI：进路接受率、平均等待时间、拒绝原因分布、循环完成率。
5. 将当前内存态计划持久化，支持仿真暂停恢复。
