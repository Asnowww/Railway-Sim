# 司机台与显示系统端口聚合协议

## 1. 最终方向约束

四个现场端口保持各自原生 TCP/UDP 协议，不在现场网络层合并。只有司机台 PLC 能向平台产生业务输入；网络屏、信号屏和视景系统只接收平台输出并显示，不得作为车辆控制输入源。

| 现场设备 | 端点 | 传输 | 平台接收 | 平台发送 | 业务方向 |
|---|---|---|---:|---:|---|
| 司机台 PLC | `192.168.100.123:8001` | TCP | 46 字节控制帧 | 26 字节状态帧 | 唯一输入源，同时接收显示输出 |
| HMI 网络屏 | `192.168.100.121:8888` | TCP | 无 | 570 字节显示帧 | 仅输出 |
| MMI 信号屏 | `192.168.100.122:9999` | TCP | 无 | 68 字节显示帧 | 仅输出 |
| 视景系统 | `18.32.115.28:8302` | UDP | 无 | Version 1.3 视景帧 | 仅输出 |

“仅输出”是业务契约，不以对端 TCP 是否发送诊断字节为转移。生产链路不解析、不转发屏幕回传，也不允许屏幕数据改变牵引、制动、车门或列车状态。离线解码器仅用于协议回放和测试验证。

## 2. 现场报文基线

- PLC 输入：固定 46 字节，24 字节头加 22 字节数据区，周期 100 ms；标识为 `55 AA 55 AA`，头内总长/数据长为小端 `46/22`。
- PLC 输出：固定 26 字节，24 字节头加 2 字节显示状态区。
- HMI 网络屏：固定 570 字节，头内总长/数据长为 `570/546`，当前持续发送周期为 500 ms。
- MMI 信号屏：TCP 线上固定 68 字节；为兼容历史设备，头内总长/数据长仍为 `62/42`，当前持续发送周期为 500 ms。TCP 拆包不得使用头内 62 作为线上帧长。
- 视景系统：UDP Version 1.3，小端编码，默认周期 100 ms；固定包含 77 个信号机状态和 29 个道岔状态，随后为本车和其他列车数据。包含 `L` 辆其他列车时，报文长度为 `128 + 9L` 字节。

## 3. 服务间端口聚合

中央 `backend:8080` 管理四个现场端口。它只把 PLC 的原始 46 字节输入加上聚合头，通过车辆微服务现有的 `9300` 端口发送到统一入口：

```http
POST http://localhost:9300/vehicle-runtime/peripherals/frame
Content-Type: application/octet-stream
```

聚合不改写 PLC 原始载荷。9300 解开聚合帧后，继续使用 `DriverCabPlcCodec` 校验、解码并写入 `DriverCommandHolder`，再在下一 tick 与 MA、车门、自检和供电约束共同仲裁。

```text
PLC :8001 --46字节--> backend:8080 --RSIM聚合帧--> vehicle-runtime:9300
PLC :8001 <--26字节-- backend:8080 <--TrainState/cabDisplay-- 9300状态镜像
HMI :8888 <--570字节-- backend:8080
MMI :9999 <-- 68字节-- backend:8080
视景:8302 <--V1.3 UDP-- backend:8080
```

## 4. RSIM Version 1 聚合帧

除多字节字段均为小端外，车号使用 UTF-8，原始现场载荷保持不变。

| 偏移 | 长度 | 字段 | 说明 |
|---:|---:|---|---|
| 0 | 4 | magic | ASCII `RSIM`，即 `52 53 49 4D` |
| 4 | 1 | version | 固定为 1 |
| 5 | 1 | channel | 外设通道码 |
| 6 | 2 | headerLength | 固定为 20 |
| 8 | 4 | sequence | 无符号语义的递增序号 |
| 12 | 2 | trainIdLength | 车号 UTF-8 字节数，最大 64 |
| 14 | 4 | payloadLength | 原始现场载荷长度 |
| 18 | 2 | flags | 当前为 0，预留 |
| 20 | N | trainId | UTF-8 车号 |
| 20+N | M | payload | 原始现场协议帧 |
| 20+N+M | 4 | crc32 | 对此前全部字节计算 CRC32 |

通道码定义如下。9300 的入站接口当前只接受 `PLC_INPUT(1)`；任何输出通道以及历史保留的网络屏输入通道均返回 HTTP 400 和 `CHANNEL_DIRECTION_INVALID`。

| 通道码 | 名称 | 9300 入站策略 |
|---:|---|---|
| 1 | `PLC_INPUT` | 接受 |
| 2 | `PLC_OUTPUT` | 拒绝，仅表示方向契约 |
| 3 | `NETWORK_SCREEN_INPUT` | 历史保留，拒绝且不得启用 |
| 4 | `NETWORK_SCREEN_OUTPUT` | 拒绝，仅表示方向契约 |
| 5 | `SIGNAL_SCREEN_OUTPUT` | 拒绝，仅表示方向契约 |
| 6 | `VISION_UDP_OUTPUT` | 拒绝，仅表示方向契约 |

聚合入口状态可通过下列接口检查：

```http
GET http://localhost:9300/vehicle-runtime/peripherals/status
```

返回 `protocolVersion`、`acceptedChannels`、`acceptedFrames`、`rejectedFrames`、`lastSequence` 和 `lastTrainId`。格式、CRC、长度或版本错误返回 `AGGREGATE_DECODE_FAILED`；方向错误返回 `CHANNEL_DIRECTION_INVALID`；未知列车和 PLC 载荷错误继续使用车辆运行时原有错误码。

## 5. 验收条件

1. 只有 PLC 46 字节输入能增加聚合入口的 `acceptedFrames`，并在下一 tick 产生可追溯的司机命令决策。
2. 以通道 2～6 向 9300 聚合入口发包时均被拒绝；特别是网络屏、信号屏和视景通道不得写入车辆状态。
3. HMI、MMI 和视景链路只有平台出站业务计数，断线重连后继续发送，不依赖任何回包才能推进仿真。
4. PLC 输入聚合前后的 46 字节载荷逐字节一致；CRC32、车号、长度和序号可校验。
5. PLC 输出保持 26 字节，HMI 保持 570 字节，MMI 线上保持 68 字节，视景 Version 1.3 满足固定信号机/道岔数量和长度公式。
